# 上游失败重试优化 Impl

## 任务概述

本次优化扩大上游失败重试范围：用户请求已经进入上游调用后，除明显由请求本身造成的问题外，HTTP 非 2xx 与 Responses SSE 首包错误均允许按系统配置的重试次数进行重试。

本地系统预检拒绝的请求不受影响，例如余额不足、上下文超出、请求长度超限等仍在调用上游前直接返回。

## 文件变更清单

- `backend/app/services/proxy_service.py`
  - 调整 `_should_retry_upstream_status`：上游 HTTP `>=400` 默认可重试，仅排除 `400`、`413`、`422`。
  - 新增 `_should_retry_upstream_response`：HTTP 错误 body 中出现请求侧错误特征时不重试。
  - 新增 `_looks_like_non_retryable_upstream_request_error`：识别上下文过长、请求体过大、缺少必填参数、非法参数/schema 等用户侧错误。
  - 调整 `_should_retry_responses_stream_error`：Responses SSE 首包 `type=error` 默认可重试，仅排除用户侧错误。
  - Responses SSE 首包错误改为外层统一控制总尝试次数，底层 `_stream_lines_with_retries` 单次执行，避免重试次数叠加。
  - Responses WebSocket 首包用户侧错误直接返回简略请求错误，不跨渠道重试，也不记录为渠道失败。
  - 调整上游错误最终展示归类：非用户侧上游 HTTP 错误最终显示通用渠道异常，原始上游详情保留给日志。
  - 保留 Responses 流式首包错误重试与 `request_id` 日志追踪。
- `backend/test_proxy_retry_error_sanitization.py`
  - 增加 HTTP 状态码重试范围测试。
  - 增加 HTTP 错误 body 禁用重试测试。
  - 增加非标准上游 HTTP 错误最终脱敏测试。
  - 增加 Responses SSE 首包错误重试/不重试测试，以及 WebSocket 首包用户侧错误不重试测试。

## 核心代码说明

重试入口统一使用 `_should_retry_upstream_status` 和 `_should_retry_upstream_response`。对于已经调用上游后返回的 HTTP 错误，除 `400`、`413`、`422` 这类明确请求问题外，其余 `401`、`402`、`403`、`404`、`429`、`5xx`、`521` 等默认允许重试；如果错误 body 明确显示请求格式、上下文、参数/schema 问题，则不重试。

Responses 流式接口如果首个可见事件就是 `type=error`，并且未向用户输出内容，则按动态重试次数重新调用上游。该逻辑由外层统一控制总尝试次数，底层单次执行，避免 HTTP/传输重试与 SSE 错误重试叠加。若已经向用户输出过内容，则不再重试，避免流式响应重复或错乱。
