# 套餐-模板额度编辑 Impl

## 任务概述

修复 `admin/subscription` 套餐模板编辑额度后页面与后续发放不体现修改值的问题。

## 文件变更清单

- `frontend/src/views/admin/SubscriptionManage.vue`
  - 套餐模板列表改用模板自身 `quota_metric/quota_value` 展示每日额度。
  - 编辑套餐模板时优先回填模板自身字段，不再优先回填运行时解析后的 `resolved_quota_*`。
- `backend/app/services/subscription_service.py`
  - 新增内置畅享默认 Token 额度常量。
  - 将畅享套餐固定美元额度兼容规则限定为“内置畅享 + 默认 1000 万 Token”。
  - 管理员修改额度值或额度口径后，后续发放按保存的模板字段生成套餐快照。
  - Review 后补充历史 `cost_usd=50/60` 用户套餐快照兼容，2026-06-18 14:00 前的内置编码快照继续按官方成本口径累计额度。
  - 模板侧和用户快照侧都只按内置套餐编码识别默认畅享包，避免自定义“畅享”命名模板被误转为固定美元额度。
  - 默认内置畅享模板发放时保存模板原始 `total_tokens=10000000` 快照，由运行时解析为官方成本额度；管理员显式配置 `cost_usd=50/60` 的新快照不再被当作历史快照。
- `backend/test_subscription_plan_quota_edit.py`
  - 覆盖默认内置畅享兼容、默认模板快照、Token 额度修改、美元额度修改、`cost_usd=50/60` 新快照、有编码/无编码自定义畅享命名模板、历史成本快照兼容和 cutover 精确边界后的解析行为。

## 核心代码说明

历史逻辑通过套餐编码或“畅享”名称把内置畅享包解析为 `$50/天` 官方成本额度，这会覆盖管理员在模板上保存的 `quota_metric/quota_value`。本次新增 `_is_legacy_enjoy_default_quota_record`，只有默认 `total_tokens=10000000` 的畅享记录继续走兼容逻辑；一旦模板额度被改成其他 Token 值或改成 `cost_usd`，解析策略直接使用保存值。

Review 发现旧代码已经发放出的畅享用户套餐可能保存为 `cost_usd=50/60` 快照。为避免历史记录从官方成本口径变成平台扣费口径，补充 `_is_legacy_enjoy_cost_snapshot_record`，只对 2026-06-18 14:00 前、且 `plan_code_snapshot` 属于内置畅享编码的历史快照继续返回 `use_official_cost=True`。同时模板侧和新用户快照都按 `plan_code` 精确判断，不再通过名称包含“畅享”命中兼容规则。

为避免新发放的默认内置畅享包和管理员显式配置的 `cost_usd=50/60` 混淆，`activate_plan_subscription` 通过 `_get_plan_subscription_quota_snapshot` 生成快照：未改过的内置畅享模板保存原始 Token 配置，后续运行时再解析为官方成本额度；管理员显式配置美元额度的新快照则按普通保存值解析。

前端管理页区分了两类展示：

- 套餐模板表：展示模板配置值，方便管理员确认刚保存的额度。
- 套餐用户/套餐记录：继续展示运行时有效额度，不改变用户使用明细查看逻辑。
