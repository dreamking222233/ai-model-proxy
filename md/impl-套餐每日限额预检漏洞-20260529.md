# 套餐每日限额预检漏洞实施记录

## 任务概述

修复套餐每日限额绕过问题：当用户套餐当日剩余额度不足以覆盖本次请求时，系统必须继续校验余额是否足以支付本次请求；套餐额度和余额都不足时禁止请求。请求完成后的实际记账也必须遵循同一规则，不能把超出每日限额的消费继续写入套餐周期。

## 文件变更清单

- `backend/app/services/subscription_service.py`
  - 新增套餐实际消费与预估消费解析方法。
  - 请求前套餐额度检查改为比较 `remaining_amount` 与本次预估消费。
  - 套餐周期更新增加 `used_amount + consumed_amount <= quota_limit` 原子条件。
  - 套餐用量汇总与周期重建排除 `billing_mode="balance"` 的余额扣费记录。
- `backend/app/services/proxy_service.py`
  - 套餐超额后的余额兜底改为校验余额是否覆盖本次预估费用。
  - 请求后记账在写套餐前按实际消费复核剩余额度；套餐不足但余额足够时按余额扣费，二者都不足时抛出中文错误。
  - 余额兜底消费不再关联 `subscription_id`，避免后续被套餐用量汇总误计。
  - 将一处英文用户不存在错误改为中文。
- `backend/app/test/test_subscription_compatibility.py`
  - 更新套餐预检测试，覆盖预估消费超过剩余额度时的拦截。
  - 增加请求后套餐不足转余额扣费测试。
  - 增加套餐和余额都不足时禁止记账测试。

## 核心代码说明

1. 成本型套餐使用 `estimated_quota_cost` 参与套餐额度预检，保证套餐限额按官方额度口径计算；余额兜底使用 `estimated_total_cost`，保证余额按真实计费口径判断。
2. Token 型套餐使用 `estimated_total_tokens` 参与套餐额度预检。
3. 请求后实际消费可能高于预估，因此在记账前再次计算套餐实际应扣额度。如果套餐剩余额度不足，则只在余额覆盖实际费用时切换到 `billing_mode="balance"`。
4. 余额兜底记录不挂套餐周期，也不会被套餐汇总/重建统计进套餐额度。
5. 套餐周期 SQL 更新添加原子条件，避免并发请求同时通过预检后把 `used_amount` 写穿每日限额。
