# 套餐美元日限额改造 Impl

日期：20260510

## 任务概述

按 `2026-05-10 00:00:00 Asia/Shanghai` 的切换规则，完成以下套餐日额度改造：

- 无限套餐（日 / 周 / 月）：
  - 存量用户 `120 USD/天`
  - 新开用户 `100 USD/天`
- 畅享日卡 / 周卡 / 月卡：
  - 存量用户 `60 USD/天`
  - 新开用户 `50 USD/天`

套餐日额度成本按模型官方价格 `1:1` 计算，缓存读取按输入价格 `10%` 计入额度；余额实际扣费逻辑保持原有倍率规则不变。

## 核心实现

### 1. 订阅额度策略统一解析

在 `SubscriptionService` 中新增订阅额度策略解析，按订阅快照识别：

- `daily-unlimited`、`weekly-unlimited`、`monthly-unlimited`
- 畅享套餐家族 `daily-10m-token`、`weekly-10m-token`、`monthly-10m-token`
- 其他普通 `daily_quota`

解析结果统一返回：

- `quota_metric`
- `quota_limit`
- `hard_limit`
- `use_official_cost`

存量用户通过 `created_at < 2026-05-10 00:00:00` 区分。

### 2. 无限套餐与畅享额度改为美元口径

`check_quota_before_request()`、`consume_quota_after_request()`、`current_cycle`、订阅摘要序列化都改为读取解析后的有效额度，而不是直接使用历史快照里的 `quota_metric/quota_value`。

这样现有存量记录无需迁移数据库，也能在切换后直接按新美元规则生效。

### 3. 套餐额度成本与余额计费成本拆分

在 `ProxyService` 中新增套餐额度专用成本：

- 输入：官方输入价
- 缓存读取：官方输入价的 `10%`
- 输出：官方输出价

并增加：

- `estimated_quota_cost`：请求前套餐预检成本
- `quota_cost`：请求后套餐实际累计成本

余额计费仍继续使用系统 `price_multiplier` 和现有账单成本口径。

### 4. 新开套餐快照归一化

新开套餐时：

- 无限套餐日 / 周 / 月快照直接写入 `cost_usd / 100`
- 畅享套餐快照直接写入 `cost_usd / 50`

这样新记录从创建开始就与切换后的业务口径一致。

### 5. 前端展示改为读取解析后的额度

管理端和代理端套餐页面新增使用解析后的展示字段：

- `resolved_quota_metric`
- `resolved_quota_value`

从而支持同一页面混合展示：

- 无限套餐 `$100/天` 或 `$120/天`
- 畅享套餐 `$50/天` 或 `$60/天`

代理端记录列表的“已用 Token”同步调整为“当日已用”，改读 `current_cycle`。

### 6. 旧周期数据按当前口径重建使用快照

线上已有存量 `subscription_usage_cycle` 是按旧 `total_tokens` 口径创建的，而切换后的运行时策略会把无限套餐解析为 `cost_usd`。如果直接复用旧周期上的 `used_amount`，会把旧 token 数值误当成美元额度，导致用户在实际只消费几美元时被误判超限。

为此在 `_get_or_create_cycle()` 中补充了周期口径校准：

- 当已存在 cycle 的 `quota_metric` 与当前订阅解析策略不一致时
- 不直接清零，也不继续沿用旧 `used_amount`
- 改为基于当日 `ConsumptionRecord` 重建周期快照
- `cost_usd` 口径取当日 `total_cost` 聚合
- token 口径优先取 `raw_total_tokens`，为空时回退到 `total_tokens`

这样老订阅在规则切换后会自动把周期累计值纠正到当前口径，避免再次出现“旧 token 周期 + 新美元限额”混算。
