# 扣费-套餐余额混合扣费 Impl

## 任务概述

修复文本请求扣费链路，统一为“先消耗套餐每日额度，再扣余额补差，套餐和余额都不足时拒绝请求”。同时修复取消套餐后 `admin/users` 页面仍显示套餐剩余天数的问题。

## 文件变更清单

- `backend/app/services/proxy_service.py`
  - 余额预检改为按预计费用判断，不再只按 `$0.1` 阈值放行。
  - 套餐额度不足时支持同一请求内 `mixed` 混合扣费。
  - 纯余额用户、套餐过期转余额用户在实际扣费前校验余额，避免余额扣成负数。
  - 并发下套餐额度被抢占时重新读取剩余额度，尽量消耗套餐后再扣余额。

- `backend/app/services/subscription_service.py`
  - 新增按指定额度消耗套餐周期的方法。
  - 套餐使用统计纳入 `billing_mode="mixed"`。
  - 周期用量重建优先使用 `quota_consumed_amount`，避免混合扣费记录把余额补差部分重算回套餐额度。
  - 周期重新加载使用 `populate_existing()`，并在记账更新失败后过期旧 cycle 缓存，避免并发场景读取到 Session 旧值。
  - 取消套餐时显式清理 `sys_user.subscription_type/subscription_expires_at` 缓存。

- `backend/app/services/auth_service.py`
  - `admin/users` 列表返回值改为按当前真实有效套餐计算。
  - 遇到历史脏缓存时自动刷新并提交，避免已取消套餐继续显示剩余天数。

- `backend/app/test/test_subscription_compatibility.py`
  - 更新余额预检测试。
  - 更新套餐余额混合扣费测试。
  - 新增 `admin/users` 套餐展示修复测试。

## 核心代码说明

扣费阶段现在根据本次真实费用和套餐剩余额度计算余额补差：

- 套餐足够：`billing_mode="subscription"`，余额不变。
- 套餐剩余大于 0 但不足：消耗剩余套餐额度，余额扣除超出部分，`billing_mode="mixed"`。
- 套餐剩余为 0：整单走余额，`billing_mode="balance"`。
- 余额不足以承担需要扣余额的部分：拒绝请求，返回中文余额不足提示。

对于成本口径套餐，余额补差按套餐计费口径占比折算；对于 Token 口径套餐，按 Token 消耗占比折算。
