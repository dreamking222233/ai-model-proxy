# 排行榜周期切换优化 Impl

## 任务概述

优化 `user/ranking` 与 `admin/ranking` 页面周期切换显示与 Top 10 数据返回。两个页面当前共用 `frontend/src/views/user/UsageRanking.vue`，因此本次前端改动会同时作用于用户端和管理端排行页。

## 文件变更清单

- `backend/app/api/user/stats.py`
  - `GET /api/user/stats/token-ranking` 增加 `limit` 查询参数，默认值为 10，范围为 1-50。
  - 调用 `LogService.get_token_ranking(db, days, limit)`，不再固定取 5 条。
  - 响应中返回 `limit`，方便前端或排查确认实际榜单数量。
- `backend/app/services/log_service.py`
  - `LogService.get_token_ranking()` 默认 `limit` 同步调整为 10，避免后续新增调用方未显式传参时退回 Top 5。
  - 服务层对直接传入的 `limit` 做 1-50 范围收敛，防止绕过 API 层时出现异常大查询。
- `frontend/src/views/user/UsageRanking.vue`
  - 将标题由固定“每周排行榜”改为按 `days` 动态显示“今日排行榜 / 近 7 天排行榜 / 近 30 天排行榜”。
  - 请求榜单时显式传入 `{ days, limit: 10 }`。
  - 副标题同步展示当前统计周期。
  - 移除无真实数据来源的随机趋势展示和相关样式/方法。
  - 增加请求序号保护，避免快速切换统计周期时旧请求后返回覆盖新周期数据。
- `backend/app/test/test_user_stats_ranking.py`
  - 新增用户榜单接口和服务层测试，覆盖默认 Top 10、`days/limit` 参数传递、非法 `limit` 的 FastAPI 参数校验，以及服务层默认值和范围收敛。
- `md/plan-排行榜周期切换-20260503.md`
  - 记录实施方案。

## 核心代码说明

`admin/ranking` 在路由中复用 `frontend/src/views/user/UsageRanking.vue`，所以只改该组件即可同步修复两个页面的标题与请求参数。

后端原实现：

```python
ranking = LogService.get_token_ranking(db, days, 5)
```

已改为：

```python
limit: int = Query(10, ge=1, le=50)
ranking = LogService.get_token_ranking(db, days, limit)
```

这样页面切换为今日、近 7 天、近 30 天时，会按对应 `days` 参数查询，并返回 Top 10。

Review 后补充处理：

- `LogService.get_token_ranking()` 默认值同步为 `limit=10`。
- 移除 `UsageRanking.vue` 中未使用的 `formatDate` 导入和 methods 暴露。
- 后端测试从函数级调用升级为基于 `httpx.ASGITransport` 的路由级测试。

第二轮 Review 后补充处理：

- `UsageRanking.vue` 使用 `rankingRequestSeq` 避免旧请求覆盖新周期数据。
- 发起新周期请求时清空旧榜单，避免标题已切换但 Top 3 暂时显示旧周期数据。
- `LogService.get_token_ranking()` 对服务层直接传入的 `limit` 做 1-50 范围收敛。
- 新增服务层默认 Top 10 和范围收敛测试。
