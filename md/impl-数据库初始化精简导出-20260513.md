## 任务概述

基于当前项目代码和初始化脚本，整理出一份适合新服务器首次部署的 `initData.sql`。

目标不是导出当前数据库全部数据，而是保留：

1. 完整建表 SQL
2. 部署后立即需要的系统初始化数据

同时移除不适合随项目迁移的运营/业务数据。

## 文件变更清单

- `sql/initData.sql`
- `backend/sql/initData.sql`
- `md/plan-数据库初始化精简导出-20260513.md`

## 核心判断

### 必须保留的数据

1. `sys_user`
   - 默认管理员账号是后台登录入口，没有管理员账号就无法进入管理端完成后续配置。

2. `user_balance`
   - 管理员资料页和余额相关接口会读取余额表，初始化管理员余额可避免空数据。

3. `user_image_balance`
   - 用户/管理员资料页和图片积分逻辑依赖该表。

4. `system_config`
   - `backend/app/main.py` 启动时会读取 `health_check_interval`
   - `admin/config` 页面需要显示配置项
   - 平台站点信息、注册开关、缓存开关、会话压缩开关、价格倍率等都来自这里
   - 因此保留了一组“部署后建议具备”的完整系统配置项，而不是只留最少的 1-2 个键

### 不导出的数据

1. `channel`
   - 属于部署环境相关配置，不同服务器、不同客户环境通常不同
   - API Key 更不应该随初始化脚本迁移

2. `unified_model`
3. `model_channel_mapping`
4. `model_override_rule`
   - 这些属于业务模型编排配置，不是系统启动硬依赖
   - 新服务器部署后可按实际渠道重新配置

5. 日志与流水类数据
   - `request_log`
   - `consumption_record`
   - `health_check_log`
   - `operation_log`
   - `cache_log`
   - `request_cache_summary`
   - 这些都是运行期数据，不适合作为初始化内容

6. 业务过程数据
   - 兑换码
   - 用户订阅记录
   - 代理余额流水
   - 代理结算记录
   - 会话状态/检查点
   - 这类数据均不应进入新环境初始化脚本

### 套餐模板为何没有预置

`SubscriptionService.list_plans()` 内部会调用 `ensure_default_plans()`，当 `subscription_plan` 为空时会自动创建默认套餐模板。

因此 `subscription_plan` 不是必须导出的初始化数据。

## 使用建议

新服务器部署顺序建议：

1. 创建数据库
2. 执行 `sql/initData.sql`
3. 启动后端
4. 使用默认管理员账号登录后台
5. 在管理端补充渠道、模型、模型映射、域名和联系信息等环境相关配置
