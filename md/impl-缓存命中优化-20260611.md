# 缓存命中优化实施记录

## 任务概述

基于三个反代项目的缓存处理研究，本次实现低风险缓存命中优化：

- 新增文本请求渠道亲和，减少同一会话在不同上游缓存池之间切换。
- 优化 Anthropic Prompt Cache 断点，避免把最新用户输入作为历史缓存断点。
- 增加配置项，管理员可在 `/admin/config` 调整渠道亲和开关、TTL 与 fallback 策略。

## 文件变更清单

- `backend/app/services/channel_affinity_service.py`
  - 新增进程内 TTL 渠道亲和服务。
  - 支持从请求体、请求头、Codex turn metadata 提取会话 key。
  - 缺少显式 key 时可基于 system/tools/首个 user 生成稳定 fallback key。

- `backend/app/services/proxy_service.py`
  - OpenAI Chat、Anthropic Messages、Responses 文本入口接入渠道亲和排序。
  - 文本请求成功后在统一记账入口记录成功渠道绑定。

- `backend/app/services/anthropic_prompt_cache_service.py`
  - 历史断点从“最新 user”改为“倒数第二个 user”。
  - 注入后规范 1h/5m TTL 顺序。
  - 注入后强制限制最多 4 个 cache_control 断点。

- `frontend/src/views/admin/SystemConfig.vue`
  - 缓存配置卡片新增渠道亲和开关、TTL、fallback 开关。
  - 系统配置列表新增中文名称映射。

- `backend/sql/upgrade_channel_affinity_20260611.sql`
  - 新增渠道亲和配置升级 SQL。

- `backend/sql/init.sql`、`backend/sql/initData.sql`、`sql/initData.sql`
  - 新部署默认配置同步新增渠道亲和配置。

- `backend/test_channel_affinity_service.py`
  - 新增渠道亲和排序与 key 隔离测试。

- `backend/test_anthropic_prompt_cache_policy.py`
  - 增加历史断点位置与 TTL 顺序测试。

## 核心代码说明

渠道亲和只记录 `affinity_key -> channel_id`，不缓存上游响应内容。`affinity_key` 使用用户 ID、代理 ID、请求模型、协议与会话来源共同组成，避免跨用户、跨模型误绑定。

Prompt Cache 不改写业务文本和消息顺序，只移动系统注入断点的位置：静态内容仍优先 tools/system，历史内容优先上一轮 user，最新 user 保持不加断点。

## 待观察项

- 进程内亲和缓存会在后端重启后清空，适合先灰度验证；如生产多进程或多节点明显，可后续升级 Redis 存储。
- zz1 与首尔渠道如果上游缓存池天然隔离，亲和只能减少切换损耗，不能把 zz1 的缓存能力提升到首尔同等水平。
