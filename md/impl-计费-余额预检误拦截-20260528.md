# 计费-余额预检误拦截 Impl

## 任务概述

修复余额或余额兜底场景下，因为请求前 `estimated_total_cost` 粗略预估过高而误返回“余额不足以承担本次请求”的问题。当前余额预检恢复为只判断余额是否大于文本请求最低阈值 `0.1` 美元；长上下文、较大的输出上限、Fast 倍率或系统价格倍率不再作为余额放行的硬拦截条件。

## 文件变更清单

- `backend/app/services/proxy_service.py`
  - 调整 `_can_balance_cover_text_precheck`。
  - 移除 `estimated_total_cost` 与当前余额的前置覆盖比较。
  - 保留余额最低阈值判断：`available_balance > MIN_TEXT_REQUEST_USD_THRESHOLD`。
- `backend/app/test/test_subscription_compatibility.py`
  - 更新余额模式测试：余额大于阈值时，即使预估成本高于余额也允许请求继续。
  - 更新套餐额度不足后的余额兜底测试：余额大于阈值时，即使预估成本很高也允许兜底放行。
- `md/plan-计费-余额预检误拦截-20260528.md`
  - 记录问题分析、方案和实施步骤。

## 核心代码说明

`ProxyService._build_text_quota_precheck` 仍会生成 `estimated_total_cost`，用于观测、日志和后续扩展；但 `_can_balance_cover_text_precheck` 不再把它作为余额预检的硬门槛。

这与既有计费语义保持一致：请求前只做最低余额门槛判断，请求完成后按真实 token 用量扣费；如果本次请求把余额扣到 0 或负数，下一次请求再被前置阈值拦截。
