# 计费-套餐预检放行 Impl

## 任务概述

修复套餐当日额度仍有剩余时，因为本次请求预计扣费高于剩余额度而提前返回 402 的问题。现在套餐用户请求前只判断当日剩余额度是否已经用尽；只要剩余额度大于 0，就允许请求继续发送到上游，请求完成后按真实用量记账，允许本次把套餐剩余额度扣成负数。

## 文件变更清单

- `backend/app/services/subscription_service.py`
  - 调整 `check_quota_before_request`。
  - 成本额度和 Token 额度都改为仅在 `remaining_amount <= 0` 时拒绝。
  - 不再使用 `quota_precheck` 的 `estimated_quota_cost` / `estimated_total_tokens` 拦截套餐请求。
  - 调整 `_serialize_cycle`，套餐剩余额度为负数时不再截断为 0。
- `backend/app/test/test_subscription_compatibility.py`
  - 更新套餐前置校验测试。
  - 覆盖“预估超出剩余额度但剩余为正时放行”。
  - 保留“剩余额度为 0 时拒绝”的测试。
  - 补充“请求后扣成负数时 summary 返回负数 remaining_amount”的序列化测试。

## 核心代码说明

`SubscriptionService.check_quota_before_request` 仍会加载当前套餐周期和有效额度，但判断条件从“预估本次消耗不能超过剩余额度”改为“剩余额度必须大于 0”。

请求后扣费逻辑未变，`consume_quota_after_request` 会继续把真实消耗累加到 `SubscriptionUsageCycle.used_amount`，因此本次请求可以把 `quota_limit - used_amount` 扣成负数。`_serialize_cycle` 现在会直接返回该负数。下一次请求进入前置校验时会因为 `remaining_amount <= 0` 被拒绝，并继续复用现有余额兜底逻辑。
