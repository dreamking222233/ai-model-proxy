# 计费并发控制 Impl

## 任务概述

本次按 plan 实施低资产计费并发控制，目标是在不改变现有最终扣费顺序的前提下，限制低余额、低套餐额度、无套餐用户的上游请求并发，降低“上游成功但本地计费失败”的风险。

最终扣费仍保持当前逻辑：

1. 优先消耗有效套餐额度。
2. 套餐不足时按未覆盖部分扣余额。
3. 无有效套餐时走余额扣费。

## 核心代码说明

### 并发租约

`BillingConcurrencyService.acquire` 使用 Redis Lua 脚本执行：

1. 清理过期租约。
2. 统计当前用户活跃请求数。
3. 超过限制则抛出 `BILLING_CONCURRENCY_LIMITED`。
4. 未超过限制则写入当前 `request_id`，并设置 key 过期时间。

`release` 使用 `ZREM` 删除当前请求租约。

Redis 不可用时，对需要并发保护的请求 fail closed，返回 `BILLING_CONCURRENCY_UNAVAILABLE`，避免恢复到旧的低资产穿透风险。

### 准入决策

`ProxyService._assert_text_request_allowed` 保持原有预检异常语义，但成功时返回 `BillingAdmissionDecision`。

默认配置：

- `billing_low_asset_threshold_usd`: `2`
- `billing_low_asset_concurrency_limit`: `3`
- `billing_concurrency_lease_ttl_seconds`: `7200`

低资产判断：

- 无有效套餐：限制并发。
- 成本口径套餐剩余额度大于等于 2：不限制。
- 套餐额度不足但余额大于 2：不限制。
- 成本口径套餐剩余额度小于 2 且无余额：限制。
- token 口径套餐不按 `2 token` 判断，主要参考余额阈值。

### 流式释放

非流式请求在上游调用完成后立即释放租约。

流式请求返回 `StreamingResponse` 时，不在入口提前释放，而是替换 `body_iterator`，在流式迭代结束或客户端断开触发 finally 时释放租约。

Responses WebSocket 按单轮请求获取租约，单轮完成、失败或异常时释放。
