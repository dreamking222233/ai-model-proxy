# 请求失败重试机制实施记录

## 任务概述

为中转站请求增加失败重试处理：上游请求遇到临时失败时不立即返回失败给用户，而是在同渠道内最多重试 3 次；超过重试次数仍失败时沿用现有失败返回与失败请求记录逻辑。

## 文件变更清单

- `backend/app/services/proxy_service.py`
  - 将 `_UPSTREAM_RETRY_ATTEMPTS` 调整为首次请求 + 3 次重试。
  - 新增 `_post_files_with_retries`，覆盖 multipart 图片编辑请求。
  - 新增 `_stream_lines_with_retries`，覆盖可安全替换的流式首个有效输出前重试。
  - Responses / OpenAI via Anthropic stream 使用 retry boundary，避免空行、`event:` 行提前关闭透明重试窗口。
  - OpenAI via Anthropic 非流式请求改为复用 `_post_with_retries`。
  - OpenAI stream 与 OpenAI via Anthropic stream 改为复用 `_stream_lines_with_retries`。
  - Responses stream/WebSocket 统一迭代器改为复用 `_stream_lines_with_retries`。
  - Google 图片生成请求改为复用 `_post_with_retries`。
  - OpenAI 图片编辑请求改为复用 `_post_files_with_retries`。
  - 图片生成/编辑上游 5xx 重试耗尽后按渠道失败处理，记录渠道失败并继续 failover；4xx 仍按请求错误处理。
- `backend/app/services/google_vertex_image_service.py`
  - Vertex 图片生成对每个候选模型增加最多 3 次失败重试。
  - 将单候选模型调用抽为 `_generate_one_candidate`，便于重试和测试。
  - 5xx 级 `ServiceException` 与普通 SDK 异常会重试，4xx 级 `ServiceException` 不重试。
- `backend/app/test/test_proxy_retry_mechanism.py`
  - 覆盖 JSON POST 失败后重试 3 次成功。
  - 覆盖不可重试 400 不重试。
  - 覆盖 multipart POST 可重试状态码。
  - 覆盖流式首行前重试成功。
  - 覆盖流式首行后异常不再透明重试。
  - 覆盖非 payload 流式行后断流仍可在首个有效 payload 前重试。
  - 覆盖图片生成 5xx 重试耗尽后记录渠道失败。
  - 覆盖 Vertex 候选模型失败后重试 3 次成功。
  - 覆盖 Vertex 4xx 业务错误不重试。

## 核心代码说明

- 文本/图片 HTTP POST 重试仍只在上游调用层发生，重试期间不写失败请求日志、不扣费。
- 流式重试只发生在首个有效输出之前；Responses 等会忽略空行或 `event:` 行的入口不会因这些控制行提前关闭重试窗口。
- 图片上游 5xx 重试耗尽后进入渠道失败路径，避免持续命中异常渠道；图片上游 4xx 仍保留为请求错误，不误伤渠道健康。
- 成功响应后才进入现有计费与成功日志流程，避免重复扣费。
- 最终仍失败时沿用入口层原有 `_log_failed_request` 逻辑。
- Anthropic passthrough stream 保留既有 prompt-cache variant fallback 与压缩 fallback，未强行套通用 helper，避免破坏既有专属 fallback。
