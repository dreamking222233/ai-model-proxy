# 错误提示与重试配置实施记录

## 任务概述

本次调整将上游错误与用户可见错误分离：上游渠道返回的原始错误不再直接返回给用户，只保存在请求日志中供管理员在 `admin/logs` 查看。上游临时失败重试次数改为读取 `system_config.max_retry_count`，并在 `admin/config` 页面可配置。针对 256k 上下文模型，超过 250000 token 的请求返回固定压缩对话提示。

## 文件变更清单

- `backend/app/services/proxy_service.py`
  - 新增动态重试次数读取，默认 3 次，最大限制 10 次。
  - JSON、multipart、Responses/OpenAI 流式、Anthropic 原生流式首个有效输出前重试统一使用运行时重试配置。
  - 最终所有渠道失败统一返回 `调用失败，渠道异常，请稍后重试`。
  - 上游 4xx 请求错误不再透传上游 body，用户看到平台简略错误；日志保留原始上游错误。
  - 上游 context length/token limit 类错误统一返回 `请求超出最大上下文,请压缩对话`。
  - 请求长度预检结合 `unified_model.max_tokens`，对 256k 模型超过 250000 token 的请求提前拦截。
  - Responses WebSocket、视频生成/查询/内容下载、图片生成/编辑失败路径均改为用户脱敏、日志保留原文。
  - 失败日志写入 `RequestLog.error_message` 前不再预截断，避免管理员详情丢失上游原始错误。
  - Responses SSE/WebSocket 上游 `type=error` 事件不再直接透传，已输出部分内容的 WebSocket 会补发脱敏错误事件。
  - 408/429/5xx 等临时上游失败不再作为 request_error 跳过渠道失败记录，仍由最终响应返回简略 503。
- `frontend/src/views/admin/SystemConfig.vue`
  - 健康/容错配置卡片新增“上游失败重试次数”输入项。
  - 保存配置时同步更新 `max_retry_count`。
- `sql/initData.sql`
- `backend/sql/initData.sql`
- `backend/sql/init.sql`
  - 更新 `max_retry_count` 描述为“上游失败重试次数”。
- `backend/sql/upgrade_upstream_retry_error_sanitization_20260604.sql`
  - 新增幂等升级脚本，确保老环境存在该配置。
- `backend/test_proxy_retry_error_sanitization.py`
  - 新增单测覆盖动态重试、错误脱敏、日志原文保留和 256k 上下文提示。

## 核心代码说明

- 重试次数通过 `ProxyService._resolve_upstream_retry_attempts()` 从系统配置读取，并在每个渠道尝试前绑定到 channel 运行时属性，避免不同分支读取不一致。
- `_map_upstream_request_error()` 返回用户可见的脱敏 `ServiceException`，同时通过 `_upstream_detail` 保存原始上游错误，最终 `_log_failed_request()` 写入原文。
- `_looks_like_raw_upstream_error()` 使用结构化错误体、HTTP 状态、request id、provider/channel/group/distributor 等上游元信息和失败语义做启发式判断，不依赖具体模型名、渠道名或单个错误样例。
- `_failure_error_log_detail()` 统一处理最终失败日志：如果异常携带 `_upstream_detail`，管理员日志记录上游原文；用户响应继续使用简略文案。
- `_sanitize_upstream_service_exception_for_user()` 二次脱敏时优先继承已有 `_upstream_detail`，避免图片 4xx 等路径丢失上游原始 body。
- `ALL_CHANNELS_FAILED` 不再拼接 `error_detail`，统一使用简略 503 文案。
- `_validate_request_length()` 新增 `unified_model` 参数：当模型 `max_tokens <= 256000` 且预估 token 超过 250000 时直接返回压缩对话提示。
- `_stream_anthropic_passthrough_request()` 在把 upstream response 交给流消费前，对 408/429/5xx 和建连类异常按配置重试；一旦响应已进入流式消费，不再重试，避免混合多个 SSE 响应。
- `_forward_responses_websocket_turn()` 遇到上游原始 error payload 时，未输出任何客户端内容则允许切换渠道重试；已输出内容则只发送脱敏错误，并把原始错误挂入内部异常供日志使用。
- `_stream_responses_request()` 遇到上游 `type=error` payload 时不透传原始 payload，而是抛给外层统一返回脱敏错误、记录渠道失败和原始日志。
- 视频 retrieve/content/wait 上游失败会写入 `RequestLog.error_message`，管理员可在日志详情查看原始 body。
- OpenAI 图片生成/编辑失败不再把上游 body 本地化后写入 `_upstream_detail`，而是用户返回简略错误、管理员日志保留原始 body。
