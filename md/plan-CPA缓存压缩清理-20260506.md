# CPA缓存压缩清理

## 用户原始需求

- 当前 `43.156.153.12-codex转claude` 渠道虽然实际上请求的是上游 CPA/GPT，但日志中仍显示系统自带的 `会话压缩 / 新会话 / NEW / 预估省 0 tok`。
- 需要彻底取消系统内部这套会话压缩与 shadow 统计，不再参与请求处理、日志写入和前端显示。
- 系统后续仅使用上游 CPA 返回的真实缓存信息进行展示与计费。

## 技术方案设计

- 后端请求链路中，停用 Anthropic/Codex 路径的 `ConversationShadowService` 分析、合并、session commit、cooldown 标记等逻辑。
- 后端日志字段汇总统一将 `conversation_*` 与 `compression_*` 输出归零或置空，避免前端再次读到这套历史逻辑字段。
- 前端日志与余额相关页面移除“会话压缩”展示区块，仅保留 CPA 上游缓存创建/读取与实际计费明细。
- 保留数据库字段兼容性，不在本次做删字段迁移，避免放大改动面。

## 涉及文件清单

- `backend/app/services/proxy_service.py`
- `backend/app/services/request_cache_summary_service.py`
- `backend/app/services/log_service.py`
- `frontend/src/views/admin/RequestLog.vue`
- `frontend/src/views/admin/AgentRequestLog.vue`
- `frontend/src/views/agent/RequestLog.vue`
- `frontend/src/views/user/BalanceLog.vue`

## 实施步骤概要

1. 定位 Anthropic/Codex 请求入口中 `conversation shadow` 的注入点。
2. 关闭后端合并、提交、降级冷却等本地会话压缩逻辑。
3. 调整日志出参，默认不再透出 `compression_*` 和 `conversation_*`。
4. 清理前端四个页面中的“会话压缩”展示。
5. 构建前端并执行关键后端校验，确认只剩 CPA 缓存展示与计费。
