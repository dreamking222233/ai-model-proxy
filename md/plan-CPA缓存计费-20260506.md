# CPA缓存计费 Plan

日期：20260506

## 用户原始需求

当前中转站已有内部缓存分析机制，但该机制不影响真实扣费，容易在日志页面造成误导。需要取消内部缓存作为主要展示与扣费依据，改为使用 CPA/上游响应返回的真实缓存用量信息。

补充要求：以后请求也不再使用系统自己处理的内部缓存分析，例如 `内部缓存/PARTIAL/读 N 段/建 N 段/复用 tok` 这类处理需要整体取消。

目标页面包括：

- `admin/logs`
- `admin/agent-logs`
- `user/balance`
- `/agent/logs`

用量区域展示：

- 输入
- 输出
- 缓存创建
- 缓存读取

没有真实缓存信息时不显示缓存项。计费逻辑：

- 普通输入按输入价格计费。
- 输出按输出价格计费。
- 缓存创建不额外计费。
- 缓存读取按输入价格的 10% 计费。

示例：

- 第一次：输入 10000、输出 100、缓存创建 10000，按输入 10000 + 输出 100 计费。
- 第二次：输入 5、缓存读取 10000、输出 200，按输入 5 + 缓存读取 10000 * 10% + 输出 200 计费。

结合 `cpa缓存说明.md`：

- `8317 /v1/chat/completions` 渠道可通过 `usage.prompt_tokens_details.cached_tokens` 观察缓存读取。
- `8080 /v1/messages` 当前未观察到可用缓存。

## 技术方案设计

### 1. 后端缓存用量口径

统一建立上游缓存计费快照：

- `billable_input_tokens`：按全价输入价计费的输入 token。
- `upstream_cache_read_input_tokens`：上游缓存读取 token。
- `billable_cache_read_input_tokens`：应用 `token_multiplier` 后用于计费和主展示的缓存读取 token。
- `upstream_cache_creation_input_tokens`：上游缓存创建 token，仅展示，不单独加价。
- `input_cost`：普通输入成本。
- `cache_read_cost`：缓存读取成本，按输入价格 10%。
- `output_cost`：输出成本。
- `total_cost`：三者合计。

OpenAI Chat Completions：

- 从 `usage.prompt_tokens_details.cached_tokens` 提取缓存读取。
- `billable_input_tokens = max(prompt_tokens - cached_tokens, 0)`。
- 若响应包含 `prompt_tokens_details` 且 `cached_tokens == 0`，将 `prompt_tokens` 作为缓存创建展示值，用于 CPA 8317 首次创建场景。
- 若响应不包含 `prompt_tokens_details`，不展示缓存创建/读取。

Anthropic Messages：

- 沿用现有 `usage.cache_read_input_tokens`、`usage.cache_creation_input_tokens` 解析。
- `billable_input_tokens = usage.input_tokens`。
- `cache_read_cost = cache_read_input_tokens * input_price * 10%`。

### 2. 取消内部缓存处理和主展示

不删除内部分析表和字段，避免破坏历史数据与调试能力；但新请求链路不再执行内部请求体分段缓存分析，不再读写 Redis 分段缓存，也不再生成 `PARTIAL/读段/建段/复用` 这类内部缓存结果。

实施方式：

- `CacheMiddleware.wrap_request` 改为纯透传上游请求。
- `StreamCacheMiddleware.wrap_stream_request` 改为纯透传上游流。
- WebSocket Responses 中直接调用 `RequestBodyCacheService.analyze_request` 的路径改为不生成内部缓存信息。
- `RequestCacheSummaryService` 只继续承接真实上游缓存和会话状态信息，不再持久化内部缓存分段明细。

前端展示只基于真实上游缓存字段：

- `upstream_cache_read_input_tokens`
- `upstream_cache_creation_input_tokens`
- `upstream_prompt_cache_status`

会话压缩仍可保留为独立调试信息，但不作为缓存扣费依据。

### 3. 数据库变更

给 `request_log` 和 `consumption_record` 增加：

- `billable_input_tokens`
- `billable_cache_read_input_tokens`
- `cache_read_cost`
- `input_price_per_million_snapshot`
- `output_price_per_million_snapshot`
- `price_multiplier_snapshot`
- `token_multiplier_snapshot`

这些字段用于账单详情稳定展示历史公式，避免后续模型价格变化影响历史账单。

### 4. 前端展示

用量区域：

- 显示 `输入 billable_input_tokens`
- 显示 `输出 output_tokens`
- 有缓存创建时显示 `缓存创建 upstream_cache_creation_input_tokens`
- 有缓存读取时显示 `缓存读取 billable_cache_read_input_tokens`
- 合计显示计费 token 口径：`输入 + 输出 + 缓存读取`

计费详情：

- 显示普通输入公式。
- 显示输出公式。
- 有缓存读取时显示缓存读取公式：`缓存读取 token * 输入价格 * 10%`。
- 有缓存创建时显示“缓存创建不计费”。

1. 新增数据库字段和 ORM 字段。
2. 增加统一的 CPA/OpenAI/Anthropic 上游缓存用量提取与计费辅助逻辑。
3. 修改文本请求成功记账逻辑，按普通输入、缓存读取、输出分别计费。

5. 修改 admin/user/agent 相关页面，用真实上游缓存字段展示用量和计费详情，移除内部缓存主展示。
6. 补充或调整后端测试，覆盖 OpenAI `cached_tokens` 与 Anthropic `cache_read_input_tokens` 的扣费口径。
7. 运行后端编译/测试和前端构建检查。
