### 评审结论

当前方案 **可行**，而且方向正确，适合作为代理端落地基线。

但这不是一个“加个代理角色、复制几张后台页面”的中型改动，而是一次标准的单租户到多租户代理体系升级。只有把下面 4 条主线一起做，方案才真正能上线：

1. 代理实体与归属模型
2. 代理库存与分发事务
3. 域名白牌与站点上下文
4. 查询隔离与权限边界

如果按当前修订后的方案推进，可以进入实施；如果删掉其中任意一条，后续大概率返工。

## 通过项

### 1. 现状判断准确

方案准确识别了当前系统的真实基础：

- 模型转发、健康检查、套餐模板、余额、图片积分、兑换码、日志等原子能力都已具备
- 真正缺的是代理这一层的租户归属、库存、白牌配置和查询隔离

这与当前代码现状一致：

- `SysUser.role` 仍按双角色使用
- 路由与前端状态只识别 `admin/user`
- `system_config` 仍是全局配置
- 注册、登录、API Key、日志查询都还没有 `agent_id` 维度

### 2. 技术路线正确

方案没有把“代理端”理解成单纯角色，而是理解成独立业务实体，这个判断是正确的。

推荐设计成立：

- 新增 `agent` 主表
- 用户通过 `agent_id` 归属代理
- 代理拥有自己的余额池、图片积分池、套餐库存
- 站点配置按代理维度管理
- 日志、兑换码、套餐记录按代理维度过滤

### 3. 白牌站点方案合理

方案采用：

- 一套前端构建产物
- 一套后端服务
- 按 `Host` 返回站点配置

这是当前项目最稳妥的落地方式。

优点：

- 不需要每个代理单独打包部署前端
- 后续统一升级成本低
- 与当前 Vue2 SPA + FastAPI 架构兼容

### 4. 实施拆解足够细

方案已经覆盖：

- 数据库结构
- 权限依赖
- 管理端新增页面
- 代理端 7 个菜单
- 用户端白牌化
- Nginx / Cloudflare / CORS 改造
- 测试与验收

To-Do 已经达到可以继续拆成实现阶段任务的粒度。

## 关键修订点

以下问题是评审时发现的关键点，当前版本方案已经补入，实施时必须严格按此执行。

### 1. 代理资产分发必须走统一事务编排层

当前系统里的：

- `BalanceService.recharge/deduct(...)`
- `ImageCreditService.recharge/deduct(...)`
- `SubscriptionService.activate_plan_subscription(...)`

都带各自的提交语义。

这意味着：

- 代理端不能直接在接口层顺序调用这些方法
- 否则非常容易出现“用户已加余额，但代理库存没扣成功”这类脏账

因此本方案新增 `AgentAssetService` 是必要的，不是可选优化。

### 2. API Key 必须增加域名归属校验

当前 `verify_api_key_from_headers(...)` 只校验 API Key 是否属于某个用户，并不关心请求从哪个域名进来。

若不补这层校验，会出现：

- 代理 A 用户拿自己的 API Key 去代理 B 域名调用
- 技术上请求仍能成功
- 但品牌归属、日志统计、代理运营边界都会失真

因此首版建议直接加上：

- `Host -> agent`
- `api_key.owner.agent_id -> agent`
- 不一致即拒绝

### 3. 登录/注册必须与站点上下文绑定

当前：

- `AuthService.register(...)`
- `AuthService.login(...)`

都没有站点上下文概念。

这意味着代理站点如果直接接入当前逻辑，会出现：

- 用户注册后不知道归属于哪个代理
- 同一账号可在错误站点登录

所以本次不能只新增代理后台接口，还必须改认证链路本身。

### 4. 前端三处假设必须同时改

当前前端不只是路由问题，还包括：

1. `router/index.js`
- 默认只有 admin/user 两套路由守卫

2. `store/index.js`
- 默认只识别 admin/user 状态

3. `Login.vue` / `Dashboard.vue` / `QuickStart.vue`
- 站点品牌与公告写死

这三处必须一起改，否则代理端会出现：

- 登录后跳错页面
- 品牌文案没切换
- API 接入地址仍显示平台主站地址

## 风险提示

### 1. 首版不要同时做“代理自定义渠道/模型”

当前方案明确把渠道、模型、真实映射继续留在平台侧，这是对的。

如果首版就放开代理自定义上游：

- 权限边界会立刻复杂化
- 成本、计费、故障、模型别名映射都会分叉
- 实施复杂度至少翻倍

### 2. 首版不要承诺“任意外部域名自动接入”

子域名白牌可做，且适合当前部署。

但自定义独立域名意味着：

- 证书签发
- Nginx 动态站点管理
- Cloudflare / DNS 指向
- CORS / 回源策略

这更适合作为第二阶段，不建议混入首版里一起做完。

### 3. 用户名/邮箱唯一性必须先定规则

当前系统是全局唯一。

若要改成按代理唯一：

- 注册逻辑要重写
- 登录逻辑要增加站点 + 账号联合查询
- 数据兼容复杂度显著上升

这不是小选项，必须在实施前拍板。

1. 数据结构与租户上下文
