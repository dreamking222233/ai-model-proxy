## 评估结论

当前方案总体可行，且与现有系统实现方向一致，可以进入第一阶段开发。但需要明确这是“单代理单前台域名”的最小可行版本，不应在本轮实现里直接扩展为多域名体系，否则会把 CORS、后台管理、支付回跳和登录归属校验同时放大为结构性改造。

## 通过项

1. 共享 API `api.xiaoleai.team` + 代理前台自定义域名的架构判断正确，当前系统的代理识别确实基于 `X-Site-Host / Origin / Referer / Host`，不是写死二级域名。
2. 当前真正的第一阻塞点是 CORS，这个判断准确。
3. 支付回跳、注册来源、登录归属校验都可以沿用现有主链路，不需要推翻支付或认证模块。

## 需要修订的关键点

1. `frontend_domain` 不能只写“允许自定义域名”，还必须补充“输入规范化与校验规则”。
当前后端的 `normalize_host()` 只适合处理 Host，不适合直接处理后台录入值；如果管理员输入 `https://test.test.com`，旧逻辑会错误归一化为 `https`。因此第一阶段必须增加：
- 去协议头
- 去端口
- 禁止路径、查询串、片段
- 统一小写
- 禁止与平台前台/API 域名冲突

2. 动态 CORS 方案要补充“缓存刷新策略”。
如果域名白名单来自数据库，则代理域名新增/修改后，不能等缓存自然过期才能生效。建议在代理创建/更新成功后主动刷新 CORS 缓存。

3. 需要明确“数据库本轮是否改结构”。
按当前需求，第一阶段不需要新增表或改索引，`agent.frontend_domain` 的长度和唯一索引已经足够支撑一个代理一个前台域名。方案里应明确：
- 本轮无结构性数据库变更
- 仅有数据录入要求变化
- 多域名表作为第二阶段预案

4. 需要补充“切换域名的业务影响”。
如果代理已存在用户，修改 `agent.frontend_domain` 不会自动回写历史用户 `source_domain`，这不是阻塞，但要在方案里注明：
- 历史用户来源展示保留旧值
- 登录归属按 `agent_id` 判断，不依赖 `source_domain`
- 支付新订单回跳按新域名生效

5. Nginx 部署说明要从“通配子域名模板”扩展到“自定义域名单独 server block”。
对于 `test.test.com` 这种第三方域名，不能复用 `*.xiaoleai.team` 通配证书配置，必须单独配置：
- `server_name test.test.com`
- 独立证书
- 同样反代到现有前端入口

## 第一阶段推荐任务范围

1. 后端新增代理前台域名输入规范化与合法性校验。
2. 后端新增数据库驱动的动态 CORS 白名单中间件。
3. 代理新增/更新后刷新动态 CORS 缓存。
4. 管理端代理管理页增加域名填写说明与前端校验。
5. 文档补充：无需改表、Nginx 自定义域名配置、证书要求、上线验证清单。

## 暂不建议纳入本轮的内容

1. `agent_frontend_domain` 多域名表
2. 域名审核流/启停状态
3. 证书状态探测
4. 多域名灰度切换

## 风险提醒

1. 如果代理前台部署在 CDN 后，必须确认 `Origin` 与浏览器真实访问域一致，且前端请求层继续带 `X-Site-Host`。
2. 动态 CORS 中间件要正确处理 `OPTIONS` 预检请求，否则浏览器仍会在正式请求前失败。
3. 后端需要避免把平台域名误绑定到代理上，否则管理员访问平台时可能被错误识别为代理站点。
