## 评估结论

当前方案总体可行，且与现有系统“统一充值订单 + 渠道侧入账 + 代理分润复用”的方向一致，可以进入实施阶段。但本次改造的真正难点不在前端二维码，而在支付服务抽象、微信 V3 验签解密和订单字段扩展。若不先把这些基础能力设计稳，后续会出现支付宝与微信逻辑互相污染、支付明细字段混乱、查单接口难以维护的问题。

## 通过项

1. 只接入微信 Native 支付的范围控制正确，符合当前 PC 充值场景。
2. 保留统一订单表 `payment_recharge_order` 的方向正确，避免重复建表。
3. 代理现金分润和用户余额入账继续复用统一主链路，这个判断正确。
4. 前端以“二维码 + 轮询 + 异步通知为准”作为微信支付交互方式，符合 Native 支付特性。

## 需要修订的关键点

### 1. `PaymentService` 不应继续以“支付宝主类 + 微信补丁”方式扩展

当前 `payment_service.py` 已大量写死支付宝概念：

- `PAY_CHANNEL = "alipay"`
- `ACCEPTED_TRADE_STATUSES` 仅适配支付宝
- `_validate_payment_config()` 仅校验支付宝配置
- `_apply_paid_order()` 内部校验 `app_id`、`trade_status`、`alipay_trade_no`
- `ConsumptionRecord.model_name` 固定写成“支付宝充值”

因此本轮应先抽象出统一入口和渠道分发，而不是直接在现有方法上塞微信分支。建议最少做到：

1. 统一创建订单入口 `create_recharge_order(..., payment_channel=...)`
2. 统一同步入口 `sync_order_status(...)`
3. 统一入账入口 `_apply_paid_order(...)` 增加渠道参数或渠道分支
4. 渠道侧私有实现：支付宝 / 微信分别处理创建、查单、通知

### 2. 数据库字段不要只加微信字段，还要同步“通用展示字段”策略

当前订单表已有 `alipay_trade_no`，而管理端页面也直接展示“支付宝流水号”。

如果本轮只新增 `wechat_transaction_id` 而不统一序列化与展示层，会导致：

- 管理端只能看支付宝流水号列
- 代理端明细也无法统一展示渠道流水
- 导出/后续统计字段难统一

建议本轮最少新增：

1. 数据层：`wechat_transaction_id`、`wechat_code_url`
2. 序列化层：增加通用 `channel_trade_no`
3. 展示层：列名改成“支付流水号”，按渠道显示对应值

### 3. 微信通知验签不能只依赖平台公钥字符串，建议优先支持平台证书文本

方案里提到 `WECHAT_PAY_PLATFORM_CERT` 或平台公钥。对现阶段实现来说，优先建议支持：

- 平台证书 PEM 文本

原因：

1. 微信通知头里会带 `Wechatpay-Serial`
2. 正常实现更接近“按证书序列号匹配平台证书”
3. 以后若平台证书轮换，也比只存裸公钥更稳妥

如果本轮为了控制复杂度，只支持单一平台证书文本也是可以的，但要在文档里明确边界。

### 4. 微信支付成功入账不能复用支付宝的 `app_id` / `trade_status` 校验逻辑

当前 `_apply_paid_order()` 中有明显支付宝专属判断：

- 校验 `app_id == ALIPAY_APP_ID`
- `TRADE_SUCCESS / TRADE_FINISHED`
- `alipay_trade_no` 冲突检查

微信支付需要单独映射：

- `appid == WECHAT_PAY_APP_ID`
- `mchid == WECHAT_PAY_MCH_ID`
- `trade_state == SUCCESS`
- `transaction_id` 唯一性检查

因此入账层必须显式区分渠道，否则会产生错误校验。

### 5. 用户端“创建订单后自动弹二维码”要考虑旧订单复用与重复创建风险

当前充值页每次点击就是新建订单。微信 Native 下如果用户多次点击：

- 会生成多个待支付订单
- 轮询状态容易混乱
- 同金额下可能扫码错单

建议本轮在体验上至少做到：

1. 创建成功后把当前订单锁定为当前处理单
2. 二维码弹窗内可重新展示当前订单二维码
3. 不自动复用历史待支付订单，但要确保前端始终围绕“当前订单”轮询

### 6. 订单关闭状态映射要明确，避免把“用户未支付”当成失败

微信查询常见状态：

- `NOTPAY`
- `USERPAYING`
- `SUCCESS`
- `CLOSED`
- `REVOKED`
- `PAYERROR`

建议映射：

- `SUCCESS` -> `paid`
- `NOTPAY` / `USERPAYING` -> `pending`
- `CLOSED` / `REVOKED` -> `closed`
- `PAYERROR` -> `failed`

并在方案中明确：只有明确失败或关闭时才改本地终态，避免过早结束订单。

## 实施可行性建议

推荐实施顺序：

1. 先扩展订单模型和请求入参，打通“按渠道创建订单”的统一接口。
2. 再实现微信后端最小链路：下单、查单、通知验签解密。
3. 然后改前端充值页为双支付方式，并新增二维码弹层。
4. 最后统一支付明细展示、测试与部署文档。

这样可以降低“前端已切微信，但后端状态链路还不完整”的风险。

## 潜在风险

1. 微信通知验签与回调资源解密实现错误，会导致支付成功但无法入账。
2. 公网通知地址若不可达或证书链异常，微信支付只能靠用户端轮询查单兜底。
3. 若 `payment_service.py` 重构不彻底，容易引入支付宝回归问题。
4. 充值明细、后台查询、代理端展示如果只改后端不改序列化字段，会出现“支付平台是微信，但流水号列空白”的问题。
5. 前端二维码组件若选型不当，可能引入体积过大或兼容性问题；建议用轻量库。

## 结论

方案可以实施，但建议明确以下修订结论后再编码：

1. 本轮必须先把支付服务改成“统一入口 + 渠道实现”。
2. 本轮数据库字段除 `wechat_transaction_id`、`wechat_code_url` 外，还要同步展示层通用流水号策略。
3. 本轮前端必须以“异步通知为准，轮询兜底”为核心设计，不依赖回跳。
4. 本轮文档必须明确微信支付所需配置项、证书要求、通知地址和 SQL 升级步骤。
