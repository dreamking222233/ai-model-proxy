# 代理充值结算比例调整 Plan

## 用户原始需求

当前系统分为用户端、管理端和代理端：

- 用户通过 `/user/recharge` 页面充值，用户充值到账比例固定为 `1:5`。
- 代理名下用户充值时，会给代理产生现金分润。
- 现有代理结算比例为 `1:10`，因此在用户 `1:5` 到账时，代理分润相当于充值金额的 50%。
- 管理端 `/admin/payment-orders` 可查看用户充值记录和代理分润明细。
- 管理端 `/admin/agent-cash` 可查看代理提现余额。
- 代理端 `/agent/payments` 可查看其名下用户充值情况和分润情况。

本次需要将代理结算比例从 `1:10` 调整为 `1:7`，并同步调整其名下用户充值产生的代理分润。

## 当前实现分析

### 核心计算位置

充值订单创建时，后端 `PaymentService._calculate_amounts()` 计算并快照以下字段到 `payment_recharge_order`：

- `credited_usd`：用户到账美元余额。
- `credited_image_credits`：用户到账图片积分。
- `agent_settlement_rate`：代理结算倍率快照。
- `agent_income_cny`：代理现金分润金额。

支付成功后，`PaymentService._apply_paid_order()` 会：

1. 给用户资产入账。
2. 调用 `_credit_agent_cash_balance()`，将订单中的 `agent_income_cny` 写入：
   - `agent_cash_balance.balance`
   - `agent_cash_balance.total_income`
   - `agent_cash_ledger`

管理端和代理端页面展示的分润、提现余额、累计分润，均读取订单和现金余额表中的落库结果，不在前端重新计算比例。

### 当前配置来源

后端默认配置在 `backend/app/config.py`：

- `RECHARGE_USER_CNY_TO_USD_RATE=5`
- `RECHARGE_AGENT_CNY_TO_USD_SETTLEMENT_RATE=10`
- `RECHARGE_IMAGE_CREDIT_USER_CNY_RATE=5`
- `RECHARGE_IMAGE_CREDIT_AGENT_CNY_RATE=10`

本地运行配置 `backend/.env` 当前也显式覆盖：

- `RECHARGE_USER_CNY_TO_USD_RATE=5`
- `RECHARGE_AGENT_CNY_TO_USD_SETTLEMENT_RATE=10`

因此只改代码默认值不足以影响当前环境，必须同步更新运行环境变量。

### 当前分润公式

余额充值：

```text
credited_usd = amount_cny * RECHARGE_USER_CNY_TO_USD_RATE
agent_cost_cny = credited_usd / RECHARGE_AGENT_CNY_TO_USD_SETTLEMENT_RATE
agent_income_cny = amount_cny - agent_cost_cny
```

当前 `用户 1:5，代理 1:10`：

```text
充值 10 元
用户到账 50 USD
代理成本 50 / 10 = 5 元
代理分润 10 - 5 = 5 元
```

调整为 `用户 1:5，代理 1:7` 后：

```text
充值 10 元
用户到账 50 USD
代理成本 50 / 7 = 7.14 元
代理分润 10 - 7.14 = 2.86 元
```

即代理分润比例约为充值金额的 `2/7 = 28.57%`，不是固定 50%。

## 技术方案设计

### 目标

将代理结算倍率从 `10` 调整为 `7`，保持用户到账倍率 `5` 不变，使后续代理站用户在线充值生成的订单、分润明细、代理现金余额均按新规则计算。

### 范围判定

本次建议覆盖 `/user/recharge` 中的按量充值：

1. 余额充值：`RECHARGE_AGENT_CNY_TO_USD_SETTLEMENT_RATE` 从 `10` 改为 `7`。
2. 图片积分充值：同步将 `RECHARGE_IMAGE_CREDIT_AGENT_CNY_RATE` 从 `10` 改为 `7`，保持同一充值页下余额和图片积分的代理拿货规则一致。

套餐在线购买不走该倍率，当前由套餐售价与代理拿货价决定：

```text
subscription_agent_rebate_cny = sale_price_cny - agent_cost_price_cny
```

本次不调整套餐逻辑。

### 历史数据处理原则

默认不重算历史订单和历史代理现金余额。

原因：

- 已支付订单的 `agent_settlement_rate` 和 `agent_income_cny` 是支付时规则快照。
- `agent_cash_balance` 与 `agent_cash_ledger` 已用于提现余额和财务流水。
- 直接回改历史已支付订单会影响已提现、已对账数据，存在财务风险。

如业务明确要求“历史未提现部分也按 1:7 重算”，应另起数据迁移方案，按订单、现金流水、提现状态逐笔对账后处理。

### 后端方案

1. 修改默认配置：
   - `backend/app/config.py`
   - `RECHARGE_AGENT_CNY_TO_USD_SETTLEMENT_RATE: Decimal = Decimal("7")`
   - `RECHARGE_IMAGE_CREDIT_AGENT_CNY_RATE: Decimal = Decimal("7")`
2. 修改运行环境配置：
   - `backend/.env`
   - `RECHARGE_AGENT_CNY_TO_USD_SETTLEMENT_RATE=7`
   - `RECHARGE_IMAGE_CREDIT_AGENT_CNY_RATE=7`
3. 保持 `PaymentService._calculate_amounts()` 公式不变，仅让其读取新配置。
4. 保持 `payment_recharge_order.agent_settlement_rate` 快照逻辑不变，用于后续审计能区分老订单 `10` 和新订单 `7`。
5. 保持 `AgentCashService` 汇总逻辑不变，因为它按已落库 `agent_income_cny` 聚合，无需前端或查询层重算。

### 前端方案

用户端 `/user/recharge` 当前展示的是用户到账比例 `1人民币 = 5美刀 / 5图片积分`，用户比例不变，因此不需要改用户端文案。

管理端 `/admin/payment-orders`、`/admin/agent-cash` 和代理端 `/agent/payments` 当前展示落库的 `agent_income_cny`，不重新计算比例，因此不需要改页面展示逻辑。

如希望运维或财务能直观看到订单采用的代理结算倍率，可作为增强项在 `/admin/payment-orders` 增加 `agent_settlement_rate` 列；但这不是本次改比例的必要条件。

### 配置发布方案

生产发布时必须确认实际进程读取的环境变量为 `7`：

```env
RECHARGE_AGENT_CNY_TO_USD_SETTLEMENT_RATE=7
RECHARGE_IMAGE_CREDIT_AGENT_CNY_RATE=7
```

如果生产使用 systemd、pm2、Docker、面板环境变量或服务器本地 `backend/.env`，需要同步更新对应配置并重启后端服务。

## 验收标准

1. 用户充值到账比例仍为 `1:5`。
2. 新创建的代理用户余额充值订单：
   - `agent_settlement_rate = 7`
   - `agent_income_cny = amount_cny - (amount_cny * 5 / 7)`，按人民币 2 位小数四舍五入。
3. 新创建的代理用户图片积分充值订单：
   - `agent_settlement_rate = 7`
   - `agent_income_cny = amount_cny - (amount_cny * 5 / 7)`，按人民币 2 位小数四舍五入。
4. 代理用户套餐订单仍按 `subscription_agent_rebate_cny = sale_price_cny - agent_cost_price_cny` 计算返现，不受本次 1:7 调整影响。
5. 支付成功后代理现金余额增加金额等于订单 `agent_income_cny`。
6. 管理端和代理端展示的充值明细、代理分润、提现余额与数据库落库金额一致。
7. 历史订单仍保留原 `agent_settlement_rate=10` 与原分润金额，不被本次修改自动影响。

## 风险点与注意事项

1. 运行环境变量优先级高于 `config.py` 默认值，生产必须同步改环境变量并重启服务。
2. 历史订单不重算会导致同一页面中老订单分润约 50%，新订单分润约 28.57%，这是按订单规则快照的预期结果。
3. 若业务要求历史订单统一口径，需要单独做迁移与对账，不能直接批量覆盖 `agent_income_cny`。
4. 套餐购买入口在 `/user/recharge`，但其返现使用套餐售价与代理拿货价，不使用余额/图片积分的 1:7 代理倍率。
