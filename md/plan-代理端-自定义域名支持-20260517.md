## 一、当前系统实现现状分析

### 1.1 当前代理识别不是“硬编码必须二级域名”

当前系统识别代理站点，核心并不是判断 `*.xiaoleai.team`，而是基于请求中的站点 Host 信息解析代理：

- 前端请求会自动带上 `X-Site-Host=window.location.host`
  - 文件：`frontend/src/utils/request.js`
- 后端会优先从以下来源解析当前站点：
  - `X-Site-Host`
  - `Origin`
  - `Referer`
  - 请求 `Host`
  - 文件：`backend/app/services/agent_service.py`

因此，理论上前台域名只要能稳定传递 Host，上层业务逻辑并不关心它是不是 `xiaoleai.team` 的子域名。

### 1.2 当前代理绑定域名模型较简单

当前 `agent` 表只有单个前台域名字段：
- `frontend_domain`
- `api_domain`

对应 ORM：
- `backend/app/models/agent.py`

这意味着当前系统只支持：
- 一个代理绑定一个前台域名
- 且这个域名默认直接存到 `agent.frontend_domain`

如果要支持“自定义域名”，最小可行方案是继续使用这一个字段；
如果要支持“一个代理多个域名”“切换主域名”“保留历史域名”，则必须扩展数据模型。

### 1.3 当前 CORS 只放行平台域名和本地域名

当前后端配置：
- `CORS_ORIGINS` 仅包含 `localhost`、`www.xiaoleai.team`、`api.xiaoleai.team`
- `CORS_ORIGIN_REGEX` 仅允许：
  - `*.xiaoleai.team`
  - `localhost / 127.0.0.1 / *.local`

文件：
- `backend/app/config.py`

这意味着：
- 如果代理站点直接改成 `https://test.test.com`
- 前端请求 `https://api.xiaoleai.team`
- 浏览器会发起跨域请求
- 当前后端大概率会被 CORS 拦截

所以，当前真正的阻塞点不是代理识别，而是跨域白名单策略。

### 1.4 当前登录归属校验与共享 API 模式兼容

当前登录态和账号归属判断逻辑：
- 平台用户只能访问平台站点
- 代理用户只能访问自己所属代理站点
- 如果请求命中了共享 API 直连上下文，则允许 API 域下访问

文件：
- `backend/app/services/agent_service.py`

这说明：
- 只要前台域名识别正确
- 代理账号仍然可以在自定义域名上保持“归属校验”正确

### 1.5 当前支付回跳逻辑天然支持“按站点回跳”

当前支付宝回跳 URL 生成逻辑：
- 优先从站点上下文 `origin/referer/host` 构造回跳地址
- 否则 fallback 到订单里的 `source_host`

文件：
- `backend/app/services/payment_service.py`

这意味着：
- 代理站如果使用 `test.test.com`
- 支付后理论上可以直接回跳 `https://test.test.com/user/recharge?...`

前提是：
- 创建订单时传上来的站点上下文正确
- 后端能把该 Host 识别成当前代理站点

## 二、目标架构方案

### 2.1 目标结论

推荐架构：

- 代理前台域名：支持使用自定义域名，例如 `https://test.test.com`
- 统一后端 API：继续使用 `https://api.xiaoleai.team`
- 前端通过跨域调用共享 API
- 通过 `X-Site-Host` / `Origin` / `Referer` 识别当前代理站点

这是与当前系统现有实现最兼容、改动成本最可控的方案。

### 2.2 不建议的方案

不建议为每个代理再配置独立 API 域名，例如：
- `api.agent-a.com`
- `api.agent-b.com`

原因：
- 证书与 Nginx 维护成本明显上升
- 支付回调、CORS、登录归属链路更复杂
- 当前系统已经具备共享 API + 前台多域名识别的基础，不需要额外加一层 API 域名复杂度

## 4.3 CORS 改造方案

这是本次改造的核心难点。

### 当前问题

当前 CORS 规则是静态的，仅支持：
- `*.xiaoleai.team`
- 本地开发域名

这无法覆盖任意第三方自定义域名，例如：
- `test.test.com`

### 推荐方案

新增“动态 CORS 白名单判断”能力。

#### 推荐实现 1：后端动态 CORS 中间件

思路：
- 不再只依赖静态 `CORSMiddleware` 正则
- 新增自定义中间件
- 从 `Origin` 解析 Host
- 判断是否属于：
  - 平台域名
  - 本地域名
  - 已启用的代理前台域名
- 如果匹配，则回写：
  - `Access-Control-Allow-Origin`
  - `Access-Control-Allow-Credentials`
  - `Access-Control-Allow-Headers`
  - `Access-Control-Allow-Methods`

优点：
- 支持任意代理自定义域名
- 与数据库绑定，最灵活

注意点：
- 建议增加缓存，避免每次预检请求都查数据库
- 可做 5~10 分钟内存缓存

#### 推荐实现 2：Nginx 层动态回写 Origin

如果后端改动希望最少，也可以在 Nginx 层做：
- 根据 `Origin` 判断是否属于已备案/已录入代理域名
- 动态写回 `Access-Control-Allow-Origin`

缺点：
- 与数据库同步成本较高
- Nginx 配置维护更复杂

### 最终建议

优先使用后端动态 CORS 中间件，Nginx 仅做转发层。

## 五、风险点与边界说明

### 风险 1：CORS 是实际第一阻塞点

即使前台域名能正确识别，若 `api.xiaoleai.team` 没放行该域名：
- 浏览器仍会拦请求

### 风险 2：单字段模型不适合长期多域名运营

如果只改 `frontend_domain`：
- 后续多域名切换会比较痛苦

### 风险 3：重复域名绑定冲突

必须保证：
- 一个域名只能归属一个代理

否则：
- 登录归属
- 支付回跳
- 注册来源

都会歧义。

### 风险 4：支付回跳与来源归属必须一起验证

自定义域名上线后，不只是首页访问要成功：
- 登录
- 注册
- 充值
- 支付回跳
- 刷新状态

都要做端到端测试。

### 风险 5：管理员访问限制逻辑不能被破坏

当前管理员只能进入平台站点。

自定义域名接入后，不应让管理员误入代理站点时出现异常越权逻辑。
