# 价格调控实施方案

## 用户原始需求

深度阅读当前目录下的项目及相关文档。

当前管理端 `admin/config` 可以管理价格倍率，但该倍率作用于所有模型。系统包含 GPT、Claude、Grok、Gemini 等不同模型，包含文本对话、图片生成、视频生成等类型，且存在按 Token、按请求次数、按媒体积分等不同计费方式。

需要新增：

- 后台 `admin/models` 页面支持模型归类，例如 GPT、Claude、Grok、Gemini 系列，以及对话/图片/视频类型。
- 后台新增“价格调控”页面，可按模型系列和模型类型设置最终计费倍率，例如 GPT 文本对话 x2、Claude 文本对话 x1。
- 价格规则支持长期或短期/定时效果，例如每天 23:00 到次日 07:00 GPT 文本对话 x1.2，其余时间 x2，Claude 全天 x2。
- 全部时间按北京时间。
- 修复 `admin/promotions`、`user/promotion`、`user/asset-source`、`agent/promotions` 页面都按北京时间显示。

## 任务级别

大型任务。

原因：涉及数据库结构、后端模型管理、计费核心链路、管理端新页面、模型管理页、路由菜单、初始化脚本与验证，步骤超过 10 个。

## 技术方案设计

### 1. 数据模型设计

#### `unified_model` 增加模型系列字段

新增字段：

- `model_series VARCHAR(32) NOT NULL DEFAULT 'other'`
- 含义：模型系列归类，建议值为 `gpt`、`claude`、`grok`、`gemini`、`other`。

兼容策略：

- 新增字段默认 `other`，不影响旧数据。
- 提供升级 SQL 根据 `model_name` 自动回填常见系列。
- `admin/models` 支持手动调整。
- 用户端模型列表可直接返回该字段，前端优先使用该字段分类；没有字段时继续按模型名前缀推断。

#### 新增 `model_price_adjustment_rule`

新增表用于价格调控：

- `id BIGINT PRIMARY KEY`
- `name VARCHAR(128)`：规则名称
- `model_series VARCHAR(32)`：模型系列，支持具体系列或 `all`
- `model_type VARCHAR(20)`：模型类型，支持 `chat`、`image`、`video`、`embedding`、`completion` 或 `all`
- `billing_type VARCHAR(20)`：计费类型，支持 `token`、`request`、`image_credit`、`free` 或 `all`
- `multiplier DECIMAL(12,6)`：价格倍率
- `schedule_type VARCHAR(20)`：`always` 或 `daily_time`
- `start_time TIME NULL`：每日开始时间，北京时间
- `end_time TIME NULL`：每日结束时间，北京时间
- `priority INT NOT NULL DEFAULT 100`：优先级，数值越小越优先
- `enabled SMALLINT NOT NULL DEFAULT 1`
- `description TEXT NULL`
- `created_at DATETIME`
- `updated_at DATETIME`

规则匹配策略：

- 只匹配启用规则。
- 按 `model_series`、`model_type`、`billing_type` 精确或 `all` 匹配。
- `schedule_type=always` 全天有效。
- `schedule_type=daily_time` 按北京时间判断；支持跨天窗口，例如 23:00 到 07:00。
- 多条规则同时有效时，取 `priority ASC, id DESC` 第一条。
- 未命中规则时分类倍率为 `1`。

### 2. 计费链路设计

当前文本计费公式：

```text
total = base_price * price_multiplier * fast_multiplier * context_multiplier
```

调整为：

```text
total = base_price * global_price_multiplier * adjustment_multiplier * fast_multiplier * context_multiplier
```

兼容策略：

- 保留 `system_config.price_multiplier`，作为全局倍率。
- 新规则返回的是 `adjustment_multiplier`。
- 写入现有 `price_multiplier_snapshot` 时使用 `global_price_multiplier * adjustment_multiplier`，避免新增日志字段导致大量前端/日志接口连锁改造。
- Fast 倍率、长上下文倍率和 token 倍率保持现有逻辑。

覆盖范围：

- Token 计费文本模型。
- 按请求次数计费文本模型。
- 余额预检/套餐额度预检，避免预检与最终扣费不一致。
- 图片生成/编辑媒体积分扣费。
- 视频生成媒体积分扣费。

媒体积分计费调整：

```text
charged_credits = base_credits * adjustment_multiplier
```

说明：

- 图片/视频使用媒体积分，不使用 `system_config.price_multiplier` 的全局 USD 价格倍率，避免“美元倍率”误影响积分。
- 对图片/视频规则仍支持按 `model_series + model_type + billing_type` 调整。
- `image_credit_record.multiplier` 记录最终每张/每秒基础积分倍率，即已乘以调控倍率后的实际单价。

### 3. 后端 API 设计

新增管理端接口，建议前缀：

`/api/admin/price-adjustments`

接口：

- `GET /api/admin/price-adjustments`：分页查询规则，可按系列/类型/状态筛选。
- `POST /api/admin/price-adjustments`：新增规则。
- `PUT /api/admin/price-adjustments/{rule_id}`：更新规则。
- `DELETE /api/admin/price-adjustments/{rule_id}`：删除规则。
- `GET /api/admin/price-adjustments/options`：返回模型系列、类型、计费类型选项。
- `GET /api/admin/price-adjustments/effective`：返回当前各系列/类型的有效倍率概览。

新增服务：

- `backend/app/services/price_adjustment_service.py`
- 负责规则校验、CRUD、北京时间有效性判断、当前倍率解析。

模型管理扩展：

- `UnifiedModel` ORM、Pydantic schema、`ModelService._model_to_dict` 增加 `model_series`。
- `create_model` / `update_model` 支持保存 `model_series`。
- 用户端 `/api/user/models`、`/api/user/chat/models` 返回 `model_series`。
- 管理端 `/api/admin/models/chat/channels-models` 返回 `model_series`。

### 4. 前端设计

#### `admin/models`

在模型表格和编辑弹窗增加“模型系列”：

- 表格列：系列。
- 表单项：系列下拉，选项 `gpt`、`claude`、`grok`、`gemini`、`other`。
- 创建模型时可根据模型名前缀自动推断默认系列，但允许手动调整。

#### 新增 `admin/price-adjustments`

新增页面：

- 文件：`frontend/src/views/admin/PriceAdjustmentManage.vue`
- API：`frontend/src/api/priceAdjustment.js`
- 路由：`/admin/price-adjustments`
- 菜单：管理端侧边栏新增“价格调控”

页面能力：

- 规则列表：名称、系列、类型、计费类型、倍率、生效方式、时间段、优先级、状态、操作。
- 新增/编辑弹窗：规则名称、系列、类型、计费类型、倍率、长期/每日时间窗、开始时间、结束时间、优先级、启用、说明。
- 当前有效倍率概览：按系列 + 类型展示当前生效倍率，提示“北京时间”。

#### 用户端模型分类

`user/models`：

- 优先使用后端返回的 `model_series` 做厂商来源分类。
- 若无 `model_series`，保留原模型名前缀推断。

### 5. 北京时间修复

当前检查结果：

- `admin/promotions` 已使用 `formatBeijingTime`。
- `user/promotion` 已使用 `formatBeijingTime`。
- `agent/promotions` 已使用 `formatBeijingTime`。
- `user/asset-source` 使用 `formatUtcDate`，需要改为 `formatBeijingTime`。

同时复核 `frontend/src/utils/index.js`：

- `formatBeijingTime` 已使用 `Intl.DateTimeFormat` 和 `Asia/Shanghai`。
- `parseUtcDate` 会将无时区后端时间视为 UTC，再转换为北京时间，符合当前后端 `datetime.utcnow()`/数据库 UTC 语义。

## 风险点与处理

- 计费预检与最终扣费不一致：统一通过 `PriceAdjustmentService.resolve_multiplier` 解析倍率，预检和扣费都使用同一服务。
- 每日跨天窗口：`start_time > end_time` 时按 `now >= start OR now < end` 判断。
- 旧全局倍率兼容：文本 USD 计费继续保留 `price_multiplier`，新分类倍率与其相乘；图片/视频积分仅使用分类倍率。
- 日志快照字段不足：短期复用 `price_multiplier_snapshot` 存储“全局倍率 x 分类倍率”，保持现有账单展示兼容。
- 数据库初始化脚本多份：同步根目录发布入口 `sql/initData.sql`，并同步 `backend/sql/initData.sql`、`backend/sql/init.sql`，避免环境不一致。
- 时间显示：前端统一使用 `formatBeijingTime`，避免浏览器本地时区影响。
