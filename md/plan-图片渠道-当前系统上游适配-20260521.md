## 用户原始需求

当前需要新增一种图片上游支持场景：有的中转站本身就是用当前源码搭建的，再接入当前系统作为上游渠道时，图片生成与图片编辑请求需要兼容当前系统自身的图片接口，以完成适配。

涉及接口：

- `/v1/images/generations`
- `/v1/image/created`
- `/v1/image/edit`
- `/v1/images/edits`

## 技术方案设计

目标是在现有 OpenAI 图片渠道体系下，新增一种“当前系统图片上游”渠道类型，用来显式区分：

- 标准 OpenAI 兼容图片上游：优先走 `/v1/images/generations`、`/v1/images/edits`
- 当前系统作为图片上游：优先走 `/v1/image/created`、`/v1/image/edit`

实现方式：

1. 在 `ChannelService` 中新增 OpenAI 图片渠道子类型常量，并纳入标准化逻辑。
2. 在图片生成/编辑 URL 解析逻辑中，根据 `provider_variant` 选择上游路径。
3. 保持现有请求体/返回体兼容，不改动现有业务字段，只改上游路由选择。
4. 将新渠道类型纳入管理端配置、展示、能力说明与健康检查逻辑。
5. 增加后端测试，覆盖新渠道类型的生成/编辑 URL、能力、健康检查目标与 UI 显示相关分支。

设计取舍：

- 不单独新增新的协议类型，继续复用 `protocol_type=openai`，避免影响现有模型映射和渠道健康逻辑。
- 不修改现有四个代理入口的对外协议，只在“本系统作为上游”时切换目标 URL。
- 分辨率能力暂与 `openai-image-compatible` 保持一致，默认 1K，编辑支持为 true；后续如有需要再扩展“当前系统原生尺寸”渠道类型。
