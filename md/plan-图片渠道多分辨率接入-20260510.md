## 用户原始需求

当前项目已经提供以下图片接口：

- `POST /v1/images/generations`
- `POST /v1/image/created`
- `POST /v1/image/edit`
- `POST /v1/images/edits`

当前接入的图片模型里，`gpt-image-2` 默认只按 `1K` 处理，默认扣费为 `0.5` 图片积分 / 次。

现在需要接入一个新的 OpenAI 兼容图片渠道。该上游：

- 图片生成接口：`POST /v1/images/generations`
- 图片编辑接口：`POST /v1/images/edits`
- 固定上游模型：`gpt-image-2`
- 支持通过 `size` 直接控制分辨率与比例
- 需要支持 `1K / 2K / 4K`

业务计费要求：

- `1K`：`0.5` 图片积分 / 次
- `2K`：`1` 图片积分 / 次
- `4K`：`2` 图片积分 / 次

目标是保持当前用户侧接口不变，在系统内部新增渠道能力、分辨率能力和计费规则。

## 当前项目现状分析

### 1. 图片接口层已经统一

入口集中在：

- `backend/app/api/proxy/image_proxy.py`

特点：

- 生成接口统一走 `ProxyService.handle_image_request()`
- 编辑接口统一走 `ProxyService.handle_image_edit_request()`
- 用户侧已有 `/v1/images/generations`、`/v1/image/created`、`/v1/image/edit`、`/v1/images/edits`

因此本次无需新增用户侧 API，只需扩展代理实现。

### 2. 图片转发层已经支持多种渠道，但 OpenAI 图片链路仍分两类语义混用风险

核心逻辑集中在：

- `backend/app/services/proxy_service.py`
- `backend/app/services/channel_service.py`
- `backend/app/services/health_service.py`

当前已支持：

- Google 官方图片链路
- Google Vertex 图片链路
- OpenAI 图片兼容链路

但现有 `openai-image-compatible` 实现本质上是：

- 不向上游传 `size`
- 不向上游传 `image_size`
- 通过 prompt 追加“比例/清晰度提示”来兼容旧渠道

这和本次新上游“原生支持 `size` 参数”的能力不同。如果继续复用旧子类型，后续会出现：

- 某些 OpenAI 图片渠道需要 prompt 适配
- 某些 OpenAI 图片渠道需要原生透传 `size`
- 运行时无法准确区分

因此需要把老逻辑和新逻辑在渠道子类型上拆开。

### 3. 分辨率计费基础设施已存在，但当前只允许 Google 图片模型配置规则

当前已有：

- `model_image_resolution_rule` 表
- `ModelImageResolutionRule` ORM
- `ModelService.resolve_image_resolution_rule(...)`
- `ProxyService._resolve_image_billing_rule(...)`

但 `ModelService._validate_resolution_rules(...)` 当前限制为：

- `model_type = image`
- `protocol_type = google`
- `billing_type = image_credit`

同时前端 `frontend/src/views/admin/ModelManage.vue` 也只对 Google 图片模型展示分辨率计费配置。

这会导致：

- 后台无法为 `gpt-image-2` 配置 `1K / 2K / 4K`
- 运行时即便逻辑支持，也没有管理入口

### 4. 当前 `gpt-image-2` 的分辨率能力是“提示词兼容”，不是“原生像素尺寸”

`ModelService` 目前对 `gpt-image-2` 的定义是：

- `PROMPT_ADAPTED_IMAGE_SIZE_CAPABILITIES = ("512", "1K", "2K", "4K")`

`ProxyService._non_stream_openai_image_request()` 当前行为是：

- 只发送 `model/prompt/n/response_format`
- 不发送 `size`
- 把 `image_size` 和 `aspect_ratio` 拼进 prompt

这适合旧渠道，不适合本次新上游。

### 5. 图片编辑链路目前只允许 OpenAI 图片渠道

当前编辑接口：

- 只要 `protocol_type == openai` 就会进入 `_non_stream_openai_image_edit_request()`

这意味着本次新渠道如果仍归类为 OpenAI 图片渠道，可以在不改用户侧协议的前提下直接复用编辑入口，只需要调整上游表单字段构造即可。

## 技术方案设计

### 1. 新增 OpenAI 图片原生尺寸渠道子类型

新增一个新的 `provider_variant`，例如：

- `openai-image-native-size`

语义：

- `openai-image-compatible`：旧逻辑，依赖 prompt 适配尺寸与比例
- `openai-image-native-size`：新逻辑，直接向上游透传 `size`

好处：

- 兼容历史渠道，不破坏既有行为
- 新旧能力边界清晰
- 后续后台配置和健康检查都可按子类型分流

### 2. 将分辨率规则从“Google 专属”放宽为“图片模型能力驱动”

后端放宽规则配置条件：

- 只要是 `model_type = image`
- `billing_type = image_credit`
- 且模型在 `ModelService` 中声明了可配置分辨率能力

不再写死必须 `protocol_type = google`。

能力来源改为统一的模型能力表：

- Google 图片模型：继续使用原生 `imageSize`
- `gpt-image-2`：允许配置 `1K / 2K / 4K`

这样后台可直接给 `gpt-image-2` 配置：

- `1K = 0.5`
- `2K = 1`
- `4K = 2`

默认档位保持 `1K`。

### 3. 请求参数兼容策略

对用户侧继续兼容以下参数：

- `image_size`
- `imageSize`
- `size`
- `aspect_ratio`

解析规则建议：

1. 若 `image_size / imageSize / size` 传的是 `1K / 2K / 4K`，则视为“计费分辨率档位”
2. 若 `size` 传的是像素尺寸，例如：
   - `1024x1024`
   - `1024x1792`
   - `1792x1024`
   - `2048x2048`
   - `3840x2160`
   则需要从像素尺寸反推出分辨率档位：
   - 长边 `<= 1536` 归 `1K`
   - 长边 `<= 3072` 归 `2K`
   - 更高归 `4K`
3. 对原生尺寸渠道：
   - 继续把原始像素 `size` 透传给上游
4. 对旧 prompt 适配渠道：
   - 继续走 prompt 适配，不透传 `size`

这样可以同时满足：

- 新渠道使用像素级 `size`
- 老调用方继续传 `image_size`
- 计费按 `1K / 2K / 4K` 统一结算

### 4. OpenAI 图片生成链路按渠道类型分流

生成链路拆为两种行为：

- Prompt 适配型 OpenAI 图片渠道
- Native Size 型 OpenAI 图片渠道

Native Size 型向上游发送：

- `model`
- `prompt`
- `n`
- `size`
- `quality`
- `response_format=b64_json`

其中：

- `size` 优先透传用户原始像素值
- 若用户只传了 `1K / 2K / 4K`，则映射到默认像素尺寸：
  - `1K -> 1024x1024`
  - `2K -> 2048x2048`
  - `4K -> 4096x4096`
- `quality` 默认 `high`

### 5. OpenAI 图片编辑链路同步支持 Native Size 渠道

编辑链路仍复用 OpenAI 表单上传方式，但按渠道子类型决定：

- 旧渠道：继续 prompt 适配，不传 `size`
- 新渠道：向上游 multipart form 同步传 `size` 和 `quality`

这样可以让：

- `/v1/image/edit`
- `/v1/images/edits`

与新渠道能力保持一致。

### 6. 计费规则

最终计费规则：

- `1K`：`0.5`
- `2K`：`1.0`
- `4K`：`2.0`

执行方式：

1. 先解析出本次请求的有效分辨率档位
2. 读取 `model_image_resolution_rule`
3. 请求前按实际档位做余额校验
4. 上游成功后按实际档位扣费
5. 上游失败时 `image_credits_charged = 0`

### 7. 管理端同步放开

需要同步修改 `frontend/src/views/admin/ModelManage.vue`：

- 分辨率配置不再只对 Google 图片模型显示
- 改为对“支持分辨率能力的图片积分模型”显示
- 为 `gpt-image-2` 追加默认预设：
  - `1K = 0.5`
  - `2K = 1`
  - `4K = 2`

同时修改 `frontend/src/views/admin/ChannelManage.vue`：

- 增加 `openai-image-native-size` 渠道类型
- 增加对应提示文案

## 实施步骤概要

1. 梳理并抽象 OpenAI 图片渠道的两种能力：prompt 适配型、native size 型。
2. 扩展 `ChannelService`、`HealthService` 与渠道管理页，新增 `openai-image-native-size`。
3. 扩展 `ModelService`，允许 `gpt-image-2` 配置分辨率规则。
4. 扩展 `ProxyService`：
   - 解析像素 `size`
   - 归一化为 `1K / 2K / 4K`
   - 对 native size 渠道透传 `size/quality`
   - 生成与编辑都支持
5. 扩展前端模型管理页，放开 `gpt-image-2` 分辨率配置。
6. 补充单测，覆盖分流、size 映射、计费与编辑透传。
7. 形成 impl 文档并执行自审。
