# 套餐-额度24小时刷新 Plan

## 用户原始需求

当前系统存在套餐逻辑，管理端 `admin/subscription` 可以设置套餐、开通套餐，代理端 `agent/subscription` 和用户端 `user/recharge` 也可以开通套餐。

现有套餐通常是每日限额 50 美元、100 美元。当前疑似每天 0 点刷新额度，导致日卡用户如果当天下午 16:00 开通套餐，0 点前用完额度后，0 点后又获得一次额度，等于一张日卡使用出两张日卡效果。

需要优化为：

1. 套餐额度不能按北京时间自然日 0 点刷新。
2. 套餐开启时间例如 `16:30:31`，下一次刷新必须是 24 小时后的 `16:30:31`。
3. 每个用户的套餐额度刷新时间都根据该用户当前套餐的开通时间计算。
4. 管理端开通、代理端开通、用户端在线购买开通都必须遵循同一套 24 小时刷新逻辑。
5. 所有时间按北京时间计算。
6. 用户端 `user/balance` 的套餐卡片区域增加“下次刷新时间”展示。
7. 已开通套餐按用户开通套餐的时间计算后续每日额度刷新。

## 当前代码现状

### 开通入口

当前三类开通入口最终都复用套餐服务：

1. 管理端 `admin/subscription`
   - API：`backend/app/api/admin/subscription.py`
   - 模板发放：`SubscriptionService.activate_plan_subscription(...)`
   - 旧手动开通：`SubscriptionService.activate_subscription(...)`

2. 代理端 `agent/subscription`
   - API：`backend/app/api/agent/subscription.py`
   - 服务：`AgentAssetService.grant_user_subscription(...)`
   - 最终调用：`SubscriptionService.activate_plan_subscription(...)`

3. 用户端 `user/recharge` 在线购买套餐
   - API：`backend/app/api/user/payment.py`
   - 服务：`PaymentService._activate_subscription_order(...)`
   - 最终调用：`SubscriptionService.activate_plan_subscription(..., activation_mode="append")`

结论：只要统一修改 `SubscriptionService` 的周期计算，三类开通入口都会继承新规则。

### 当前周期计算问题

核心问题在 `backend/app/services/subscription_service.py`：

- `_get_cycle_window(now_local, tz_name)` 当前按 `reset_timezone` 所在自然日 0 点计算周期：
  - `target_start = target_now.replace(hour=0, minute=0, second=0, microsecond=0)`
  - `target_end = target_start + timedelta(days=1)`
- `_get_or_create_cycle(...)`、`_get_cycle_for_summary(...)` 都使用该函数。
- `subscription_usage_cycle` 当前按 `(subscription_id, cycle_date)` 唯一，`cycle_date` 是自然日日期。

这会导致 16:30 开通的日卡，在 00:00 后进入新自然日周期，提前刷新额度。

### 计费和余额扣费链路

文本请求扣费主要在 `backend/app/services/proxy_service.py`：

1. 请求前预检：`ProxyService._assert_text_request_allowed(...)`
   - 调用 `SubscriptionService.refresh_user_subscription_state(...)`
   - 当前有有效套餐时调用 `SubscriptionService.check_quota_before_request(...)`
   - 如果套餐额度不足，会尝试判断余额是否可兜底。

2. 请求后记账：`ProxyService._deduct_balance_and_log_once(...)`
   - 余额模式：直接从 `UserBalance` 扣除全额。
   - 套餐模式：解析当前有效套餐，调用 `SubscriptionService._get_or_create_cycle(...)` 获取当前周期。
   - 按套餐口径计算本次应消耗额度：
     - `cost_usd` 套餐按本次美元成本。
     - `total_tokens` 套餐按原始 Token。
   - 如果周期剩余额度足够：写入套餐周期用量，`billing_mode=subscription`。
   - 如果周期额度不足但用户余额可承担超出部分：套餐消耗剩余额度，余额扣超出部分，`billing_mode=mixed`。
   - 如果套餐额度不足且余额不足：抛 `INSUFFICIENT_BALANCE`。

结论：只要 `SubscriptionService._get_or_create_cycle(...)` 和相关 summary 使用 rolling 24h 周期，套餐扣费、混合扣费和余额兜底逻辑都会按新周期生效。

### 用户端展示

`frontend/src/views/user/BalanceLog.vue` 当前套餐卡片显示：

- 模式
- 到期
- 剩余

数据来源是 `getProfile()`，后端为 `AuthService.get_current_user_info(...)`，其中已经返回 `subscription_summary = SubscriptionService.get_current_subscription_summary(...)`。

因此新增刷新时间应优先从 `get_current_subscription_summary(...)` 增加字段，再在 `BalanceLog.vue` 中展示。

## 技术方案设计

### 总体方案

不新增数据库字段，不改开通入口，不改套餐模板结构。

利用现有 `user_subscription.start_time` 作为当前套餐的刷新锚点，以北京时间计算 rolling 24 小时周期：

- 第 1 个周期：`start_time` 到 `start_time + 24h`
- 第 2 个周期：`start_time + 24h` 到 `start_time + 48h`
- 依次类推
- 最后一个周期结束时间不超过 `subscription.end_time`

这样已经开通的套餐也可以直接按原始 `start_time` 生效，无需数据迁移。

### 时间规则

所有套餐刷新时间按北京时间 `Asia/Shanghai` 计算。

现有字段 `reset_timezone` 保留兼容，但新额度刷新逻辑固定使用 `SubscriptionService.DEFAULT_TIMEZONE = "Asia/Shanghai"`。原因：

1. 用户需求明确所有时间按北京时间。
2. 当前系统 `get_current_time()` 默认返回去掉 tzinfo 的北京时间。
3. 继续允许套餐模板配置其他 `reset_timezone` 会使刷新规则和用户需求冲突。

### 周期窗口计算

新增或替换为内部函数：

```python
def _get_subscription_cycle_window(subscription, now_local) -> tuple[date, datetime, datetime]:
    anchor = subscription.start_time
    end_time = subscription.end_time
    effective_now = now_local
    if end_time and effective_now >= end_time:
        effective_now = end_time - timedelta(microseconds=1)
    elapsed_seconds = max(0, (effective_now - anchor).total_seconds())
    cycle_index = int(elapsed_seconds // 86400)
    cycle_start_at = anchor + timedelta(days=cycle_index)
    cycle_end_at = anchor + timedelta(days=cycle_index + 1)
    if end_time and cycle_end_at > end_time:
        cycle_end_at = end_time
    cycle_date = cycle_start_at.date()
    return cycle_date, cycle_start_at, cycle_end_at
```

边界说明：

- 16:30:31 开通，则每天 16:30:31 才进入下一周期。
- 00:00 不再触发刷新。
- `now == end_time` 时不创建一个新的完整额度周期，避免到期瞬间出现额外额度。
- 日卡如果 `duration_days=1`，下一周期边界通常等于到期时间，因此不会产生可继续使用的第二周期。

### 周期表兼容策略

保留 `subscription_usage_cycle` 表结构和唯一键 `(subscription_id, cycle_date)`。

新规则下：

- `cycle_date` 使用 `cycle_start_at.date()`，表示 rolling 24h 周期的开始日期。
- `cycle_start_at`、`cycle_end_at` 才是精确周期边界。

需要处理旧自然日周期已经存在的情况：

1. 命中同一个 `cycle_date` 的旧周期时，如果 `cycle_start_at/cycle_end_at` 和新计算窗口不一致，必须重建快照。
2. 新周期不存在时，也不能直接 `used_amount=0`，必须先按新窗口从 `consumption_record` 聚合已有套餐消费，初始化 `used_amount/request_count/last_request_id`。
3. `IntegrityError` 并发兜底后重新查询到周期，也要执行同样的窗口校验和快照同步。

这样可以避免部署后已有用户在同一个 24h 窗口内，因为旧周期行日期不同而再次获得空额度。

### 快照重建策略

复用现有 `_rebuild_cycle_usage_snapshot(...)`，按以下条件触发：

- 周期行的 `quota_metric` 和当前套餐有效口径不一致。
- 周期行的 `cycle_start_at` 或 `cycle_end_at` 与 rolling 24h 计算结果不一致。
- 周期行额度快照需要更新。
- 旧兼容逻辑中 `cost_usd` 额度出现异常快照，需要重建。

重建范围：

- `ConsumptionRecord.created_at >= cycle_start_at`
- `ConsumptionRecord.created_at < cycle_end_at`
- 仍使用 `_subscription_usage_filter(subscription)` 限定当前套餐相关记录。

### Summary 和前端字段

后端 `SubscriptionService.get_current_subscription_summary(...)` 增加字段：

- `next_refresh_at`：当前周期的结束时间，格式 ISO 字符串。
- `refresh_period_hours`：固定 `24`。
- 可选 `refresh_anchor_at`：当前套餐 `start_time`，便于前端或调试确认刷新锚点。

`_default_subscription_summary()` 对应补充：

- `next_refresh_at: None`
- `refresh_period_hours: 24`
- `refresh_anchor_at: None`

`_serialize_cycle(...)` 也可以增加别名字段：

- `next_refresh_at = cycle.cycle_end_at.isoformat()`

前端 `frontend/src/views/user/BalanceLog.vue`：

- 在套餐卡片 `package-info` 中新增一项：
  - 标签：`下次刷新`
  - 值：`formatTime(subscriptionSummary.next_refresh_at || subscriptionSummary.current_cycle.cycle_end_at)`
- 显示完整北京时间日期时间，例如 `2026-06-25 16:30:31`。
- 如果没有 `next_refresh_at`，不展示该项。

### 文案调整

后端额度不足文案当前包含“请明天再试”。新规则不再是自然日刷新，建议改为更准确的周期语义：

- `已超出实际使用额度，每 24 小时最多可使用 {limit_text}，请在下个额度周期后重试`
- `当前套餐额度已用尽，请在下个额度周期后重试或联系管理员升级套餐`

这是可选但建议同步调整，避免用户误解为 0 点刷新。

## 风险点与规避

1. 旧周期行导致额度被清零
   - 规避：新建和命中周期都必须按 rolling 窗口聚合已有 `ConsumptionRecord`。

2. 并发请求同时创建周期
   - 规避：保留现有 nested transaction 和唯一键冲突处理；冲突后重查并同步快照。

3. 日卡到期瞬间出现新周期
   - 规避：`now >= end_time` 时使用 `end_time - 1 microsecond` 计算周期。

4. 前端读取接口不是 `/api/user/balance`
   - 规避：`BalanceLog.vue` 当前用 `getProfile()`，所以必须通过 `AuthService.get_current_user_info(...)` 返回的 `subscription_summary` 生效。

5. `reset_timezone` 历史配置不一致
   - 规避：需求指定北京时间，rolling 24h 逻辑固定 `Asia/Shanghai`，保留字段但不作为刷新依据。

## 验收标准

1. 任意入口开通的套餐都以该套餐 `start_time` 作为额度刷新锚点。
2. 0 点不会刷新套餐额度。
3. 满 24 小时后才进入新额度周期。
4. 老用户已有套餐按原始 `start_time` 自动适配新规则。
5. 用户余额页套餐卡片显示下次刷新时间。
6. 套餐额度不足时，余额兜底和混合扣费逻辑不退化。
7. 自动化测试覆盖关键周期和兼容场景。
