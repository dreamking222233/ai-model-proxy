# 套餐列表性能优化 Plan

## 用户原始需求

生产环境部署套餐 24 小时刷新逻辑后，`/api/admin/subscription/active-users?sort_order=asc&page=1&page_size=20` 很久没有反应，同时 `/api/user/profile` 响应速度也变慢。需要在确保套餐刷新、扣费逻辑正确的前提下优化响应速度。

## 问题定位

生产日志显示：

- `/api/admin/subscription/active-users` 多次耗时 30-42 秒。
- `/api/user/profile` 同时段偶发 4.8-6.1 秒。

代码路径中，列表和 profile 都会调用 `SubscriptionService._get_cycle_for_summary(...)`。此前该方法在周期行缺失或旧 0 点窗口不一致时，会即时扫描 `ConsumptionRecord` 重建快照；列表页每页 20 条会形成多次大表聚合，拖慢后端 worker，并连带影响 profile。

## 技术方案

1. 保留扣费写路径精确逻辑：
   - `_get_or_create_cycle(...)` 仍会在请求前/扣费时按 rolling 24h 周期创建或重建周期快照。
   - 不降低套餐额度校验和扣费准确性。

2. 优化展示读路径：
   - `_get_cycle_for_summary(...)` 默认只读 `subscription_usage_cycle`。
   - 如果周期缺失，返回轻量周期快照，`used_amount=0`。
   - 如果周期窗口不一致，默认使用已有周期用量但展示新窗口边界。
   - 需要精确明细的单条使用详情页显式传 `rebuild_from_consumption=True`。

3. 补 SQL 索引：
   - 新增 `idx_consumption_subscription_created_id(subscription_id, created_at, id)`。
   - 降低后续必要快照重建和详情查询成本。
