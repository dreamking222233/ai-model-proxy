# 套餐美元日限额改造 Plan

日期：20260509

## 用户原始需求

系统将在今晚 12 点后调整套餐每日额度口径，取消部分套餐原有的按 token 计量方式，改为按美元计量。

明确规则如下：

- 生效时间按 `2026-05-10 00:00:00 Asia/Shanghai` 处理。
- 无限月卡：
  - 已开通用户：`120 USD/天`
  - 新开通用户：`100 USD/天`
- 畅享套餐（日/周/月）：
  - 已开通用户：`60 USD/天`
  - 新开通用户：`50 USD/天`
- 模型倍率按官方价格 `1:1` 计算，不使用系统价格倍率。
- 缓存读取按输入价格的 `10%` 计入每日额度。

## 范围与假设

- 本次调整范围包括两类套餐：
  - `无限套餐` 日卡 / 周卡 / 月卡
  - `畅享` 日卡 / 周卡 / 月卡
- “已开通”按订阅记录 `created_at < 2026-05-10 00:00:00 Asia/Shanghai` 判定，避免顺延套餐因为 `start_time` 晚于零点而被误判为新开通。
- 畅享套餐本次只调整“套餐每日额度口径和值”，默认沿用当前 `daily_quota` 的余额兜底行为；如果业务希望超额后完全阻断，需要单独再收敛规则。

## 现状分析

- 当前无限套餐在 `backend/app/services/subscription_service.py` 中统一按 `PLAN_KIND_UNLIMITED` 处理，默认固定为 `total_tokens / 300000000`。
- 当前畅享套餐属于 `PLAN_KIND_DAILY_QUOTA`，其额度来自订阅快照中的 `quota_metric/quota_value`，请求前后由 `SubscriptionService.check_quota_before_request()` 与 `consume_quota_after_request()` 做周期校验。
- 当前文本请求成功后的 `total_cost` 已经具备：
  - 输入按输入价计费
  - 输出按输出价计费
  - 缓存读取按输入价 `10%` 计费
  但该成本会叠加系统 `price_multiplier`，不满足“套餐限额按官方价格 1:1 计算”的新要求。
- 当前前端和接口层对无限套餐仍使用 `unlimited_daily_token_limit` 这类 token 专用字段，无法表达“无限套餐按美元、畅享套餐按美元”的新口径。
- 现有 `subscription_usage_cycle`、`request_log`、`consumption_record` 已支持 `quota_metric=cost_usd`，数据库结构足以承载本次改造。

## 当前线上基线

- 当前线上畅享套餐：
  - `daily-10m-token`
  - `weekly-10m-token`
  - `monthly-10m-token`
  实际业务口径已调整为“每日 `5000w token` 限额”。
- 当前线上无限套餐：
  - `daily-unlimited`
  - `weekly-unlimited`
  - `monthly-unlimited`
  刚切换前的实际业务口径是“每日 `3 亿 token` 限额”，现在需要统一切到每日美元限额。

## 代码与线上配置差异

- 当前代码内置畅享模板仍写死为 `quota_value=10000000`。
- 当前代码中的无限套餐日限额仍由服务端常量兜底为 `300000000`，已经落后于新的业务规则。

因此实施时不能把“畅享套餐”识别死绑到代码内置 `quota_value=10000000`，也不能只依赖当前代码模板值。应优先按套餐家族快照识别，例如：

- `plan_code_snapshot`
- `plan_name`
- `plan_kind=daily_quota`
- `duration_mode/plan_type/duration_days_snapshot`

避免线上数据库已是 `5000w` 时，代码仍按本地内置 `1000w` 模板误判。

## 技术方案设计

### 1. 统一抽象“订阅每日额度策略”

在 `SubscriptionService` 中新增统一策略解析方法，根据订阅快照返回：

- 该订阅每日额度是否按周期限制
- 额度口径：`total_tokens` 或 `cost_usd`
- 当日额度值
- 是否允许超额后走余额兜底

目标策略矩阵：

- 无限套餐（日 / 周 / 月）：
  - 存量：`cost_usd / 120`
  - 新开：`cost_usd / 100`
  - 超额：直接阻断，不走余额兜底
- 畅享日卡 / 周卡 / 月卡：
  - 存量：`cost_usd / 60`
  - 新开：`cost_usd / 50`
  - 超额：默认沿用现有 `daily_quota` 余额兜底
- 其他 `daily_quota` 套餐：
  - 保持现有 `quota_metric/quota_value` 逻辑

### 2. 为套餐限额新增“官方价格口径”的独立成本计算

无限月卡和畅享套餐的新日额度都不能直接复用当前 `total_cost`，因为当前 `total_cost` 受以下因素影响：

- `price_multiplier`
- `token_multiplier`
- 展示与实际账单口径

本次需要新增一套仅用于套餐日额度的成本口径：

- 普通输入成本：`billable_input_tokens / 1_000_000 * input_price_per_million`
- 缓存读取成本：`cache_read_input_tokens / 1_000_000 * input_price_per_million * 0.1`
- 输出成本：`output_tokens / 1_000_000 * output_price_per_million`
- 套餐日额度消耗：以上三者之和

这套额度成本需要满足：

- 不使用 `price_multiplier`
- 不使用 `token_multiplier`
- 严格按模型官方单价 `1:1`
- 缓存读取严格按输入价 `10%`

### 3. 调整请求前预检

当前 `ProxyService._build_text_quota_precheck()` 会生成：

- `estimated_total_tokens`
- `estimated_total_cost`

其中 `estimated_total_cost` 使用的是倍率后的价格，不适合本次套餐额度预检。

需要补充“套餐额度预检成本”：

- 输入/输出 token 继续沿用现有估算逻辑
- 价格按模型官方价格 `1:1`
- 预检阶段无法准确预测缓存命中，默认不对缓存读取做减免
- 对无限月卡和畅享套餐，预检使用新的官方价格成本

这样可以在请求前阻断明显超额的请求；请求后仍以真实 usage 为最终裁决。

### 4. 调整请求后额度累计

当前 `SubscriptionService.consume_quota_after_request()` 在 `quota_metric=cost_usd` 时直接使用 `total_cost`。本次需要允许外部显式传入“套餐额度消耗成本”：

- 对无限套餐（日 / 周 / 月）：写入 `官方价格 1:1 + cache 10%` 计算出的美元成本
- 对畅享套餐：也写入这套美元成本
- 对其他原生 `cost_usd` 套餐：保留兼容能力

最终以下字段都写入真实套餐额度消耗值：

- `subscription_usage_cycle.used_amount`
- `request_log.quota_consumed_amount`
- `consumption_record.quota_consumed_amount`

### 5. 存量用户兼容判定

为避免误伤已开通用户，需要在订阅快照层做兼容判定。

无限套餐识别建议：

- `plan_kind_snapshot == unlimited`
- 且 `plan_code_snapshot` 落在 `daily-unlimited` / `weekly-unlimited` / `monthly-unlimited`
- 或 `plan_type/duration_days_snapshot` 识别为日 / 周 / 月无限套餐
- 且 `created_at < 2026-05-10 00:00:00 Asia/Shanghai`

畅享套餐识别建议：

- `plan_kind_snapshot == daily_quota`
- 且 `plan_code_snapshot` / `plan_name` 落在畅享套餐家族
- 且 `plan_type/duration_days_snapshot` 识别为日/周/月畅享
- 且 `created_at < 2026-05-10 00:00:00 Asia/Shanghai`

### 6. 后端接口与前端展示改为通用额度字段

套餐展示不能再只靠 token 专用字段，应改为返回解析后的通用额度信息，例如：

- `resolved_daily_quota_metric`
- `resolved_daily_quota_value`
- `resolved_daily_quota_fallback_mode`

或直接复用现有订阅摘要中的：

- `quota_metric`
- `quota_value`
- `current_cycle`

但这些字段必须是“解析后的有效口径和值”，而不是原始快照值。

前端展示目标：

- 无限套餐存量：`$120.00/天`
- 无限套餐新开：`$100.00/天`
- 畅享套餐存量：`$60.00/天`
- 畅享套餐新开：`$50.00/天`
- 当前周期已用 / 剩余按各自口径展示

需要同步调整：

- 管理端套餐模板页
- 管理端套餐记录页
- 管理端旧版无限套餐开通区域文案
- 代理端套餐库存页
- 代理端发放记录页

## 数据库影响

- 本次预计不需要新增表或新增字段。
- 继续复用 `subscription_usage_cycle.quota_metric/quota_limit/used_amount`。
- 继续复用 `request_log`、`consumption_record` 中已有的：
  - `quota_metric`
  - `quota_consumed_amount`
  - `quota_limit_snapshot`
  - `quota_used_after`

## 风险点

- 如果直接复用当前 `total_cost`，会把 `price_multiplier` 算进套餐额度，导致套餐用户额度被提前耗尽。
- 如果把畅享套餐识别死绑到 `quota_value=50000000`，会与当前代码内置 `10000000` 模板冲突，也可能漏掉线上已经改过模板的存量用户。
- 如果仅凭 `start_time` 判定“已开通用户”，顺延套餐会被误判成新开通。
- 畅享套餐当前存在余额兜底；如果产品希望也改成硬阻断，需要在实施前再确认，否则行为会和无限套餐不同。
- 前端如果继续只读 `unlimited_daily_token_limit`，会出现美元套餐仍显示 token 或单位错乱的问题。

## 实施 Tasks

1. 在 `SubscriptionService` 中抽象订阅每日额度策略解析，覆盖无限套餐日/周/月、畅享日/周/月和其他 `daily_quota` 套餐。
2. 调整 `check_quota_before_request()`、`consume_quota_after_request()`、套餐摘要序列化逻辑，让无限套餐日/周/月与畅享套餐统一走 `cost_usd`。
3. 在 `ProxyService` 中新增“套餐额度专用成本”计算与预检逻辑，按官方价格 `1:1` 和缓存 `10%` 计算，不使用系统倍率。
4. 修改请求成功后的套餐额度累计参数传递，确保无限套餐日/周/月与畅享套餐的 `quota_consumed_amount` 写入真实美元成本。
5. 调整 admin/agent 相关接口与页面展示字段，支持无限套餐与畅享套餐按 `USD/天` 或 `Token/天` 混合显示。
6. 补充测试，覆盖：
   - 无限套餐存量 `120 USD`
   - 无限套餐新开 `100 USD`
   - 畅享套餐存量 `60 USD`
   - 畅享套餐新开 `50 USD`
   - 套餐额度不受 `price_multiplier` 影响
   - 缓存读取按 `10%` 计入额度
7. 运行后端单测与前端构建验证，确认旧版无限开通、模板发放、代理发放、请求前阻断、请求后累计与展示逻辑均正常。
