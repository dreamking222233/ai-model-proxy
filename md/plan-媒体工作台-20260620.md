# 媒体工作台实施方案 v2

## 用户原始需求

用户端新增一个独立页面「媒体工作台」，用于生图、生成视频，并支持预览、下载。现有 `user/chat` 页面虽然支持生图和视频，但用户反馈不好用，且经常因为超时或渠道异常调用失败。

页面布局参考 `/Volumes/project/modelInvocationSystem/image-api-page/index.html`，在用户端新增菜单，点击后跳转独立工作台。

生图渠道分两类：

- `OpenAI Image Compatible`：只能生成 1K 图片。
- `OpenAI Image Native Size`：可根据参数生成 1K、2K、4K 图片。

需要灵活路由：

- 用户选择 1K 时，优先选择 compatible 类型渠道。
- 用户选择 2K、4K 时，走 Native Size 类型渠道。

视频生成只有一种渠道：`Grok-video`。

页面最好添加渠道类型检测，例如展示最近调用生图 `gpt-image-2` 的成功率健康度、Grok 视频生成成功率健康度等。

## 技术方案设计

### 现状梳理

- 前端为 Vue 2 + Ant Design Vue，用户端路由位于 `frontend/src/router/index.js`，用户菜单位于 `frontend/src/layout/UserLayout.vue`。
- 现有 `frontend/src/views/common/AiChat.vue` 已通过用户 API Key 调用：
  - 图片生成：`POST /v1/images/generations`
  - 图片编辑：`POST /v1/image/edit`
  - 视频生成：`POST /v1/created/video`
- 用户端模型列表接口 `GET /api/user/chat/models` 已返回图片/视频模型、图片尺寸能力、图片编辑能力、视频尺寸能力等字段。
- 后端图片路由已按渠道能力过滤尺寸：
  - `openai-image-compatible` 支持 `1K`
  - `openai-image-native-size` 支持 `1K/2K/4K`
- 现有过滤能保证 2K/4K 不会命中 compatible，但 1K 在 compatible 与 native 都可用时，仍主要依赖渠道优先级。需补充 1K compatible 优先排序。
- 请求日志 `request_log` 已记录 `requested_model`、`request_type`、`status`、`response_time_ms`、`image_size` 等，可用于聚合最近成功率。

### 前端页面方案

新增 `frontend/src/views/user/MediaWorkbench.vue`：

- 页面结构参考静态页，但使用项目现有 Ant Design Vue 组件和用户端布局。
- 左侧为固定控制面板：
  - 模式切换：生图、图生视频。
  - 模型选择：图片模型默认优先 `gpt-image-2`，视频模型默认优先 `grok-imagine-video`。
  - 提示词输入，字符计数。
  - 生图参数：分辨率 `1K/2K/4K`、比例、数量。
  - 视频参数：时长、画面尺寸、清晰度、预设。
  - 图生视频参考图上传与预览。
- 右侧为结果画布：
  - 请求中状态、错误状态、空状态。
  - 图片网格预览、放大预览、下载。
  - 视频播放器预览、下载。
  - 原始响应折叠展示，方便排查。
- 顶部状态条：
  - 媒体积分余额。
  - 生图健康度卡片。
  - 视频健康度卡片。
  - 当前路由策略提示：`1K 优先 Compatible`、`2K/4K 使用 Native Size`。

### 前端 API 方案

新增 `frontend/src/api/mediaWorkbench.js`：

- 仅新增 `getMediaHealth()`，调用用户端健康摘要接口。
- `getChatModels()` 继续从 `frontend/src/api/chat.js` 导入。
- `getBalance()`、`getSiteConfig()` 继续从 `frontend/src/api/user.js` 导入。
- 新增 `frontend/src/utils/userApiKey.js`，抽出用户端 API Key 准备逻辑：
  - 先读取与聊天页一致的 sessionStorage key。
  - 再 `listApiKeys()` 查找 active key。
  - active key 存在则 `revealApiKey()`。
  - 没有 active key 时才 `createApiKey()`。
  - 供 `AiChat.vue` 和 `MediaWorkbench.vue` 复用，避免两处重复创建密钥。
- 代理请求仍用 `fetch`，原因是图片/视频接口不是统一 `ResponseModel`，且需要长超时和二进制视频下载。

### 后端健康摘要接口方案

新增 `backend/app/api/user/media_workbench.py`，注册到 `backend/app/main.py`：

- `GET /api/user/media-workbench/health`
- 返回最近 24 小时内媒体请求聚合：
  - `image_gpt_image_2`
  - `video_grok`
- 固定返回上述两个 key，即使无数据也返回 `health_level=unknown`、`request_count=0`。
- 统计口径：
  - `image_gpt_image_2`：`requested_model in ("gpt-image-2", "codex-gpt-image-2", "plus-codex-gpt-image-2")` 且 `request_type in ("image_generation", "image_edit")`。
  - `video_grok`：`requested_model = "grok-imagine-video"` 且 `request_type = "video_generation"`。
  - 时间范围：默认最近 24 小时。
  - 成功条件：复用 `LogService._visible_success_condition()`，保持与现有日志统计一致。
  - 排除明显本地前置失败：`status in ("error", "failed")` 且 `channel_id is null` 且错误码表现为余额/参数/模型不存在等本地错误时不计入健康样本；无法准确区分的失败仍作为“最近调用成功率”统计。
- 成功率分级：
  - `unknown`：样本数为 0。
  - `good`：成功率 >= 95%。
  - `warning`：成功率 >= 80% 且 < 95%。
  - `bad`：成功率 < 80%。
- 聚合字段：
  - `label`
  - `model`
  - `request_types`
  - `request_count`
  - `success_count`
  - `failed_count`
  - `success_rate`
  - `avg_response_time_ms`
  - `last_request_at`
  - `health_level`：`good/warning/bad/unknown`
- 基于 `RequestLog` 聚合，不新增数据库表。
- 用户端只展示总体健康，不泄露渠道 ID、渠道名称、上游错误细节。
- UI 文案使用「最近调用成功率」，不宣称是严格渠道可用率。

### 后端路由排序方案

在 `backend/app/services/proxy_service.py` 新增图片渠道偏好排序：

- 对图片生成和图片编辑，在 `_filter_channels_by_image_size()` 后执行排序。
- 当 `image_size == "1K"` 时：
  - `openai-image-compatible` 排在 `openai-image-native-size` 前。
  - 只调整 OpenAI 图片 compatible/native 两类的相对顺序。
  - 非 OpenAI 渠道、非图片专用渠道、同类渠道内部保持原有相对顺序。
- 当 `image_size in ("2K", "4K")` 时：
  - 过滤逻辑已只保留支持对应尺寸的渠道，native 自然命中。
- 不改变管理端渠道优先级的主语义，只在同一候选序列上增加图片 1K 专用偏好。

实现函数：

- `_prefer_openai_compatible_for_1k_image(channels, image_size)`
- 在 `handle_image_request()` 与 `handle_image_edit_request()` 中调用。

### 视频长耗时策略

首版仍使用现有同步等待接口 `POST /v1/created/video`，理由：

- 后端已实现等待完成、计费、日志和 `content_url` 返回。
- 可以快速形成完整闭环。

前端策略：

- 视频请求超时设置为 20 分钟。
- 超时提示明确说明可能是长任务或代理超时，可稍后重试。
- 请求中锁定提交按钮，避免重复提交。
- 视频预览和下载均通过带 Authorization 的 `fetch` 读取 blob，再创建 object URL。
- 页面销毁、清空结果或替换视频结果时释放 object URL。

后续优化项：

- 将视频生成改为 `POST /v1/videos` 创建异步任务，再轮询 `GET /v1/videos/{video_id}`，完成后请求 `/v1/videos/{video_id}/content`。

## 影响范围分析

### 涉及模块

- 用户端菜单与路由。
- 新增用户端媒体工作台页面。
- 用户端媒体健康摘要 API。
- 图片渠道候选排序。
- 文档与实施记录。

### 风险点

- 图片/视频请求超时受浏览器、Nginx、后端和上游共同影响，页面只能改善等待提示和错误展示，不能单独消除所有超时。
- 1K compatible 优先可能改变现有部分 1K 请求命中的渠道顺序，因此排序逻辑应限定在图片媒体请求候选里。
- 用户端自动创建/读取 API Key 需要抽公共 helper，避免聊天页和工作台重复创建密钥。
- 视频内容下载需携带 API Key，预览时使用 object URL，避免 `<video>` 直接带不了鉴权头。
- 健康度展示为最近调用成功率，可能受用户错误请求影响，需在 UI 上避免表述为绝对渠道健康。
