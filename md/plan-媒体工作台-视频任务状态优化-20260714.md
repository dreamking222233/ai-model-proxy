# 媒体工作台-视频任务状态优化 Plan

## 用户原始需求

- 分析请求 `9856057a-b91b-4eb7-911a-4692f073ca85` 在 `/user/media-workbench` 长时间显示“正在生成视频”的原因。
- 修复问题并更新本地、GitHub 与生产服务器。

## 问题证据与根因

- 生产日志显示请求先命中旧渠道 `zz.211b`，`POST /v1/video/generations` 连续返回 404，约 7 秒后才自动切换到新 119337 渠道。
- 新渠道成功创建任务 `task_ryOYEEpqvdIZNZDbjmBrBR5h6EXUoida`，约 90 秒后完成并成功获取视频内容。
- 请求日志最终为 `success`，扣除 7.500 媒体积分，不存在任务永久卡死或重复扣费。
- 页面生成期间只显示固定文案，没有任务 ID、排队/生成状态、上游进度和已等待时间，正常的异步等待容易被误认为卡死。
- 旧渠道仍保留 Grok 视频模型映射，因此即使新渠道正常，每次视频请求仍可能先经过已失效渠道。

## 技术方案

1. 新增幂等 SQL，仅停用 `zz.211b` 旧渠道的两个 Grok 视频映射，保留该渠道已有文本模型映射。
2. 视频创建开始后显示“正在提交任务”，获得任务 ID 后切换为排队/生成状态。
3. 每次轮询同步显示上游 `status`、`progress`、任务 ID与已等待时间。
4. 对缺少进度的上游响应保持不确定进度展示，不伪造百分比。
5. 完成、失败或组件销毁时清理计时器，避免页面资源泄漏。
