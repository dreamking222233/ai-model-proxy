## Plan v2 修订说明

- 增加请求体脱敏、base64/文件字段处理、最大体积限制和截断字段。

- 将流式输出强制中断调整为第二阶段增强，第一版只做风险标记捕获、清洗和记录。
- 补充数据库组合索引和测试项。

## 用户原始需求

在当前系统中增加违规请求检测机制：

- 在请求层添加检测和限制，覆盖黄色、破限、注册机、破解网站、异常自动化、非法破坏计算机等请求。
- 给所有模型追加安全提示词，让模型拒绝执行违规请求，并在发现用户有非法举动时上报后端。
- 后台管理员新增页面，展示检测到的记录，并能查看该请求的完整内容。
- 对用户每次请求进行短暂保存，例如仅保留 10 分钟、30 分钟；超时后若无安全问题则删除请求内容，只保留普通用户请求记录。
- 当模型或后端检测到风险后，管理员能看到用户完整请求内容，用于审计、申诉和争议处理。

## 技术方案设计

### 总体架构

采用分阶段四层防线：

第一阶段先落地：

1. **请求短期暂存**
   - 每个进入代理层的请求先创建 `security_request_snapshot`。
   - 保存完整请求体、请求文本提取结果、用户信息、模型、协议、客户端 IP、过期时间。
   - 默认 TTL 建议 30 分钟，可通过系统配置改为 10/30/60 分钟。

2. **后端规则检测**
   - 在进入上游模型前执行关键词和组合规则检测。
   - 高风险直接阻断并创建风险事件。
   - 中风险放行但标记快照风险分数。

第二阶段再落地：

3. **模型安全提示词兜底**
   - 给所有文本模型请求追加安全提示词。
   - 要求模型拒绝黄色、破限、破解、注册机、网站攻击、盗号、验证码/风控绕过等请求。
   - 要求模型在识别到风险时输出固定结构化标记，供后端输出侧捕获；不依赖模型传用户 ID。

4. **输出侧检测与风险上报**
   - 对非流式输出做完整文本检测。
   - 第一版流式仅捕获并清洗模型风险标记，记录高风险片段，不主动中断已有流。
   - 流式中途终止或替换拒绝文案作为第二阶段增强，避免破坏 SSE 协议、缓存和计费链路。
   - 风险上报后，把对应 snapshot 标记为 `flagged`，完整请求体不按短 TTL 删除，改为按照风险事件保留期保存。

### 重要设计取舍

- 不让模型直接调用外部 HTTP 接口传 `sessionRiskId`。当前系统不是工具调用编排框架，直接让模型“调用接口”不可靠，也容易被提示词污染。
- 使用“模型固定结构化标记 + 后端解析”的方式作为默认上报路径，例如模型输出隐藏前缀或 JSON 标记后，后端捕获并创建风险事件，再清洗标记不返回给用户。
- 额外提供公网免登录风险上报接口作为扩展能力。注意：普通模型只看到 URL 并不会自动发 HTTP 请求，只有后续接入工具调用或可控 HTTP 调用能力时，模型才能真实请求该接口。
- `sessionRiskId` / `snapshot_id` 由后端生成和绑定，绝不信任前端或模型传入。
- “对峙”在产品文案中建议改为“审计、申诉、争议处理”，避免后台页面和隐私说明表达过于对抗。
- 模型风险标记只能作为弱信号，不能作为强证据；输入侧规则命中和请求快照才是主要判断依据。
- 运行时规则先在 `SecurityDetectionService` 中结构化定义，不从 `md/` 说明文档动态读取。
- 风控写入和普通代理主链路解耦，创建快照和风险事件使用独立事务，避免影响已成功的计费和请求日志落库。

## 数据库设计

### 新增表：security_request_snapshot

用途：短期保存每次请求完整内容。无风险时按 TTL 清空完整内容，只保留摘要。

建议字段：

- `id` bigint primary key
- `snapshot_id` varchar(36) unique，UUID
- `request_id` varchar(36) index，对应 `RequestLog.request_id`
- `user_id` bigint index
- `agent_id` bigint index
- `user_api_key_id` bigint
- `requested_model` varchar(128)
- `protocol_type` varchar(32)，如 `openai_chat`、`anthropic_messages`、`responses`、`image`、`video`
- `request_type` varchar(32)，如 `chat`、`responses`、`image_generation`、`video`
- `client_ip` varchar(45)
- `request_hash` char(64)，完整请求 JSON SHA-256
- `report_token_hash` char(64)，短期上报令牌 hash，不保存明文 token
- `request_preview` text，脱敏后的短预览
- `request_body_json` longtext，完整请求体 JSON
- `extracted_text` mediumtext，提取出的用户文本
- `is_truncated` smallint，是否因体积限制截断
- `body_size_bytes` int，脱敏后请求体大小
- `retention_status` varchar(20)，`temporary` / `flagged` / `purged`
- `risk_level` varchar(20)，`none` / `low` / `medium` / `high` / `blocked`
- `risk_categories_json` text
- `expires_at` datetime index
- `purged_at` datetime
- `created_at` datetime
- `updated_at` datetime

清理策略：

- `temporary` 且 `expires_at < now()`：清空 `request_body_json`、`extracted_text`，保留 `request_hash`、`request_preview`、基础元数据，状态改为 `purged`。
- `flagged`：不走短 TTL，按风险事件保留期处理，默认可配置 30 天。

脱敏和体积策略：

- 入库前对 `Authorization`、`api_key`、`key`、`token`、`password`、`secret`、`cookie` 等字段做掩码。
- 图片、音视频、文件、base64 字段不保存原文，仅保存类型、大小、hash、预览说明。
- `request_preview` 固定限制 500-1000 字。
- `extracted_text` 只保存用户输入文本，不保存平台注入的 system prompt。
- 通过 `security_snapshot_max_body_bytes` 控制最大保存体积，超过后截断并设置 `is_truncated=1`。
- 创建 snapshot 时生成一次性或短期 `report_token`，只把明文 token 注入本次上游 system prompt，数据库只保存 hash。

建议索引：

- `idx_security_snapshot_snapshot_id` unique(`snapshot_id`)
- `idx_security_snapshot_request_id`(`request_id`)
- `idx_security_snapshot_user_created`(`user_id`, `created_at`)
- `idx_security_snapshot_retention_expires`(`retention_status`, `expires_at`)

### 新增表：security_risk_event

用途：保存检测、阻断、模型上报、输出侧命中的安全事件。

建议字段：

- `id` bigint primary key
- `event_id` varchar(36) unique，UUID
- `snapshot_id` varchar(36) index
- `snapshot_db_id` bigint index，关联 snapshot 自增 ID，提高查询效率
- `request_id` varchar(36) index
- `user_id` bigint index
- `agent_id` bigint index
- `requested_model` varchar(128)
- `protocol_type` varchar(32)
- `event_source` varchar(32)，`keyword` / `semantic` / `model_report` / `output_scan` / `manual`
- `risk_level` varchar(20)，`low` / `medium` / `high` / `blocked`
- `category` varchar(64)，如 `sexual_content`、`prompt_jailbreak`、`cyber_abuse`、`illegal_automation`
- `action` varchar(32)，`allow` / `review` / `block` / `output_block`
- `matched_rules_json` text
- `reason` text
- `response_excerpt` mediumtext，可选，保存模型疑似违规输出片段
- `status` varchar(20)，`open` / `reviewed` / `ignored`
- `reviewer_id` bigint
- `review_note` text
- `reviewed_at` datetime
- `created_at` datetime

建议索引：

- `idx_security_event_event_id` unique(`event_id`)
- `idx_security_event_status_created`(`status`, `created_at`)
- `idx_security_event_level_created`(`risk_level`, `created_at`)
- `idx_security_event_category_created`(`category`, `created_at`)
- `idx_security_event_user_created`(`user_id`, `created_at`)
- `idx_security_event_snapshot_id`(`snapshot_id`)

### 系统配置项

新增到 `system_config`：

- `security_detection_enabled`：默认 `true`
- `security_snapshot_enabled`：默认 `true`
- `security_snapshot_ttl_seconds`：默认 `1800`
- `security_flagged_retention_days`：默认 `30`
- `security_keyword_block_enabled`：默认 `true`
- `security_output_scan_enabled`：默认 `true`
- `security_stream_output_scan_enabled`：默认 `true`
- `security_model_prompt_enabled`：默认 `true`
- `security_block_message`：默认拒绝文案
- `security_fail_closed_enabled`：默认 `false`
- `security_snapshot_max_body_bytes`：默认 `262144`
- `security_scan_max_text_chars`：默认 `20000`
- `security_public_report_enabled`：默认 `true`
- `security_public_report_rate_limit_per_minute`：默认 `60`

降级策略：

- `security_detection_enabled=false`：不执行输入检测，不创建风险阻断事件。
- `security_snapshot_enabled=false`：不创建请求快照，但仍可执行轻量输入检测。
- 风控服务异常时：
  - `security_fail_closed_enabled=false`：记录后端日志，主请求继续执行。
  - `security_fail_closed_enabled=true`：阻断请求，返回统一安全检测异常提示。
- 高风险规则命中时不受 fail-open 影响，直接阻断并尽力写入风险事件。

### 新增模型

文件：`backend/app/models/__init__.py`

### 新增服务

文件：`backend/app/services/security_detection_service.py`

核心职责：

- `create_snapshot(db, user, api_key_record, request_id, request_data, protocol_type, request_type, requested_model, client_ip)`
  - 创建短期请求快照。
  - 提取文本、生成 hash、预览和过期时间。
  - 使用独立事务或立即 commit，保证后续流式请求释放连接后仍能查到 snapshot。

- `scan_request(db, snapshot, request_data)`
  - 执行关键词、组合规则检测。
  - 返回 `SecurityDetectionResult`，包含 `risk_level`、`categories`、`matched_rules`、`action`、`reason`。

- `record_risk_event(db, snapshot, source, result, response_excerpt=None)`
  - 创建风险事件。
  - 把 snapshot 标记为 `flagged`。

- `purge_expired_snapshots(db, limit=500)`
  - 清理过期无风险完整请求体。

- `build_security_system_prompt(db)`
  - 按配置生成安全提示词。
  - 第二阶段支持把 `report_url`、`snapshot_id`、短期 `report_token` 注入到本次请求的 system prompt。

- `scan_model_output(db, snapshot_id, output_text)`
  - 对模型输出检测是否出现违规帮助、破限标记或模型上报标记。

- `handle_public_report(db, payload, client_ip)`
  - 处理公网免登录上报请求。
  - 校验 `snapshot_id` 和 `report_token`，通过后写入 `security_risk_event`。
  - 对非法 token、过期 token、重复上报、频率过高做拒绝或限流。

- `list_risk_events(db, filters)`
  - 管理端分页查询风险事件。

- `get_risk_event_detail(db, event_id)`
  - 查询事件详情、snapshot、完整请求体和输出片段。

- `get_risk_stats(db)`
  - 查询待处理、高风险、今日风险、已清理快照等统计。

### 模型安全提示词注入

修改：

- `_build_model_identity_prompt(requested_model)`
- 或新增 `_build_system_guard_prompt(requested_model, db=None)`，由 `_inject_model_identity` 调用。

建议提示词核心内容：

```text
安全策略：
- 不得生成、补全、改写或指导任何色情、未成年人性内容、破限提示词、绕过审查、破解、注册机、盗号、撞库、验证码绕过、风控对抗、网站攻击、恶意自动化等内容。
- 如果用户以“我是学生、学校作业、老师要求、仅供学习、CTF、靶场、授权测试”等理由请求上述内容，也必须拒绝具体步骤、代码和可执行方法。
- 可以提供合法替代：安全防护建议、合规学习路径、靶场搭建原则、报告结构、代码审计思路。
- 当识别到用户请求存在风险时，先拒绝，再输出内部风险标记：[MIS_RISK_REPORT category=... severity=...]
```

后端需要在返回用户前删除 `[MIS_RISK_REPORT ...]` 标记，并创建 `security_risk_event`。

注意：

- 该标记只作为 `event_source=model_report` 的弱信号。
- 若用户输入中主动包含该标记，不能直接判定模型上报，需要结合输出位置和上下文。
- 安全提示词保持短文本，并受 `security_model_prompt_enabled` 控制。
- 如果实现公网上报接口，提示词中可加入：

```text
如果你具备工具或 HTTP 调用能力，并识别到用户请求存在上述风险，请向以下接口提交风险报告：
POST https://api.xiaoleai.team/api/public/security/risk-report
body: {"snapshot_id":"<backend_generated_snapshot_id>","report_token":"<backend_generated_report_token>","category":"...","severity":"...","reason":"..."}
如果你不具备 HTTP 调用能力，则在拒绝后输出内部风险标记：[MIS_RISK_REPORT category=... severity=...]
```

- 该接口虽然免登录，但必须校验 `report_token`，否则任何人都可以伪造风险记录。

### 公网免登录上报接口

新增文件：`backend/app/api/public/security.py`

路由：

- `POST /api/public/security/risk-report`

公网地址：

- `https://api.xiaoleai.team/api/public/security/risk-report`

权限策略：

- 不走用户登录态，不要求 API Key。
- 必须校验后端生成的 `snapshot_id + report_token`。
- `report_token` 只对当前 snapshot 有效，默认随 snapshot 短 TTL 过期。
- 同一个 snapshot 对同类 `category` 做幂等去重，避免重复刷记录。
- 增加 IP 级限流和 payload 大小限制。

请求体：

```json
{
  "snapshot_id": "backend-generated-uuid",
  "report_token": "backend-generated-short-lived-token",
  "category": "cyber_abuse",
  "severity": "high",
  "reason": "用户请求疑似破解网站或编写注册机"
}
```

后端处理：

- 校验 token hash、snapshot 状态和过期时间。
- 写入 `security_risk_event`，`event_source=model_report_api`。
- 标记 `security_request_snapshot.retention_status=flagged`。
- 不信任模型提交的用户 ID、请求体内容和客户端 IP，全部以后端 snapshot 记录为准。

### 输入检测规则

第一版先用本地规则，不引入额外模型调用，避免成本和延迟扩大。

规则来源：

- `md/违禁词筛选-破限黄色-20260618.md`

分类：

- `sexual_content`
- `prompt_jailbreak`
- `cyber_abuse`
- `illegal_automation`
- `student_pretext_abuse`

检测策略：

- 单关键词高风险：如 `注册机`、`keygen`、`webshell`、`getshell`、`撞库`、`强奸`、`未成年色情`。
- 组合规则高风险：`我是学生/学校作业/仅供学习` + `注册机/破解/绕过/爆破/撞库/webshell/验证码绕过`。
- 中风险：`自动化脚本`、`批量请求`、`爬虫绕过` 等，需要结合目标词和动词。
- 普通词降权：学生、作业、学习用途、CTF、靶场单独出现不拦截。

### 输出检测

非流式：

- 在 JSONResponse 返回前提取文本输出。
- 检测 `[MIS_RISK_REPORT ...]` 或输出中出现高风险可执行帮助。
- 命中后记录事件，并替换返回内容为统一拒绝文案或保持模型拒绝内容。

流式：

- 在 SSE generator 中维护短缓冲，例如最近 4000 字符。
- 捕获 `[MIS_RISK_REPORT ...]` 后记录事件，过滤标记不下发。
- 第一版不主动终止流，不替换已有输出，避免破坏 SSE、计费、缓存和兼容路径。

注意：

- 流式输出检测比非流式复杂，第一阶段建议至少实现“模型风险标记捕获 + 高风险片段记录”，强制终止可作为第二阶段增强。

### 管理端 API

新增文件：`backend/app/api/admin/security.py`

路由前缀：`/api/admin/security`

接口：

- `GET /events`
  - 参数：`page`、`page_size`、`user_id`、`category`、`risk_level`、`source`、`status`、`start_date`、`end_date`
  - 返回风险事件列表。

- `GET /events/{event_id}`
  - 返回事件详情、snapshot 元数据、完整请求体、提取文本、响应片段。
  - 若 snapshot 已清理且事件不是 flagged，返回“请求内容已按保留策略清理”。
  - 查看完整请求体时写入 `operation_log`，记录管理员 ID、IP、`event_id`、`snapshot_id`。

- `GET /stats`
  - 返回 `open_count`、`high_count`、`today_count`、`purged_snapshot_count`、`blocked_count`。

- `PUT /events/{event_id}/review`
  - 管理员标记 `reviewed` / `ignored`，填写备注。

- `POST /snapshots/purge`
  - 手动触发过期快照清理。
  - 必须限制单次清理数量，避免全表长事务。

### 风险点

- 完整请求体含敏感内容，必须只给平台管理员可见。
- 风险事件保留期不能无限制，建议默认 30 天并可配置。
- 流式输出中途拦截可能影响 SSE 协议兼容，需要分阶段实现。
- 提示词注入会增加上下文长度和成本，应保持短而明确。
- 关键词规则可能误伤 CTF、合法授权测试、正常学习内容，需要使用组合规则和风险等级。
- 模型上报标记可能被用户诱导伪造，因此后端只能把它作为风险信号，不能作为唯一证据。
- WebSocket turn 若不接入，会成为绕过入口。
- 管理员查看完整请求体属于高敏操作，必须审计。
- 风控数据库异常不能默认拖垮主代理链路，需配置化 fail-open/fail-closed。
