# 推广链接实施方案

## Plan v2 修订说明

根据 `./md/plan-review-推广链接-20260610.md`，本方案修订为 v2 后进入实施。v2 固定以下关键边界：

- 推广码非法、禁用、跨站或跨代理时，注册整体失败并回滚；不做静默忽略。
- 注册事务调整为：创建用户与初始资产记录后 `flush()`，绑定推广关系成功后统一 `commit()`。
- “已充值人数”统计口径为：同一被推广用户首次出现任一成功线上充值时计数一次，不因余额充值和图片积分充值重复计数。
- 返现口径为：按被推广用户实际到账资产的 50% 返现，即 `credited_usd * 0.5` 或 `credited_image_credits * 0.5`，不是按人民币实付金额返现。
- 推广返现不计入推广人的 `UserBalance.total_recharged` / `UserImageBalance.total_recharged`，避免“本人充值总额”和“返现所得”混淆；仅增加可用余额并写入明细流水。
- `user_promotion_reward` 明细表为最终对账依据；`user_promotion_link`、`user_promotion_relation` 的累计字段仅作查询冗余，必须和 reward 明细同事务更新。
- 资产来源页合并余额与图片积分时使用 `union_all` 或等价 SQL 统一分页，不采用“两表各查一页再内存合并”。
- 推广返现来源识别以 `user_promotion_reward` 结构化记录或图片积分 `action_type='promotion_reward'` 为准，`model_name/remark` 只做展示。
- 三端查询逻辑收敛在 `PromotionService`，用户端、管理端、代理端只做权限范围与参数差异。
- 第一阶段每个用户一个推广码，不支持多渠道推广码归因；旧数据不补历史推广关系。

## 用户原始需求

当前需要在用户端新增“生成推广链接”页面。系统分直营站和代理站点：

- 直营站如 `www.xiaoleai.team`，直营用户生成的推广链接以 `https://www.xiaoleai.team` 开头。
- 代理站点如 `*.xiaoleai.team`，代理站点用户生成的推广链接以对应代理站点域名开头，并携带推广链接编码。
- 其他用户通过推广链接注册后，绑定推广关系，推广用户成为新用户的推广人。
- 被推广用户通过 `user/recharge` 线上充值后，按充值到账资产给推广者返现：
  - 余额充值示例：充值 1 人民币到账 5 美元额度，则推广者返现 2.5 美元额度。
  - 图片积分充值也需要按同样规则返现图片积分。
- 推广者可在 `user/asset-source` 页面看到返现余额来源。
- 用户端推广页面可看到通过链接注册的用户、是否充值等信息。
- 管理端和代理端新增对应页面，可查看用户推广信息记录。

## 当前系统分析

### 后端结构

- 技术栈：FastAPI + SQLAlchemy ORM。
- 用户模型：`backend/app/models/user.py` 的 `SysUser`，已有 `agent_id`、`source_domain`，可区分直营用户和代理用户。
- 站点解析：`backend/app/services/agent_service.py` 的 `AgentService.get_site_context_from_request()`，会基于 `Host`、`X-Site-Host`、`Origin`、`Referer` 解析直营/代理站点。
- 注册入口：`backend/app/api/auth.py` 调用 `AuthService.register()`，当前只根据站点绑定 `agent_id`，未接收推广码。
- 在线充值入口：`backend/app/api/user/payment.py` 调用 `PaymentService.create_recharge_order()`。
- 在线充值入账：`backend/app/services/payment_service.py` 的 `_apply_paid_order()` 统一处理支付宝/微信回调与查单入账。
- 支付入账幂等：`payment_recharge_settlement` 通过 `order_no + asset_type` 唯一约束防止重复给用户到账。
- 余额流水：`ConsumptionRecord` 使用 `total_cost < 0` 表示余额增加。
- 图片积分流水：`ImageCreditRecord` 使用 `change_amount > 0` 表示图片积分增加。
- 资产来源：`BalanceService.get_asset_source_records()` 目前只支持 `balance`，并能将线上充值订单识别为“线上充值”。

### 前端结构

- 路由：`frontend/src/router/index.js` 已有用户端 `/user/recharge`、`/user/asset-source`，管理端、代理端独立布局。
- 用户侧菜单：`frontend/src/layout/UserLayout.vue`。
- 管理侧菜单：`frontend/src/layout/AdminLayout.vue`。
- 代理侧菜单：`frontend/src/layout/AgentLayout.vue`。
- API 封装：`frontend/src/api/user.js`、`frontend/src/api/payment.js`、`frontend/src/api/agent.js`。
- 页面风格：现有页面多使用 Ant Design Vue 的 `a-card`、`a-table`、筛选工具栏、统计卡片。

## 技术方案设计

### 1. 数据模型

新增推广相关表，使用单独升级脚本 `backend/sql/upgrade_user_promotion_20260610.sql`，并同步 `backend/sql/init.sql`、`backend/sql/initData.sql` 与根目录 `sql/initData.sql` 中必要结构。

#### `user_promotion_link`

记录用户生成的推广链接。

- `id`
- `user_id`：推广人用户 ID
- `agent_id`：推广人所属代理，直营为 NULL
- `site_scope`：`platform` / `agent`
- `site_host`：生成链接时使用的前台域名
- `invite_code`：唯一推广码
- `status`：`active` / `disabled`
- `register_count`
- `recharge_user_count`
- `total_reward_usd`
- `total_reward_image_credits`
- `created_at`
- `updated_at`

唯一约束：

- `uk_user_promotion_link_code(invite_code)`
- `uk_user_promotion_link_user(user_id)`，第一阶段每个用户固定一个推广码，便于统计和幂等。

#### `user_promotion_relation`

记录被推广用户与推广人的绑定关系。

- `id`
- `promoter_user_id`
- `promoter_agent_id`
- `invite_code`
- `invite_link_id`
- `invited_user_id`
- `invited_agent_id`
- `site_scope`
- `site_host`
- `first_recharged_at`
- `total_recharge_cny`
- `total_reward_usd`
- `total_reward_image_credits`
- `created_at`
- `updated_at`

唯一约束：

- `uk_user_promotion_relation_invited(invited_user_id)`，一个用户只绑定一个推广人。

关键限制：

- 不允许自己推广自己。
- 不允许跨站推广：直营推广码只能绑定直营注册用户；代理推广码只能绑定同一个 `agent_id` 的注册用户。
- 管理员/代理管理员不参与普通用户推广返现，推广人和被推广人都限定 `role='user'`。

#### `user_promotion_reward`

记录每笔充值产生的推广返现。

- `id`
- `relation_id`
- `promoter_user_id`
- `promoter_agent_id`
- `invited_user_id`
- `invite_code`
- `order_id`
- `order_no`
- `recharge_type`：`balance` / `image_credit`
- `amount_cny`
- `credited_usd`
- `credited_image_credits`
- `reward_asset_type`：`balance` / `image_credit`
- `reward_amount`
- `reward_rate`：默认 0.5
- `status`：`applied`
- `created_at`

唯一约束：

- `uk_user_promotion_reward_order_asset(order_no, reward_asset_type)`，同一支付订单同一返现资产只返现一次。

当前系统一笔订单最多产生一种到账资产，因此第一阶段一笔订单最多产生一条推广返现记录；使用 `order_no + reward_asset_type` 是为了保留后续组合充值扩展空间。

### 2. 推广链接生成

新增 `PromotionService`：

- `get_or_create_user_link(db, user, site_context)`：
  - 复用已有推广码；没有则生成短码。
  - 邀请码规则：8 位大写字母数字，排除 `0/O/1/I/L` 等易混淆字符；唯一键冲突时最多重试 5 次，仍失败则抛出 `ServiceException`。
  - 直营用户链接域名固定为平台前台域名，优先从 `settings.PLATFORM_FRONTEND_HOSTS` 取非 API 域名，默认 `www.xiaoleai.team`。
  - 代理用户链接域名固定使用代理档案中的 `frontend_domain`；本地开发环境才允许回显当前请求域名。
  - 返回完整链接：`https://{host}/register?invite_code={code}`。
- `get_user_dashboard(db, user)`：
  - 返回链接信息、推广注册数、已充值人数、累计返现、近期被推广用户。

### 3. 注册绑定

后端改动：

- `RegisterRequest` 新增可选 `invite_code`。
- `AuthService.register()` 新增 `invite_code` 参数。
- 注册创建用户并 `flush()` 后调用 `PromotionService.bind_invited_user()`：
  - 校验推广码存在、启用。
  - 校验站点归属与推广人归属一致。
  - 校验推广人不是当前新用户。
  - 插入 `user_promotion_relation`。
  - 更新 `user_promotion_link.register_count`。
  - 任一校验失败时抛出 `ServiceException`，注册事务整体回滚。

前端改动：

- 登录/注册页读取 URL 查询参数 `invite_code`，注册提交时传给 `/api/auth/register`。
- 保留现有注册页使用 `Login.vue` 的结构，不新增单独公开注册页面。

### 4. 支付返现

在 `PaymentService._apply_paid_order()` 中，当前仅当 `is_new_settlement` 为 true 时执行用户入账和代理现金分润。推广返现也放在该分支内，确保只有真实首次入账时执行。

新增 `PromotionService.apply_recharge_reward(db, order)`：

- 通过 `order.user_id` 查询推广关系。
- 若无推广关系则返回。
- 查询 `user_promotion_reward` 是否已存在该 `order_no`，存在则返回，保证幂等。
- 按 `order.recharge_type` 计算返现：
  - `balance`：`reward_amount = order.credited_usd * 0.5`，资产类型 `balance`。
  - `image_credit`：`reward_amount = order.credited_image_credits * 0.5`，资产类型 `image_credit`。
- 给推广人增加对应资产：
  - 余额：锁定 `UserBalance`，只增加 `balance`，不增加 `total_recharged`；写入 `ConsumptionRecord`，`total_cost = -reward_amount`，`request_id = order.order_no`，`model_name = '推广返现'`，`remark = '推广返现：{被推广用户名} 在线充值'`。
  - 图片积分：锁定/创建 `UserImageBalance`，只增加 `balance`，不增加 `total_recharged`；写入 `ImageCreditRecord`，`action_type='promotion_reward'`，`request_id = order.order_no`。
- 插入 `user_promotion_reward`。
- 更新 `user_promotion_relation` 的累计充值/返现/首充时间。
- 更新 `user_promotion_link` 的已充值人数、累计返现。

幂等要求：

- `apply_recharge_reward()` 必须允许重复调用。
- 统计字段更新必须发生在成功插入 `user_promotion_reward` 之后，并与 reward 明细同事务提交。
- 重复调用遇到唯一键冲突时，回滚嵌套事务并安全返回，不再次改余额或累计字段。

### 5. 资产来源展示

- `BalanceService.get_asset_source_records()` 增加识别 `ConsumptionRecord.model_name == '推广返现'` 或推广返现记录，展示来源为“推广返现”。
- 将 `ASSET_TYPES` 扩展为 `{"all", "balance", "image_credit"}`，并把 `ImageCreditRecord` 合并到资产来源列表，用户可看到图片积分返现来源。
- 合并分页使用标准字段 `union_all`：
  - `id`
  - `asset_type`
  - `signed_amount`
  - `balance_before`
  - `balance_after`
  - `request_id`
  - `model_name`
  - `agent_id`
  - `operator_id`
  - `action_type`
  - `billing_mode`
  - `remark`
  - `created_at`
- 对当前页 `request_id` 批量查询 `payment_recharge_order` 和 `user_promotion_reward`，来源判定顺序为：推广返现、线上充值、代理端、管理端、历史充值/系统扣减。
- 用户端 `AssetSource.vue` 增加资产类型筛选和图片积分余额卡片。

## 风险点与处理

- 重复返现风险：通过 `_apply_paid_order()` 的 `is_new_settlement` 分支和 `user_promotion_reward(order_no, reward_asset_type)` 唯一键双重保证。
- 跨站绑定风险：注册绑定时校验推广码所属 `agent_id/site_scope` 与新用户一致。
- 推广链接域名错误风险：统一通过 `AgentService` 解析站点；直营使用平台前台域名，代理使用 `frontend_domain`。
- 老数据兼容：新增表不影响旧用户；未通过推广注册的用户充值不触发返现；旧用户首次访问推广页时懒创建推广链接，不补历史关系。
- 资产来源混合展示：余额与图片积分精度不同，前后端需要明确 `asset_type` 和展示单位。
- 代理现金分润与用户推广返现并存：代理现金分润仍走 `AgentCashLedger`；用户推广返现走用户余额/图片积分流水，两者互不覆盖。
- 统计漂移风险：推广统计以 `user_promotion_reward` 明细为准；如累计字段漂移，后续可通过明细重算。
- 前端邀请码丢失风险：`Login.vue` 登录/注册模式切换时保留 `invite_code` query。
