# 数据库初始化脚本同步 Plan

## 用户原始需求

阅读当前目录下 `sql/initData.sql` 和 `sql/ai_model_proxy.sql`。当前 `initData.sql` 是推送到 GitHub 上用于创建表和初始化数据的脚本，而 `ai_model_proxy.sql` 是从正式环境数据库导出的所有表和字段 SQL。需要确保初始化脚本与正式环境数据库的表字段一致。

用户后续确认采用的新方案：以正式环境导出的完整结构为准，保留最新、最全面的一份初始化脚本，并统一收敛为 `sql/initData.sql` 用于提交到 GitHub。

## 技术方案设计

以正式环境导出的 `sql/ai_model_proxy.sql` 作为结构基准，因为它来自正式环境导出，表和字段完整性更高；最终再将其收敛为仓库唯一初始化入口 `sql/initData.sql`。

实施方案：

1. 保留 `ai_model_proxy.sql` 的正式环境表结构定义作为同步基准。
2. 将原 `initData.sql` 中“最小系统初始化数据”追加到完整结构脚本末尾，并放在 `SET FOREIGN_KEY_CHECKS = 1;` 之前。
3. 清理不适合 Git 初始化脚本提交的导出噪音：
   - 删除 Navicat 导出头注释。
   - 删除各表 `ENGINE=InnoDB AUTO_INCREMENT=...` 中的当前自增值。
   - 统一排序规则到 `utf8mb4_unicode_ci`，消除导出环境差异。
4. 保留 `DROP TABLE IF EXISTS`、`SET NAMES utf8mb4`、`SET FOREIGN_KEY_CHECKS` 等可执行初始化语句，并补充旧视图清理逻辑。
5. 将最终脚本落为 `sql/initData.sql`，删除同目录重复的 `sql/ai_model_proxy.sql`，并更新部署说明中的初始化入口。
