# 无限套餐每日限额 Plan

## 用户原始需求

当前项目有无限包月、包周和包日套餐，管理员可在 `admin/subscription` 页面开通套餐，代理端可在 `agent/subscription` 页面发放套餐。目前无限套餐可能被用户分发或分享给多人使用，需要限制无限套餐每天最多 3 亿 token，用完后提示已超出实际使用额度，防止滥用。

## 现状分析

- 后端已有 `subscription_plan`、`user_subscription`、`subscription_usage_cycle`、`request_log`、`consumption_record` 等套餐和用量表。
- `daily_quota` 套餐已经通过 `SubscriptionService.check_quota_before_request()` 和 `consume_quota_after_request()` 做每日周期校验。
- `unlimited` 套餐当前在请求前直接放行，请求后也只记录套餐消费，不写入每日周期额度，因此日包、周包、月包无限套餐都没有 token 天花板。
- 代理端发放和管理员发放最终都会进入 `SubscriptionService.activate_plan_subscription()` 或旧版 `activate_subscription()`，因此限制应放在实际请求计费链路，而不是只改开通页面。
- 现有 `daily_quota` 超额后允许余额兜底；本需求面向无限套餐防滥用，超出 3 亿后应直接阻断，不走余额兜底。

## 技术方案设计

- 在 `SubscriptionService` 中新增无限套餐每日 token 默认额度常量：`300000000`。
- 复用 `subscription_usage_cycle` 作为无限套餐每日周期账本，`unlimited` 套餐使用 `total_tokens` 口径，额度快照固定为 3 亿 token。
- 请求前：
  - 有活动 `unlimited` 套餐时也调用额度预检。
  - 如果当天已用 >= 3 亿，直接返回 “已超出实际使用额度...”。
  - 如果本次请求预计 token 会超过剩余额度，也提前阻断。
- 请求后：
  - `unlimited` 套餐成功请求也写入周期 used_amount/request_count/last_request_id。
  - 如果实际 raw token 使当天累计超过 3 亿，抛出明确业务错误，不按余额兜底。
- 展示层：
  - 套餐模板和发放记录中不再只显示“无限/不限量”，改为“每日 3 亿 Token”。
  - 后端摘要给 unlimited 也返回 current_cycle，便于管理端看到当天已用和剩余。
