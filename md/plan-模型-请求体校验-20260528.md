# 模型-请求体校验 Plan

## 用户原始需求

当前系统作为用户请求到上游模型服务的中转站，会处理用户请求并转发给上游。现状缺少对用户请求体中 `model` 字段的检查和限制，例如系统已配置 `gpt5.4`、`gpt5.5`，但用户请求不存在的 `gpt5.2` 时可能报错或继续转发给上游。需要在本系统前置拦截：当请求模型不存在时直接返回错误信息，提醒用户请求模型不存在，并阅读模型列表页进行配置。

## 技术方案设计

1. 统一模型存在性标准：用户请求的 `model` 必须在 `unified_model` 表中存在且 `enabled = 1`。
2. 保留现有模型覆盖规则能力：只有请求模型本身已存在时，才允许后续 `ModelService.resolve_model()` 应用覆盖规则，将已配置模型重定向到目标模型。
3. 在代理入口统一拦截，不让未知模型进入额度预检、渠道筛选或上游请求流程。
4. 覆盖范围包括：
   - OpenAI Chat Completions：`/v1/chat/completions`、`/chat/completions`
   - OpenAI Responses：`/v1/responses`、`/responses`、Responses WebSocket
   - Anthropic Messages：`/v1/messages`、`/messages`
   - 图片生成/编辑：`/v1/images/generations`、`/v1/image/created`、`/v1/image/edit`、`/v1/images/edits`
   - 视频创建：`/v1/videos`、`/v1/created/video`
5. 错误响应使用现有 `ServiceException` 和全局异常处理器，返回中文提示：`请求模型 '{model}' 不存在，请阅读模型列表页进行配置`。

## 实施步骤概要

1. 在 `ModelService` 增加 `get_enabled_model_by_name()`，只做 `unified_model.model_name` 精确匹配。
2. 在 `ProxyService` 增加 `_resolve_requested_model_or_raise()`：
   - `model` 为空时返回 400。
   - `model` 不在启用模型列表时返回 404，并提示阅读模型列表页配置。
   - 模型存在时再调用 `ModelService.resolve_model()` 保留覆盖规则。
3. 替换文本、Responses、图片、视频入口中的直接 `resolve_model()` 调用。
4. 增加单测覆盖：
   - 未配置模型即使覆盖规则可命中，也不会继续解析或转发。
   - 已配置模型仍可应用覆盖规则目标模型。
5. 运行相关测试，创建实施记录和 Review 文档。
