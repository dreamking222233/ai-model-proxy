# 模型身份保护实施方案

## 用户原始需求

用户请求 `claude-opus-4-8`、`claude-opus-4-6` 等映射模型时，上游回复会暴露 `Codex`、`GPT-5` 等实际调用模型信息。需要确保用户询问“当前是什么模型、具体型号”时，只看到用户请求模型 ID；同时不能使用粗暴硬编码替换，不能影响“GPT 是谁开发的”等正常知识问题。

## 技术方案设计

1. 保留动态厂商映射，按请求模型判断公开模型身份。
2. 对 Anthropic `/v1/messages` 请求，在注入系统提示前保存原始请求快照。
3. 窄范围识别“当前助手身份/模型型号”问题，命中时返回本地 Anthropic 兼容响应，不请求上游。
4. 合成本地成功日志使用合法协议类型，避免数据库枚举不兼容。
5. 移除流式广义身份文本缓冲清洗的参与，避免把普通知识回答中的 `GPT/OpenAI` 错误替换为用户请求模型。

## 涉及文件清单

- `backend/app/services/proxy_service.py`

## 实施步骤概要

1. 调整合成日志协议字段。
2. 调整 Anthropic 入口短路判断时机。
3. 增加短路命中日志，便于服务器验证。
4. 移除流式文本中的广义身份清洗副作用。
5. 本地语法检查和核心函数验证。
