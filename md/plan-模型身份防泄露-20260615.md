# 模型身份防泄露 Plan

## 用户原始需求

当前系统已经对模型映射、页面展示和部分请求 prompt 做了处理，但用户请求 `claude-opus-4-8` 时，上游回复了 `Codex，based on GPT-5` 等实际底层身份信息。需要确保用户询问模型身份或具体型号时，最终只看到用户请求的模型型号，不能得知真实实际调用模型。

同时，服务端添加的内部提示词不能出现在用户可见请求体或最终响应内容中。

## 技术方案设计

1. 统一增强模型身份提示
   - 不再只对 `claude-opus-4-6` 使用强提示。
   - 所有 `claude-*` 模型都追加明确身份规则：对外只使用用户请求模型名，不提及 Codex、GPT、OpenAI、上游模型、代理、路由等内部信息。

2. 调整内部提示插入位置
   - OpenAI / Responses 保持前置系统提示。
   - Anthropic 协议中将身份保护提示追加到顶层 `system` 末尾，让它在客户端自带 system 之后兜底约束身份表达。

3. 增加响应正文兜底清洗
   - 在用户可见文本中检测内部模型泄露信号。
   - 将 `actual_model`、`responses:` 后缀模型、Codex/GPT/OpenAI 等内部身份片段替换或收敛为 `requested_model`。
   - 对明显身份询问类回复，直接收敛成 `当前模型：{requested_model}`，避免流式文本拼接后仍泄露。

4. 保持管理员排查能力
   - `request_log.actual_model` 和 admin/logs 仍保存真实实际模型。
   - 只清洗用户侧响应正文和模型字段。
