# 生图调用优化实施方案

## 用户原始需求

当前系统接入上游的生图调用走 OpenAI 渠道、OpenAI Image Native Size 类型，上游 CPA 支持以下接口：

- `POST https://api.xiaoleai.team/v1/images/generations`
- `POST https://api.xiaoleai.team/v1/image/created`
- `POST https://api.xiaoleai.team/v1/image/edit`
- `POST https://api.xiaoleai.team/v1/images/edits`

需要优化当前系统生图调用，解决用户在 `user/chat` 和 `user/m-chat` 页面调用生图时等待一百多秒、页面反馈“没返回”、以及偶发报错“生图请求未发送成功，请检查API基础地址配置、同源代理或浏览器控制台中的跨域报错”的问题。

同时要求：

- 优化上游调用方式，适配 CPA 支持的 OpenAI 图片接口。
- `user/chat` 和 `user/m-chat` 增加更持久的等待时间。
- 优化生图计费：确保上游返回 base64，并且系统返回给用户的是有效图片后才计费。
- 扩展 `/v1/responses` 支持调用生图请求。
- 更新 `user/quickstart` 页面图像 API 区域介绍。
- 提供一版最详细的 Python 调用示例代码。

## 技术方案设计

### 后端上游调用

当前 `ProxyService` 已存在 OpenAI 图片分支，并且 `openai-image-native-size` 会向上游透传 `size`、`quality`、`response_format=b64_json`。本次保留该主链路，补强以下能力：

1. 将图片请求上游超时时间从 300 秒提高到 600 秒，覆盖 CPA 生图 100-200 秒的正常等待区间。
2. 对 OpenAI Native Size 图片生成和编辑补充同族接口 fallback：
   - 生成优先 `/v1/images/generations`，当路由不可用时 fallback `/v1/image/created`。
   - 编辑优先 `/v1/images/edits`，当路由不可用时 fallback `/v1/image/edit`。
   - fallback 只针对 404/405 等路由级失败，避免 400 内容错误导致重复生成。
3. OpenAI 图片响应解析时验证 `data[].b64_json`：
   - 必须是可解码 base64。
   - 解码后必须符合 PNG/JPEG/WebP/GIF 等常见图片魔数。
   - 校正或补齐 `mime_type`。
   - 只有通过验证的图片才进入返回 payload 和计费流程。
4. 计费仍保持当前后置扣费逻辑：上游成功、图片解析和有效性校验通过之后再扣图片积分并写请求日志。

### `/v1/responses` 生图支持

当前 `/v1/responses` 会拒绝图片模型，并会移除 `image_generation` 工具。本次新增独立识别和转换流程：

1. 如果请求满足任一条件，识别为 Responses 生图请求：
   - `model` 是图片模型或图片积分模型。
   - `tools` 或 `tool_choice` 中包含 `type=image_generation`。
2. 从 Responses 请求中提取图片请求参数：
   - `model` 优先使用 image tool 的 `model`，其次使用请求 `model`。
   - `prompt` 从 `input` 中最后一条用户文本或全部 `input_text/text` 拼接提取。
   - 透传 `size`、`quality`、`image_size`、`aspect_ratio`、`n`，固定 `response_format=b64_json`。
3. 内部复用 `handle_image_request`，让图片渠道选择、余额预检、图片有效性校验、扣费和日志保持一致。
4. 返回格式：
   - 非流式返回标准 Responses object，`output` 中包含 `image_generation_call`，`result` 为 base64，附带 `size/quality/revised_prompt` 等可用元信息。
   - 流式返回 SSE，等待期间通过已有 heartbeat 保持连接，完成后发送 `response.created`、`response.completed` 和 `[DONE]`。

### 前端等待与错误处理

1. `frontend/src/views/common/AiChat.vue`
   - `IMAGE_REQUEST_TIMEOUT_MS` 从 300000 提升到 600000。
   - 生成/编辑占位文案改为“可能需要数分钟”。
   - `AbortError` 文案明确为“等待超过 10 分钟”。
   - `Failed to fetch` 文案改为网络/代理/服务连接问题，不再优先引导 CORS。
2. `frontend/src/views/common/MobileChat.vue`
   - 增加同样的 600 秒 AbortController。
   - 增强错误响应 JSON 解析。
   - 校验 `data` 非空后再渲染。
   - 失败时刷新余额，避免用户误以为已扣费。

### QuickStart 文档

更新 `frontend/src/views/user/QuickStart.vue` 图像 API 区域：

- 明确四个图片接口均可用。
- 说明 OpenAI Image Native Size 渠道支持 `image_size=1K/2K/4K` 和 `aspect_ratio` 映射原生像素尺寸。
- 说明请固定 `response_format=b64_json`，返回后自行 base64 解码保存。
- 说明长耗时建议客户端 timeout 设置为 600 秒以上。
- 提供详细 Python 示例：生成、编辑、Responses 生图、base64 校验、保存文件、错误处理。

### 风险点

- Responses 生图如果直接走文本计费会产生错误账单，因此必须在文本模型上下文前先分流到图片处理。
- 图片 URL fallback 不能对已经可能触发生成的 5xx/超时盲目切换接口，否则可能重复生成和重复扣上游费用。
- base64 校验不能过度严格到拒绝真实图片格式，因此只做通用魔数校验。
- 前端 timeout 增长会让用户等待更久，需要文案明确“仍在生成”。

## Plan v2 修订

根据 `md/plan-review-生图调用优化-20260609.md` 的评估意见，补充以下实施约束，作为最终执行版本。

### Responses 生图分流位置

- `handle_responses_request` 中在完成 `client_request = copy.deepcopy(request_data)`、`requested_model` 和 `is_stream` 标准化后，立即执行 `_is_responses_image_request(db, client_request)`。
- 命中生图请求时直接进入 `_handle_responses_image_request`。
- 该分支不得先执行：
  - `_build_text_billing_context`
  - `_inject_model_identity`
  - `_prepare_responses_request_context`
  - `_remove_responses_image_generation_tool`
- 目的：避免图片模型被文本 Responses 校验拒绝，也避免 `image_generation` tool 被剔除后丢失图片请求意图。

### Responses 工具型生图模型来源

- `image_generation` tool 支持字段：
  - `model`
  - `size`
  - `quality`
  - `image_size`
  - `imageSize`
  - `aspect_ratio`
  - `n`
  - `response_format`
- 模型解析规则：
  - 优先使用 tool 内的 `model`。
  - 若请求 `model` 本身是图片模型，则使用请求 `model`。
  - 若 tool 未提供 `model` 且请求 `model` 不是图片模型，返回 400，提示 `Responses image_generation 工具请求必须指定图片模型`。
- 不使用隐式默认图片模型，避免多租户/多模型场景中意外扣错模型积分。

### Responses websocket 范围

- 本次仅支持 HTTP `POST /v1/responses` 生图。
- websocket `/v1/responses` 暂不支持图片生成，继续沿用现有文本 Responses 流程和工具拦截逻辑。
- 原因：websocket 会维护多轮状态和上轮 output，图片生图需要新增状态同步与事件序列约定，超出本次故障修复范围。

### 有效图片数计费

- 余额预检仍按请求 `n` 计算最高可能扣费，避免上游成功后余额不足。
- 实际扣费必须按 `len(valid_images)` 重新计算：
  - `actual_charged_credits = model_multiplier * len(valid_images)`
  - 返回 `usage.image_credits_charged` 使用实际扣费金额。
  - `RequestLog.image_count` 使用有效图片数。
- 如果上游返回图片全部无效，则不扣费，记录失败请求，返回上游图片结果无效。

### OpenAI 图片接口 fallback

- 新增图片专用 helper，不修改通用 `_post_with_retries`。
- 生成候选 URL：
  - 首选 `_resolve_openai_image_generation_url(...)`
  - 同族备用 URL：`/v1/image/created` 或 `/v1/images/generations`
- 编辑候选 URL：
  - 首选 `_resolve_openai_image_edit_url(...)`
  - 同族备用 URL：`/v1/image/edit` 或 `/v1/images/edits`
- 仅当首选 URL 返回 404 或 405 时 fallback。
- 不对 408/409/429/5xx/连接超时做跨 URL fallback，避免上游可能已开始生成导致重复生成。
- 日志记录首选 URL、fallback URL、fallback 原因、最终 URL、有效图片数和过滤掉的无效图片数；不记录 base64 原文。

### Base64 图片校验

- 支持去除 `data:image/...;base64,` 前缀。
- 支持去除空白换行和补齐 padding。
- 解码使用严格校验。
- 图片魔数支持 PNG、JPEG、WebP、GIF。
- 暂不接受 SVG，避免可执行脚本类内容被当作图片返回。
