# 用户-套餐选购 Plan

日期：20260622

## Plan v2 修订说明

根据 `./md/plan-review-用户-套餐选购-20260622.md`，本方案已补充以下硬性约束：

1. 套餐订单 `amount_cny` 可不传，`subscription_plan_id` 必填，最终金额完全以后端套餐售价为准。
2. 支付成功入账必须显式区分 `balance/image_credit/subscription`，套餐订单不能落入余额入账。
3. `PaymentRechargeSettlement.asset_type` 必须使用独立的 `subscription` 幂等类型。
4. 套餐订单必须跳过现有推广返利和代理现金余额自动入账。
5. 代理用户购买时必须满足 `agent_cost_price_cny > 0` 且 `sale_price_cny >= agent_cost_price_cny`，否则拒绝下单。
6. 套餐返现金额不复用 `payment_recharge_order.agent_income_cny`，只写入套餐快照字段和代理套餐销售记录。
7. 在线自购套餐的 `UserSubscription.created_by` 保持为空，来源通过支付订单 `order_no/subscription_id` 追溯。
8. 数据库升级脚本必须按项目现有幂等风格编写。
9. 线上开通套餐后必须进入当前系统既有套餐扣费链路：每日套餐额度优先消耗，额度不足部分继续检查并扣用户余额，套餐额度和余额都不足时停止使用并返回余额不足错误。

## 用户原始需求

当前系统已有套餐能力：

- 管理端 `admin/subscription` 可以新增套餐模板、为用户发放套餐。
- 代理端 `agent/subscription` 可以查看套餐库存、为代理名下用户发放套餐。
- 用户端 `user/recharge` 目前只能按量充值余额或图片积分。
- 余额充值比例示例：用户 `1 RMB = 5 USD`；代理结算比例 `1 RMB = 10 USD`，因此代理用户充值时会产生代理现金分润。

本次需要新增：

1. 用户端充值页面新增“套餐选购”选项，用户可自行购买套餐。
2. 管理端套餐模板支持配置不同价格：
   - 用户售价，例如至尊包月套餐售价 `450` 元。
   - 代理拿货价，例如代理拿货 `360` 元。
3. 直营客户购买套餐：直接付款并开通套餐。
4. 代理名下用户购买套餐：付款并开通套餐，同时产生代理返现数据，返现金额示例为 `售价 450 - 代理拿货价 360 = 90` 元。
5. 管理端新增“代理套餐销售页”，用于查看代理用户购买套餐情况，后续线下给代理返现，并支持核销。
6. 代理端新增页面，代理可查看自己名下套餐销售数据。
7. 线上购买套餐后的订阅效果必须与管理员/代理为用户发放套餐保持一致，统一走现有套餐开通逻辑；当前套餐发放支持“追加/覆盖”，但线上购买一律按“追加”处理，不提供覆盖选项。

## 方案设计

### 设计原则

1. 用户前台购买套餐必须走现有支付链路，复用支付宝/微信下单、回调、查单、幂等保护、订单列表与回跳机制。
2. 套餐价格必须在创建订单时写快照，后续管理员修改套餐价格不能影响已创建订单。
3. 支付成功后的套餐开通必须幂等，重复回调或手动同步不能重复开通套餐、不能重复生成代理返现记录。
4. 代理用户购买套餐产生“待核销返现记录”，默认不自动增加 `agent_cash_balance`，因为需求强调后续人工返现和核销。
5. 直营用户购买套餐不产生代理返现记录。
6. 代理端只能看到自己代理名下的销售数据；管理端可全局查看并核销。
7. 前台只展示启用且允许在线购买且价格有效的套餐。
8. 线上套餐购买必须复用 `SubscriptionService.activate_plan_subscription`，不能另写一套套餐订阅计算逻辑；`activation_mode` 固定为 `append`，效果与管理端/代理端选择“追加”一致。
9. 线上购买只负责创建标准 `UserSubscription` 记录并刷新用户套餐缓存态，不改动 `ProxyService` 现有请求扣费策略；后续模型请求仍按当前系统的套餐额度、余额兜底、余额不足拦截逻辑执行。

### 数据库设计

#### 1. 扩展 `subscription_plan`

新增字段：

- `sale_price_cny DECIMAL(12,2) NOT NULL DEFAULT 0 COMMENT '用户在线购买售价 RMB'`
- `agent_cost_price_cny DECIMAL(12,2) NOT NULL DEFAULT 0 COMMENT '代理拿货价 RMB'`
- `online_sale_enabled TINYINT NOT NULL DEFAULT 0 COMMENT '是否允许用户前台在线购买'`

校验规则：

- `sale_price_cny >= 0`
- `agent_cost_price_cny >= 0`
- 若 `online_sale_enabled=1`，则 `sale_price_cny > 0`
- 直营用户购买：只校验 `sale_price_cny > 0`。
- 代理用户购买：必须满足 `agent_cost_price_cny > 0` 且 `sale_price_cny >= agent_cost_price_cny`，否则拒绝下单。
- 代理用户返现金额按 `sale_price_cny - agent_cost_price_cny` 计算。

#### 2. 扩展 `payment_recharge_order`

将 `recharge_type` 扩展为 `balance/image_credit/subscription`。

新增套餐购买快照字段：

- `subscription_plan_id BIGINT NULL`
- `subscription_id BIGINT NULL`
- `plan_code_snapshot VARCHAR(64) NULL`
- `plan_name_snapshot VARCHAR(128) NULL`
- `plan_kind_snapshot VARCHAR(32) NULL`
- `duration_days_snapshot INT NULL`
- `quota_metric_snapshot VARCHAR(32) NULL`
- `quota_value_snapshot DECIMAL(20,6) NULL`
- `subscription_activation_mode VARCHAR(20) NOT NULL DEFAULT 'append'`
- `subscription_sale_price_cny DECIMAL(12,2) NOT NULL DEFAULT 0`
- `subscription_agent_cost_cny DECIMAL(12,2) NOT NULL DEFAULT 0`
- `subscription_agent_rebate_cny DECIMAL(12,2) NOT NULL DEFAULT 0`

说明：

- `amount_cny` 对套餐订单等于 `subscription_sale_price_cny`。
- `credited_usd/credited_image_credits` 对套餐订单保持 `0`。
- 套餐订单不复用 `agent_income_cny`，该字段保持 `0`，避免现有代理现金分润/支付明细页面误认为已进入现金余额。
- 套餐返现金额只写入 `subscription_agent_rebate_cny` 和 `agent_subscription_sale_record.agent_rebate_cny`。
- `subject/body` 使用套餐名称，例如 `AI 平台套餐购买 - 至尊包月套餐`。

#### 3. 扩展 `payment_recharge_settlement`

允许 `asset_type='subscription'`。

支付成功时继续通过唯一键 `(order_no, asset_type)` 做幂等，避免重复开通套餐。

实现要求：

- `_settlement_asset_type` 对 `subscription` 必须返回 `subscription`。
- 升级脚本和初始化 SQL 中 `asset_type` 注释同步更新为 `balance/image_credit/subscription`。
- 不允许套餐订单复用 `balance` 幂等类型。

#### 4. 新增 `agent_subscription_sale_record`

用于管理端代理套餐销售页、代理端销售数据页和核销。

字段建议：

- `id BIGINT PK`
- `order_id BIGINT NOT NULL`
- `order_no VARCHAR(64) NOT NULL UNIQUE`
- `agent_id BIGINT NOT NULL INDEX`
- `user_id BIGINT NOT NULL INDEX`
- `subscription_id BIGINT NULL INDEX`
- `plan_id BIGINT NULL INDEX`
- `plan_code_snapshot VARCHAR(64) NULL`
- `plan_name_snapshot VARCHAR(128) NOT NULL`
- `plan_kind_snapshot VARCHAR(32) NULL`
- `duration_days_snapshot INT NULL`
- `sale_price_cny DECIMAL(12,2) NOT NULL DEFAULT 0`
- `agent_cost_price_cny DECIMAL(12,2) NOT NULL DEFAULT 0`
- `agent_rebate_cny DECIMAL(12,2) NOT NULL DEFAULT 0`
- `payment_channel VARCHAR(32) NOT NULL`
- `payment_status VARCHAR(16) NOT NULL DEFAULT 'paid'`
- `rebate_status VARCHAR(16) NOT NULL DEFAULT 'pending' COMMENT 'pending/settled/void'`
- `settled_at DATETIME NULL`
- `settled_by BIGINT NULL`
- `settlement_remark VARCHAR(255) NULL`
- `created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP`
- `updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP`

索引：

- `uk_agent_subscription_sale_order(order_no)`
- `idx_agent_subscription_sale_agent_status(agent_id, rebate_status)`
- `idx_agent_subscription_sale_created(created_at)`
- `idx_agent_subscription_sale_plan(plan_id)`
- `idx_agent_subscription_sale_user(user_id)`

状态语义：

- `pending`：已支付、已开通套餐、待给代理返现。
- `settled`：管理员已线下返现并核销。
- `void`：异常作废，例如订单退款或人工确认无需返现。本期不做退款闭环，只保留状态。

#### 3. 套餐扣费链路保持不变

当前系统请求扣费链路已在 `ProxyService` 和 `SubscriptionService` 中实现：

- 请求前通过 `SubscriptionService.refresh_user_subscription_state` 刷新用户套餐状态。
- 有有效套餐时，通过 `SubscriptionService.check_quota_before_request` 做每日额度预检。
- 如果套餐每日额度不足，`ProxyService._estimate_balance_charge_after_quota` / `_can_fallback_to_balance_for_quota_precheck` 会检查余额是否能承担超出套餐额度的部分。
- 请求后实际扣费时，`ProxyService._deduct_balance_and_log_once` 会按以下顺序处理：
  - 优先消费套餐每日额度。
  - 套餐额度不足但用户余额可支付时，超出部分扣余额，账单 `billing_mode` 记录为 `mixed` 或 `balance`。
  - 套餐额度不足且余额也不足时，抛出 `INSUFFICIENT_BALANCE`，停止继续使用。

本次线上套餐购买不得新增或替换上述扣费逻辑，只能通过 `SubscriptionService.activate_plan_subscription(..., activation_mode='append')` 生成与管理员/代理发放一致的 `UserSubscription`。只要线上购买生成的套餐快照字段与模板发放一致，后续每日 100 美元额度、额度用尽后扣余额、余额不足停止使用的行为会自动复用现有链路。

推广返利处理：

- 当前 `_apply_paid_order` 会无差别调用 `PromotionService.apply_recharge_reward(db, locked)`。
- 本次必须在 `PaymentService._apply_paid_order` 调用侧或 `PromotionService.apply_recharge_reward` 方法入口显式跳过 `recharge_type=subscription`，避免套餐订单误按余额/图片积分充值返利。
- 若后续需要“套餐购买也参与推广返利”，应单独设计返利口径，不能复用余额充值的资产返现。

#### 4. 支付请求 Schema 设计

`backend/app/schemas/payment.py` 中 `UserRechargeOrderCreateRequest` 调整为分支校验：

- `amount_cny: Optional[Decimal] = None`
- `payment_channel: alipay/wechat`
- `recharge_type: balance/image_credit/subscription`
- `subscription_plan_id: Optional[int] = None`

校验规则：

- `balance/image_credit`：`amount_cny` 必填且 `amount_cny >= 1`，`subscription_plan_id` 忽略。
- `subscription`：`subscription_plan_id` 必填且大于 0，`amount_cny` 可为空；即使前端传了金额，后端也必须忽略。

#### 5. 新增 `AgentSubscriptionSaleService`

建议文件：`backend/app/services/agent_subscription_sale_service.py`

职责：

### 风险点

1. 支付回调幂等：支付宝/微信异步通知、用户手动同步都可能触发入账，必须保证套餐只开通一次。
2. 价格快照：订单创建后套餐价格被修改，支付成功必须按订单快照处理。
3. 代理返现语义：本次是“待核销返现”，不应误进入现有 `agent_cash_balance`，否则会和余额充值自动分润重复。
4. 推广返利冲突：套餐购买如果继续触发余额充值推广返利，会产生错误资产返现，必须显式跳过。
5. 代理拿货价配置缺失：代理用户购买时必须有有效代理成本价，且代理成本价不能大于用户售价，否则拒绝下单。
6. 套餐开通时间：用户已有套餐时线上购买固定 `append` 顺延，避免前台购买覆盖当前套餐；管理员/代理手动发放仍保留 `append/override` 选择。
7. 用户角色限制：只有终端用户可购买套餐，管理员/代理账号不能走用户充值页购买。
8. 代理数据隔离：代理端销售 API 必须使用登录态 `agent_id`，不能信任前端传参。
9. 前端展示：套餐购买订单 `credited_usd=0`，原有订单表和弹窗必须按 `recharge_type=subscription` 分支展示，避免显示“到账 $0”。
10. 扣费链路回归：线上购买生成的套餐如果快照字段、用户缓存态或激活时间与后台发放不一致，可能导致每日额度、余额兜底或余额不足拦截异常。必须用测试覆盖线上购买套餐后的真实请求扣费行为。
