# 用户-资产来源 Plan

## 用户原始需求

深度分析当前项目及相关代码。当前管理端可以直接给用户添加余额或图片积分，代理端也可以给其用户添加，用户也可以通过在线充值页面充值余额或积分。需要在用户端新增一个页面，可以显示余额/积分来源。例如：

- 管理端给用户充值时可以添加备注，用户端看到 `+50 来源：管理端 备注：充值`，备注也可以是 `活动赠送`。
- 用户通过线上支付充值，用户端看到 `+50 来源：线上充值 备注：支付宝/微信`。

## 需求调整

2026-06-08 后续确认：当前用户端“资产来源”页面不需要显示用户调用模型产生的扣费记录，也不需要展示图片积分流水；页面只展示用户余额增加/减少记录，例如管理端或代理端为用户新增/扣除余额，以及用户线上余额充值。

## 技术方案设计 v2

### 现状分析

- 余额账户表是 `user_balance`，余额流水目前复用 `consumption_record`：
  - `total_cost < 0` 表示余额增加（充值）。
  - `total_cost > 0` 表示余额消耗或扣减。
  - 线上余额充值会写入 `request_id=order_no`、`model_name=支付宝/微信充值`。
  - 管理端/代理端余额充值目前没有备注字段落库，代理端余额充值甚至传入固定 `agent_recharge_balance`，用户无法看到真实业务备注。
- 图片积分账户表是 `user_image_balance`，图片积分流水是 `image_credit_record`：
  - `change_amount > 0` 表示积分增加。
  - 已有 `agent_id/operator_id/remark/action_type` 字段，可以承载来源与备注。
  - 线上图片积分充值会写入 `remark=用户在线充值图片积分`，支付渠道在 `model_name` 里。
- 用户端现有 `/api/user/balance/consumption` 与 `BalanceLog.vue` 更偏请求用量/账单，不适合直接塞入“来源记录”，否则会混淆请求消费、套餐消耗、充值来源。

### 设计原则

- 新增用户端独立页面和独立接口：只展示当前登录用户的余额增减来源。
- 页面定位为“余额明细/资产来源”：默认展示余额增减变动，支持按变动方向筛选，满足充值和人工扣减来源查看。
- 备注以“业务可读”为准：
  - 管理端、代理端充值/扣减请求都允许传 `reason`。
  - 线上充值备注显示支付渠道中文：`支付宝` / `微信支付`。
  - 模型调用扣费不在该页面展示，继续在消费明细页面查看。
- 数据库最小扩展：给 `consumption_record` 增加 `operator_id` 和 `remark` 字段，用于余额人工充值/扣减备注。图片积分表已有字段无需扩展。

### 用户端接口契约

新增接口：`GET /api/user/balance/asset-records`

查询参数：

- `page`：默认 1，最小 1。
- `page_size`：默认 20，范围 1-100。
- `direction`：`all/increase/decrease`，默认 `all`。

响应结构：

```json
{
  "list": [
    {
      "id": 123,
      "record_key": "balance-123",
      "asset_type": "balance",
      "asset_type_text": "余额",
      "direction": "increase",
      "amount": 50,
      "source": "线上充值",
      "remark": "支付宝",
      "request_id": "ALP202606081200000001",
      "model_name": "支付宝充值",
      "action_type": "recharge",
      "balance_before": 0,
      "balance_after": 50,
      "created_at": "2026-06-08T12:00:00"
    }
  ],
  "total": 1,
  "page": 1,
  "page_size": 20
}
```

字段语义：

- `asset_type` 固定为 `balance`，`amount` 单位为美元，`balance_before/after` 为美元余额。
- `direction=increase` 表示资产增加，`decrease` 表示资产减少。
- 用户端不返回 `operator_id`，避免暴露内部账号 ID。

### 分页策略

- 后端查询 `consumption_record` 余额流水。
- 只保留 `request_id/model_name` 为空的人工余额操作，以及已支付且 `recharge_type=balance` 的线上充值订单。
- 按 `created_at desc, id desc` 排序、分页和计数。
- 对当前页结果中的 `request_id` 批量查询已支付的 `payment_recharge_order`，构建 `order_no -> payment_channel/recharge_type` 映射，避免逐条查订单。

### 来源判定

- 余额流水 `consumption_record`：
  - `request_id` 对应已支付的 `payment_recharge_order.order_no` 且 `total_cost < 0`：来源 `线上充值`，备注取支付渠道中文。
  - `agent_id` 不为空且无线上订单：来源 `代理端`，备注取 `remark`。
  - `operator_id` 不为空且 `agent_id` 为空：来源 `管理端`，备注取 `remark`。
  - 其他无请求信息的扣减：来源 `管理端` 或 `系统扣减`，备注取 `remark`。

### 历史数据降级

- `total_cost < 0` 且没有在线订单、`agent_id/operator_id` 均为空：来源显示 `历史充值`，备注为空或取旧 `model_name`。
- `total_cost > 0` 且没有请求信息、`agent_id/operator_id` 均为空：来源显示 `系统扣减`。
- 空白备注在服务层 trim 后按 `None` 处理，前端显示 `-`。

## 风险点

- `consumption_record` 被请求账单、统计、套餐逻辑大量复用，新增字段必须保持可空并避免改变旧查询语义。
- 线上充值可能来自微信回调/支付宝回调/手动同步，来源接口应通过订单号关联订单，而不是依赖回调路径。
- 代理端发放余额存在“资产池”和“授信额度”两条路径，两条都必须写用户余额备注。
- 历史余额人工充值没有备注字段，展示时只能显示默认来源/空备注。
- 资产来源页必须在后端排除模型调用扣费，不能只在前端隐藏，否则分页总数会不准确。
- `consumption_record(user_id, created_at, id)` 与 `request_id` 需要补充索引，避免资产明细页随数据增长变慢。
