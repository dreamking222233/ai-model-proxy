# 用户专属倍率实施方案 v2

## Plan v2 修订说明

根据 `md/plan-review-用户专属倍率-20260627.md` 的评估意见，本版做以下关键修订：

- `user_price_adjustment_rule` 增加 `name` 字段，保持与现有全局价格调控规则一致，便于表单、审计和展示。
- 明确用户专属倍率的核心能力是“某个用户 + 某个模型系列/类型/计费类型”的规则，命中后覆盖全局分类规则。
- 新增 `image_credit_record` 媒体积分倍率来源快照，避免图片/视频积分流水无法审计。
- 明确异步视频任务创建时冻结调整后积分单价或规则解析结果，完成扣费不重新按当前规则漂移。
- 明确用户规则更新/删除必须校验 `rule.user_id == path user_id`。
- 将日志、余额、套餐序列化兼容检查纳入必做范围，不再作为纯可选增强。
- 新增 `admin/logs` 表格展示要求：在“响应时间”和“计费”之间增加“用户倍率”列，展示本次请求命中的倍率快照和来源。
- 增加自动化测试要求，覆盖用户规则优先级、回退、预检/扣费一致性和媒体积分倍率。

## 用户原始需求

深度分析当前目录下项目及相关文档。

当前系统可以在管理端 `admin/price-adjustments` 页面按模型系列、模型类型、计费类型设置价格倍率，例如 Claude、GPT 系列不同倍率，但这些规则对所有用户统一生效。

需要新增：

- 管理员可在 `admin/users` 页面为某个用户设置独属倍率。
- 例如给张三设置 `GPT 系列 x1`、`Claude 系列 x2`。
- `admin/price-adjustments` 页面需要能看到“给哪个用户设置了哪个模型系列的多少倍率”。
- `admin/logs` 页面需要在“响应时间”和“计费”之间增加一列，展示该用户本次请求实际使用的倍率。
- 计费链路需要按用户专属倍率执行，而不是所有用户只能共用一套分类倍率。

## 任务级别

大型任务。

原因：涉及数据库新表、价格规则解析优先级、文本和媒体积分计费主链路、管理端用户页、价格调控页、日志审计字段、初始化脚本和回归验证，步骤超过 10 个。

## 当前系统分析

### 项目结构

- 后端：FastAPI + SQLAlchemy，同步 MySQL ORM，核心代理和计费集中在 `backend/app/services/proxy_service.py`。
- 前端：Vue 2 + Ant Design Vue，管理端页面集中在 `frontend/src/views/admin/`。
- 管理端用户页：`frontend/src/views/admin/UserManage.vue`，接口为 `/api/admin/users`。
- 管理端价格调控页：`frontend/src/views/admin/PriceAdjustmentManage.vue`，接口为 `/api/admin/price-adjustments`。
- 文档规范：计划、实现、评审文档统一放在 `md/`。

### 现有价格调控

现有价格调控由 `model_price_adjustment_rule` 表承载：

- ORM：`backend/app/models/model.py` 的 `ModelPriceAdjustmentRule`。
- Schema：`backend/app/schemas/model.py` 的 `ModelPriceAdjustmentRuleCreate/Update/Info`。
- Service：`backend/app/services/price_adjustment_service.py`。
- API：`backend/app/api/admin/price_adjustment.py`。
- 前端 API：`frontend/src/api/priceAdjustment.js`。
- 前端页面：`frontend/src/views/admin/PriceAdjustmentManage.vue`。

规则维度：

- `model_series`：`gpt/claude/grok/gemini/other/all`
- `model_type`：`chat/image/video/embedding/completion/all`
- `billing_type`：`token/request/image_credit/free/all`
- `multiplier`
- `schedule_type`：`always/daily_time`
- `start_time/end_time`
- `priority/enabled`

当前解析入口：

```text
PriceAdjustmentService.resolve_multiplier(db, unified_model)
```

当前问题：

- 没有用户维度参数。
- 所有用户命中同一套 `model_price_adjustment_rule`。
- `admin/price-adjustments/effective` 只展示各系列/类型当前有效倍率，不展示用户专属配置。

### 现有计费链路

文本计费：

- 请求前额度预检：`ProxyService._build_text_quota_precheck(...)`
- 请求完成后实际扣费和日志：`ProxyService._deduct_balance_and_log_once(...)`
- 当前文本余额计费倍率：

```text
price_multiplier_snapshot = system_config.price_multiplier * model_price_adjustment_rule.multiplier
effective_price_multiplier = price_multiplier_snapshot * fast_price_multiplier * context_price_multiplier
```

媒体积分计费：

- 图片生成、图片编辑、视频生成复用媒体积分体系。
- 媒体积分不叠加 `system_config.price_multiplier`。
- 当前媒体积分倍率入口：

```text
ProxyService._apply_media_credit_price_adjustment(db, unified_model, base_multiplier)
```

当前问题：

- 文本和媒体积分都调用 `PriceAdjustmentService.resolve_multiplier(db, unified_model)`，没有传入用户。
- 若只改某一个调用点，会造成预检、扣费、图片、视频规则不一致。

### 现有日志审计

`request_log` 和 `consumption_record` 当前有：

- `price_multiplier_snapshot`
- `fast_price_multiplier_snapshot`
- `context_price_multiplier_snapshot`
- `token_multiplier_snapshot`

当前 `price_multiplier_snapshot` 已经混合存储：

```text
系统全局价格倍率 * 分类调控倍率
```

当前问题：

- 无法从历史账单拆分出系统倍率、分类倍率、用户倍率。
- 用户专属倍率上线后，如果继续只写入一个最终倍率，管理端可以显示当前配置，但历史账单难以审计“当时是否命中用户专属规则”。

## 技术方案设计

### 1. 数据模型设计

新增 `user_price_adjustment_rule` 表，用独立表承载用户专属倍率，不复用 `sys_user` 字段，也不把用户规则混入全局规则表。

建议结构：

```sql
CREATE TABLE `user_price_adjustment_rule` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  `name` VARCHAR(128) NOT NULL COMMENT '规则名称',
  `user_id` BIGINT UNSIGNED NOT NULL COMMENT '用户ID',
  `model_series` VARCHAR(32) NOT NULL DEFAULT 'all' COMMENT 'gpt/claude/grok/gemini/other/all',
  `model_type` VARCHAR(20) NOT NULL DEFAULT 'all' COMMENT 'chat/image/video/embedding/completion/all',
  `billing_type` VARCHAR(20) NOT NULL DEFAULT 'all' COMMENT 'token/request/image_credit/free/all',
  `multiplier` DECIMAL(12, 6) NOT NULL DEFAULT 1 COMMENT '用户专属价格倍率',
  `schedule_type` VARCHAR(20) NOT NULL DEFAULT 'always' COMMENT 'always/daily_time',
  `start_time` TIME DEFAULT NULL COMMENT '每日开始时间，北京时间',
  `end_time` TIME DEFAULT NULL COMMENT '每日结束时间，北京时间',
  `priority` INT NOT NULL DEFAULT 100 COMMENT '优先级，数字小优先',
  `enabled` TINYINT NOT NULL DEFAULT 1,
  `description` TEXT DEFAULT NULL,
  `created_by` BIGINT UNSIGNED DEFAULT NULL COMMENT '创建管理员ID',
  `updated_by` BIGINT UNSIGNED DEFAULT NULL COMMENT '最后更新管理员ID',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  KEY `idx_user_price_adjustment_user` (`user_id`, `enabled`),
  KEY `idx_user_price_adjustment_match` (`user_id`, `enabled`, `model_series`, `model_type`, `billing_type`, `priority`),
  KEY `idx_user_price_adjustment_schedule` (`schedule_type`, `start_time`, `end_time`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='用户专属模型价格调控规则表';
```

唯一性策略：

- 不建议加 `(user_id, model_series, model_type, billing_type)` 唯一索引，因为现有全局规则支持时间窗和优先级，用户规则也应保留“白天/夜间”或“临时活动价”能力。
- 冲突由 `priority ASC, id DESC` 决定，与全局规则保持一致。

级联策略：

- 当前项目大量表未显式使用外键，建议保持同风格，不加外键。
- 删除用户时由 `AuthService.delete_user()` 同步删除该用户的 `UserPriceAdjustmentRule`。

### 3. 规则解析优先级

推荐采用“用户专属规则覆盖全局分类规则”，而不是和全局分类倍率相乘。

解析顺序：

1. 传入用户后，先在 `user_price_adjustment_rule` 中按 `user_id + model_series/model_type/billing_type + enabled + schedule` 匹配。
2. 如果命中当前有效的用户专属规则，使用该用户专属规则倍率。
3. 如果未命中用户专属规则，回退现有全局分类规则 `model_price_adjustment_rule`。
3. 如果全局分类规则也未命中，倍率为 `1`。

原因：

- 用户需求示例是“给张三设置 GPT 系列 1 倍、Claude 2 倍”，语义更像为该用户指定最终系列倍率，而不是在平台全局 GPT x2 基础上再乘用户 GPT x1。
- 管理员在用户页设置“GPT x1”时，直觉上应使张三 GPT 按 1 倍分类倍率计费。
- 若后续需要叠加能力，可增加 `mode=override/multiply` 字段；本次不建议增加，避免配置理解成本过高。

示例：

- 全局规则：GPT 文本 `x2`。
- 用户张三专属规则：GPT 文本 `x1`。
- 张三请求 GPT 文本模型时，分类调控倍率为 `x1`，不是 `x2 * x1`。
- 李四未配置专属规则，请求 GPT 文本模型时继续使用全局 `x2`。

最终文本计费公式：

```text
price_multiplier_snapshot =
  system_config.price_multiplier * resolved_adjustment_multiplier

effective_price_multiplier =
  price_multiplier_snapshot * fast_price_multiplier * context_price_multiplier
```

其中：

```text
resolved_adjustment_multiplier =
  用户专属倍率（优先覆盖） 或 全局分类倍率 或 1
```

媒体积分公式：

```text
charged_credits = base_credits * resolved_adjustment_multiplier
```

媒体积分继续不叠加 `system_config.price_multiplier`。

### 4. PriceAdjustmentService 改造

保留现有全局规则 CRUD，不破坏 `admin/price-adjustments` 现有能力。

新增或调整方法：

- `resolve_user_rule(db, user_id, unified_model, now=None)`
- `resolve_global_rule(db, unified_model, now=None)`：可由现有 `resolve_rule` 改名或保留别名。
- `resolve_adjustment(db, unified_model, user_id=None, now=None)`：返回结构化结果。
- `resolve_multiplier(db, unified_model, user_id=None, now=None)`：保留旧调用兼容，返回倍率。
- `list_user_rules(db, page, page_size, user_id=None, keyword=None, model_series=None, model_type=None, enabled=None)`
- `create_user_rule(db, data, operator_id)`
- `update_user_rule(db, rule_id, data, operator_id)`
- `delete_user_rule(db, rule_id)`
- `list_effective_matrix(db)`：保留全局概览。
- `list_user_effective_matrix(db, user_id)`：返回单用户各系列/类型当前有效倍率。

结构化解析结果建议：

```python
PriceAdjustmentResolution(
    multiplier=Decimal("2.000000"),
    source="user",  # user/global/default
    rule_id=123,
    rule_name="张三 Claude 专属倍率",
    user_id=1001,
)
```

这样计费层可以写入审计快照，前端也可展示命中来源。实现时可使用 `dataclass` 或项目内简单类，避免散落 dict 键名导致 `rule_id/user_rule_id/global_rule_id` 语义重复。

用户规则与全局规则必须共享匹配逻辑：

- `model_series` 精确匹配或 `all`
- `model_type` 精确匹配或 `all`
- `billing_type` 精确匹配或 `all`
- `schedule_type=always` 直接有效
- `schedule_type=daily_time` 按北京时间判断，支持跨天
- 多条命中时按 `priority ASC, id DESC`

### 5. 计费链路改造

所有价格倍率解析点必须传入当前用户 ID。

文本预检：

- 文件：`backend/app/services/proxy_service.py`
- 方法：`_build_text_quota_precheck`
- 调整入参：增加 `user_id: Optional[int] = None` 或直接传 `user: Optional[SysUser] = None`。
- 调用 `PriceAdjustmentService.resolve_multiplier(db, unified_model, user_id=user_id)`。
- 确保 `estimated_total_cost` 与最终扣费同一倍率。

文本实际扣费：

- 文件：`backend/app/services/proxy_service.py`
- 方法：`_deduct_balance_and_log_once`
- 使用 `fresh_user.id` 解析用户专属倍率。

媒体积分：

- 方法：`_apply_media_credit_price_adjustment`
- 调整入参：增加 `user_id: Optional[int] = None`。
- 图片生成、图片编辑、视频生成调用点都传入 `user.id`。
- 媒体积分扣减写入 `image_credit_record` 时同步保存倍率来源快照，至少包含用户/全局/默认来源、命中规则 ID 和最终调控倍率。

异步视频注意点：

- 现有视频任务可能在创建任务与完成扣费之间跨越规则修改。
- 必须在任务创建时保存本次已解析的调整后媒体积分单价，或保存完整 `PriceAdjustmentResolution`。
- 完成扣费时使用任务上下文中的倍率/单价，不再按完成时的当前规则重新解析，避免规则变更导致扣费漂移。
- 若现有路径已经保存调整后的积分成本，则只需确保保存时传入 `user_id`。

### 6. 审计快照设计

建议新增字段，避免继续把所有倍率压扁到 `price_multiplier_snapshot`。

在 `request_log` 与 `consumption_record` 增加：

- `global_price_multiplier_snapshot DECIMAL(12,6) DEFAULT 1`
- `adjustment_price_multiplier_snapshot DECIMAL(12,6) DEFAULT 1`
- `price_adjustment_source_snapshot VARCHAR(20) DEFAULT NULL COMMENT 'user/global/default/legacy'`
- `price_adjustment_rule_id_snapshot BIGINT DEFAULT NULL`

在 `image_credit_record` 增加：

- `adjustment_price_multiplier_snapshot DECIMAL(12,6) DEFAULT 1`
- `price_adjustment_source_snapshot VARCHAR(20) DEFAULT NULL COMMENT 'user/global/default/legacy'`
- `price_adjustment_rule_id_snapshot BIGINT DEFAULT NULL`

兼容策略：

- 保留并继续写 `price_multiplier_snapshot = global_price_multiplier * adjustment_price_multiplier`，不破坏现有前端和报表。
- 新字段用于账单详情、审计和排查。
- 老数据新字段为空时按历史数据处理，前端不强行显示“默认规则”，避免误导。

日志与账单响应兼容：

- `log_service.py` 必须返回新增倍率来源字段，供 `admin/logs` 表格展示。
- `balance_service.py`、`subscription_service.py` 必须做响应字段兼容检查：旧字段展示不能破坏，新字段如展示应统一返回。
- `admin/logs` 新增“用户倍率”列，展示请求快照中的调控倍率和来源，不按当前配置实时重算历史请求。
- 如展示倍率来源，详情区域统一显示“用户专属/全局/默认/历史未记录”。

`admin/logs` 列展示口径：

- 位置：在“响应时间”列之后、“计费”列之前。
- 列名：`用户倍率`。
- 展示值：优先使用 `adjustment_price_multiplier_snapshot`，格式如 `x2`。
- 来源标签：根据 `price_adjustment_source_snapshot` 显示 `用户专属`、`全局`、`默认`、`历史`。
- 旧数据：无新字段时回退展示 `price_multiplier_snapshot`，来源显示 `历史` 或 `-`，避免误认为当前用户规则。
- 文本请求：该列展示分类/用户调控倍率，不包含 Fast 倍率和长上下文倍率；综合倍率仍在计费详情中展示。
- 图片/视频请求：展示媒体积分调控倍率，与 `image_credit_record` 快照保持一致。

### 7. 后端 API 设计

#### 用户页专属倍率接口

建议挂在 `/api/admin/users/{user_id}/price-adjustments`，贴合“通过 admin/users 页面为用户设置”。

接口：

- `GET /api/admin/users/{user_id}/price-adjustments`
  - 查询某用户专属规则列表。
- `GET /api/admin/users/{user_id}/price-adjustments/effective`
  - 查询某用户当前有效倍率矩阵。
- `POST /api/admin/users/{user_id}/price-adjustments`
  - 为用户新增专属规则。
- `PUT /api/admin/users/{user_id}/price-adjustments/{rule_id}`
  - 更新用户专属规则。
- `DELETE /api/admin/users/{user_id}/price-adjustments/{rule_id}`
  - 删除用户专属规则。

权限：

- 全部使用 `require_admin`。
- 创建/更新时校验目标用户存在。
- 更新和删除时必须校验 `rule.user_id == path user_id`，防止通过 URL 修改其他用户规则。
- `POST` 的 body 不接受或忽略 `user_id`，以 path 中的 `user_id` 为准，避免前端传错目标用户。

#### 价格调控页汇总接口

扩展 `/api/admin/price-adjustments`：

- 新增 `GET /api/admin/price-adjustments/user-rules`
  - 分页查看所有用户专属倍率规则。
  - 支持筛选：`keyword`、`user_id`、`model_series`、`model_type`、`enabled`。
  - 返回字段包含：用户 ID、用户名、邮箱、系列、类型、计费类型、倍率、生效方式、状态、当前是否生效。
  - 使用 left join 用户表；用户已删除时保留规则 ID 和 user_id，用户名显示“用户已删除”。

也可以扩展现有 `GET /api/admin/price-adjustments` 加 `scope=global/user`，但更推荐新增 `user-rules` 子接口，避免影响现有全局规则列表。

#### 选项接口

复用现有：

- `GET /api/admin/price-adjustments/options`

用户专属规则使用同一套系列、模型类型、计费类型、生效方式选项。

### 11. 兼容性与边界

默认行为：

- 无用户专属规则时，所有用户继续使用现有全局价格调控规则。
- 无全局规则时，倍率为 1。

禁用规则：

- 用户专属规则禁用后，不参与解析，自动回退全局规则。

时间窗：

- 使用北京时间 `Asia/Shanghai`。
- 支持跨天窗口，例如 `23:00 - 07:00`。
- 与全局规则保持一致。

管理员用户：

- 允许给任意用户角色设置规则，还是仅普通用户？
- 建议第一版允许所有用户 ID，但前端默认从用户列表进入，实际主要覆盖普通用户。

代理用户：

- 管理员可以给代理旗下用户设置专属倍率。
- 代理端不新增管理能力，避免代理绕过平台定价策略。

套餐用户：

- 成本型套餐和余额计费已有不同口径：`estimated_total_cost` 用真实收费口径，`estimated_quota_cost` 用官方额度口径。
- 用户专属倍率应影响真实收费口径。
- 是否影响套餐官方额度口径需保持现有逻辑：全局系统价格倍率不进官方额度，分类调控目前实际也未进入 `official_quota_multiplier`。本次用户专属倍率也不进入官方额度口径，避免破坏套餐额度含义。

## 风险点与处理

### 1. 用户倍率语义不清

风险：用户专属倍率到底是覆盖全局分类倍率，还是与全局分类倍率相乘。

方案：第一版明确为覆盖全局分类倍率；系统全局价格倍率 `system_config.price_multiplier` 仍然叠加。页面提示“用户专属规则优先于全局分类规则”。

### 1.1 用户规则数据模型不一致

风险：用户规则如果缺少 `name` 字段，会与全局规则表单、校验和审计模型不一致。

方案：`user_price_adjustment_rule` 必须包含 `name`，前端新增规则时可自动生成默认名称，管理员可修改。

### 2. 预检与实际扣费不一致

风险：只改实际扣费不改预检，会导致余额检查放行或拦截错误。

方案：统一通过 `PriceAdjustmentService.resolve_adjustment` 解析，预检和实际扣费都传入同一个用户 ID。

### 3. 媒体积分漏传用户

风险：文本计费正确，但图片/视频仍用全局规则。

方案：改造 `_apply_media_credit_price_adjustment` 签名，并逐一检查图片生成、图片编辑、视频生成调用点；`image_credit_record` 同步保存倍率来源快照。

### 4. 历史账单不可审计

风险：只保存最终 `price_multiplier_snapshot`，无法确认当时是否命中用户规则。

方案：新增倍率来源快照字段，同时保留旧字段兼容展示；历史数据来源为空时展示为“历史未记录”或不展示来源，不误标为默认规则。

### 5. 删除用户后残留规则

风险：删除用户时规则残留，价格调控页出现孤儿数据。

方案：`AuthService.delete_user()` 同步删除用户专属规则；列表 join 用户时也可兼容已删除用户显示 `用户已删除`。

### 6. 前端重复实现过多

风险：用户页和价格调控页都写一套规则表单，后续维护成本高。

方案：优先抽取用户专属倍率抽屉或规则表单组件；若时间紧，先保证功能闭环，再在实现记录中标记可抽取项。

### 7. 用户规则越权修改

风险：`PUT /api/admin/users/{user_id}/price-adjustments/{rule_id}` 如果只按 `rule_id` 更新，可能通过 URL 修改其他用户规则。

方案：更新和删除必须校验 `rule.user_id == path user_id`；创建时以 path `user_id` 为唯一可信目标用户。

### 8. 异步视频倍率漂移

风险：视频任务创建后、完成扣费前，管理员修改用户倍率，导致完成时重新解析出的倍率与创建时不一致。

方案：任务创建时冻结调整后积分单价或完整倍率解析结果，完成扣费使用冻结值。

### 9. 日志倍率展示误用实时规则

风险：`admin/logs` 如果按当前规则实时计算“用户倍率”，历史请求会随着后台配置变化而变化，导致账单解释不可信。

方案：`admin/logs` 只展示请求落库时的倍率快照；无快照的老数据显示为“历史”或 `-`，不按当前用户规则重算。

## 验收标准

- 管理员可在 `admin/users` 为指定用户新增 GPT/Claude 等系列倍率。
- 管理员可为同一个用户分别设置不同模型系列倍率，例如张三 GPT `x1`、Claude `x2`。
- 用户专属倍率启用时，该用户请求对应系列模型按专属倍率计费。
- 未配置用户专属倍率的用户仍按 `admin/price-adjustments` 的全局规则计费。
- 用户专属规则禁用或删除后自动回退全局规则。
- `admin/price-adjustments` 能看到所有用户专属规则，至少包含用户、模型系列、模型类型、计费类型、倍率、状态。
- `admin/logs` 在“响应时间”和“计费”之间展示“用户倍率”列，显示本次请求实际使用的调控倍率和来源。
- 文本 token、按请求计费、图片积分、视频积分均使用一致的用户专属规则解析。
- 请求日志、消费记录和媒体积分记录保留兼容的既有字段，并新增可审计的倍率来源快照。
