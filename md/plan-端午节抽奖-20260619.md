# 端午节抽奖页面实施方案 v2

## Plan 评估与 v2 修订说明

`codex exec` 评估结论：修订后通过。

已根据评估意见补充：

- 明确接口返回结构。
- 明确用户端使用 `get_current_user`，管理端使用 `require_admin`。
- 增加并发抽奖保护：数据库层 `prize_rank` 唯一约束，服务层在事务中使用 `SELECT ... FOR UPDATE` 锁定报名记录，锁定后再次检查是否已有中奖结果，避免两个管理员同时抽奖导致名单变化或覆盖。

## 用户原始需求

当前系统新增端午节抽奖活动：

- 用户端新增报名页，用户点击报名时检测资格，符合后记录报名。
- 参与用户需要满足以下任一条件：
  - 平台开通过日卡及以上套餐。
  - 充值或消费额度大于 100 美元。
  - 充值包含管理端、代理端直接为用户添加额度，以及用户通过在线充值页使用支付宝或微信充值。
- 报名时间：2026-06-19 00:00:00 至 2026-06-21 20:00:00。
- 2026-06-21 23:00:00 抽奖。
- 奖励：
  - 第一名：300 美元。
  - 第二名：200 美元。
  - 第三名：100 美元。
  - 第 4-10 名：50 美元。
- 管理端新增页面：
  - 查看所有报名用户。
  - 报名结束后可点击抽奖，从所有报名者中抽出前 10 名并显示名单。
  - 管理员后续手动逐个给获奖用户充值。

## 技术方案设计

### 活动时间口径

- 统一使用北京时间 `Asia/Shanghai`。
- 报名开始：`2026-06-19 00:00:00`。
- 报名结束：`2026-06-21 20:00:00`。
- 抽奖最早时间：`2026-06-21 23:00:00`。
- 后端用无时区 `datetime` 保存北京时间，与现有套餐服务 `SubscriptionService.get_current_time()` 的本地时间口径保持一致。

### 数据库设计

新增表 `dragon_boat_lottery_entry`：

- `id`：主键。
- `user_id`：报名用户 ID，唯一索引，保证每个用户只能报名一次。
- `username`、`email`：报名时用户信息快照，便于管理端直接查看。
- `agent_id`：用户所属代理，便于后续排查来源。
- `qualification_type`：资格来源，取值 `subscription` / `recharge` / `consume`。
- `qualification_detail`：资格说明文本。
- `total_recharged_snapshot`：报名时累计充值美元快照。
- `total_consumed_snapshot`：报名时累计消费美元快照。
- `subscription_id_snapshot`：命中的套餐记录 ID。
- `status`：`registered` / `winner`。
- `prize_rank`：中奖名次，未中奖为空。
- `prize_amount`：奖金美元，未中奖为 0。
- `drawn_by_user_id`：执行抽奖的管理员用户 ID。
- `drawn_at`：开奖时间。
- `created_at`、`updated_at`。

索引：

- `uk_dragon_boat_lottery_user`：`user_id` 唯一。
- `idx_dragon_boat_lottery_status`：`status`。
- `idx_dragon_boat_lottery_prize_rank`：`prize_rank`。
- `idx_dragon_boat_lottery_created_at`：`created_at`。
- `uk_dragon_boat_lottery_prize_rank`：`prize_rank` 唯一，MySQL 允许多个 NULL，中奖名次非空后不可重复。

设计原因：

- 报名记录和中奖结果放在同一张表，管理端列表简单。
- 中奖结果持久化，避免每次刷新重新随机。
- 通过用户唯一索引和事务处理保证重复点击报名不会产生重复记录。

### 资格判定

用户满足任一条件即可报名：

1. 套餐资格：
   - 查询 `user_subscription` 中该用户历史套餐记录。
   - “开通过”按历史记录处理，不要求当前仍有效。
   - 套餐天数标准：`duration_days_snapshot >= 1`，若快照为空则关联 `subscription_plan.duration_days >= 1`。
   - “日卡及以上套餐”按当前系统内所有 `duration_days >= 1` 的套餐记录处理。
2. 充值资格：
   - 查询 `user_balance.total_recharged > 100`。
   - 当前系统管理端手动充值、在线支付宝/微信充值都会进入 `UserBalance.total_recharged`。
   - 代理端如已通过同一余额服务给用户加额度，也会进入同一字段。
3. 消费资格：
   - 按真实模型请求消费统计 `consumption_record.total_cost`，要求 `request_id` 和 `model_name` 非空且 `total_cost > 0`，累计结果大于 100。
   - 不把管理端扣款、代理回收余额等人工资金调整计入消费资格。

金额判断按需求使用“大于 100 美元”，即 `> 100`，不是 `>= 100`。

### 抽奖规则

- 只允许北京时间 `2026-06-21 23:00:00` 后执行。
- 抽奖前必须存在报名用户。
- 若已经存在中奖结果：
  - 不重新抽取，直接返回已有中奖名单。
  - 避免误点导致名单变化。
- 并发保护：
  - 抽奖服务在同一事务内用 `SELECT ... FOR UPDATE` 锁定所有报名记录。
  - 获取锁后再次查询中奖结果；若已有结果，直接返回。
  - `prize_rank` 增加唯一约束作为数据库兜底。
- 抽奖候选：`dragon_boat_lottery_entry` 全部报名记录。
- 若报名人数不足 10 人，则抽出全部报名人数。
- 使用 Python `random.SystemRandom().sample()` 进行无放回随机抽取。
- 奖励金额按名次映射：
  - 1 -> 300
  - 2 -> 200
  - 3 -> 100
  - 4-10 -> 50
- 抽奖只记录名单和奖金，不自动充值。

### 权限依赖

- 用户端接口统一使用 `get_current_user`，只允许登录用户查看个人活动状态和报名。
- 管理端接口统一使用 `require_admin`，只允许平台管理员查看报名名单和执行抽奖。
- Controller 层不做 try-catch，业务异常统一抛出 `ServiceException`，由全局异常处理器返回。

## 影响范围分析

### 业务影响

- 新增活动报名/抽奖能力，不改变现有充值、套餐、消费逻辑。
- 抽奖不会自动给用户加余额，避免误发奖；管理员仍需手动充值。
- 用户端只允许登录用户报名，管理端仅平台管理员可查看和抽奖。

### 数据一致性风险

- 重复报名风险：通过 `user_id` 唯一索引和服务层幂等处理规避。
- 重复抽奖风险：服务层先查已有中奖结果，存在则直接返回。
- 报名资格快照风险：报名后用户数据可能变化，但活动资格按报名时快照记录，便于追溯。

### 时间风险

- 活动时间已经是 2026-06-19，当前开发环境日期同日。需要确保后端按北京时间判断。
- 管理端抽奖按钮前端会根据后端 `can_draw` 展示，但最终以服务端校验为准。

### 随机性风险

- 抽奖使用 `SystemRandom`，不暴露随机种子。
- 抽奖结果写入数据库，后续刷新不重新随机。
