# 缓存命中优化方案

## 用户原始需求

基于 `缓存研究/` 中 CLIProxyAPI、new-api、sub2api 三个反代项目的缓存处理方式，判断是否能优化当前系统的上游缓存命中率。若有把握，进行优化、提交 GitHub、更新服务器并监控日志。提交时不包含 `缓存研究/` 目录。

## 技术方案设计

### 目标

- 不做通用 LLM 响应本地缓存，避免错误复用不同用户、不同上下文的生成结果。
- 优化 Anthropic Prompt Cache 的 cache_control 断点，使静态内容和历史内容更稳定地命中上游缓存。
- 增加会话级渠道亲和：同一用户、同一模型、同一会话尽量复用上一次成功的渠道，降低首尔/zz1 等不同上游缓存池切换导致的缓存失效。

### 核心设计

1. 新增 `ChannelAffinityService`
   - 从请求中提取稳定会话 key：`prompt_cache_key`、`metadata.user_id`、`session_id`、`conversation_id`、`X-Codex-Turn-Metadata`。
   - 缺少显式 key 时，按模型、用户、代理、system/tools/首个 user 内容生成 fallback key。
   - 使用进程内 TTL 映射保存 `affinity_key -> channel_id`。
   - 渠道可用时将命中的 channel 排到候选列表前面；成功请求后更新绑定。

2. 优化 `AnthropicPromptCacheService`
   - history breakpoint 优先打在倒数第二个 user 消息，避免最新用户输入变化导致缓存前缀不稳定。
   - 注入后强制限制 cache_control 数量不超过 Anthropic 的 4 个断点。
   - 规范 1h/5m TTL 顺序：一旦较早位置出现 5m，后续 1h 降级为 5m，避免上游拒绝非法 TTL 顺序。

3. 新增系统配置
   - `channel_affinity_enabled`
   - `channel_affinity_ttl_seconds`
   - `channel_affinity_fallback_enabled`

## 实施步骤概要

1. 新增渠道亲和服务，完成 key 提取、候选排序、成功绑定、TTL 清理。
2. 在 OpenAI、Anthropic、Responses 文本入口接入渠道亲和排序。
3. 在文本请求成功记账入口记录成功渠道绑定。
4. 优化 Anthropic cache_control 断点选择、数量和 TTL 顺序。
5. 补充 SQL 默认配置。
6. 补充单元测试并运行验证。
7. 创建实施记录与自审文档，提交时排除 `缓存研究/` 和无关文件。
