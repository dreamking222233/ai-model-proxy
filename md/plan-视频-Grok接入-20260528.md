## 用户原始需求

深度阅读当前目录下项目及相关文档，结合 grok2api 视频生成接口文档，为当前系统新增 Grok 视频生成模型接口开发。

目标能力：

- 接入 `grok-imagine-video`
- 用户可通过 `/v1/created/video` 生成视频
- 支持文生视频与图生视频，图生视频最多 7 张参考图
- 支持异步生成、任务查询和视频内容获取
- `/v1/created/video` 需要参考实测脚本自动创建任务并查询进度，直到视频完成后再返回用户
- 视频按 1 秒 0.5 积分计费，而不是 10 秒 0.5 积分
- `/v1/chat/completions` 支持视频模型文生视频，流式返回视频 URL
- `user/chat` 页面支持 Grok 视频模型，提供文生视频、图生视频参数配置与结果展示
- `user/quickstart` 页面同步补充视频生成模型 API 调用方式和支持参数
- 结合当前系统架构完成渠道路由、计费、日志与文档

## 技术方案设计

当前系统已有 OpenAI/Gemini 图片接口、统一模型、渠道映射、图片积分与请求日志能力。视频生成与图片生成同属媒体类请求，推荐复用现有 `image_credit` 余额体系作为媒体积分计费，不新增余额表，使用 `request_type=video_generation` 区分日志。

上游 grok2api 已确认支持：

- `POST /v1/chat/completions`
- `POST /v1/videos`
- `GET /v1/videos/{video_id}`
- `GET /v1/videos/{video_id}/content`

本次新增当前系统对外接口：

- `POST /v1/videos`
- `POST /v1/created/video`
- `POST /v1/chat/completions`
- `GET /v1/videos/{video_id}`
- `GET /v1/videos/{video_id}/content`

核心实现：

1. 新增视频代理路由，创建接口解析 `multipart/form-data`，支持 `input_reference[]` 与兼容字段 `input_reference`。
2. 在 `ProxyService` 增加视频请求处理：
   - 校验 `model`、`prompt`、`seconds`、`size`、`resolution_name`、`preset`
   - 校验模型类型为 `video`
   - 支持 `billing_type=image_credit/free`
   - 按 `image_credit_multiplier * seconds` 作为视频生成积分成本，其中 `image_credit_multiplier` 表示每秒单价
   - 通过模型渠道映射选择上游 grok2api 渠道
   - 使用 multipart 转发到上游 `/v1/videos`
   - 上游成功创建任务后扣减积分并记录请求日志
3. `/v1/created/video` 在创建成功后复用任务路由轮询上游状态，完成后返回 `final_status` 和本地 `content_url`；`/v1/videos` 保持异步任务创建语义。
4. 保存 `video_id -> channel/model/user` 的短期内存映射，查询和下载优先使用创建时渠道，避免多渠道环境查询错上游。
5. 查询、等待轮询和下载接口不重复计费，仅鉴权后代理到上游。
6. 数据库初始化和升级脚本支持 `unified_model.model_type='video'`。
7. 管理端模型类型增加“视频”，计费文案从“图片积分”调整为“媒体积分/图片积分兼容”。
8. `handle_openai_request` 识别 `model_type=video` 后进入视频 Chat Completions 链路，解析 `video_config` 并转发上游 `/v1/chat/completions`。
9. `user/chat` 复用聊天页会话体系，视频模型显示视频参数工具栏；文生视频走流式 Chat Completions，图生视频走 `/v1/created/video` multipart，同步等待完成后展示 `<video>`。
10. 用户快速开始文档补充视频 API 示例。
