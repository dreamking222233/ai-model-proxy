> Plan v2：已根据 `md/plan-review-视频-zz1cc接入-20260628.md` 修订。修订重点为 chat 兼容入口策略、秒数校验、任务 ID 归一、错误兼容、SQL 策略和参考图 MIME 限制。

## 用户原始需求

阅读 `https://zz1cc.cc.cd/docs#video` 文档，了解该平台 video 调用方式，并结合当前系统已有 CPA Grok 视频生成接入能力，为当前系统接入 zz1cc 平台的 `video-ds-2.0` 和 `video-ds-2.0-fast` 出具实施方案。

当前系统在新建渠道时，协议类型可选择 `OpenAI`，OpenAI 渠道类型已有 `CPA Grok 视频`。本次需要扩展当前系统的视频生成能力，接入 zz1cc video 模型。

## 外部文档阅读结论

zz1cc 文档站是前端渲染页面，直接 HTML 中没有文档正文。已从其 docs 页面异步 JS chunk 中提取 `#video` 章节内容，结论如下：

- 鉴权：`Authorization: Bearer <API_KEY>`。
- 创建视频任务：`POST /v1/videos`。
- 查询视频任务：`GET /v1/videos/{task_id}`。
- 下载/播放视频内容：`GET /v1/videos/{task_id}/content`。
- 创建任务请求体为 JSON，而不是 multipart。
- 文档示例模型为 `video-ds-2.0-fast`。
- 用户要求同时接入 `video-ds-2.0` 和 `video-ds-2.0-fast`，其中 `video-ds-2.0` 未在提取到的参数表中单独列出差异，首版按同一接口形态接入，联调时确认普通版是否支持同样参数。

### zz1cc video 支持参数

| 参数 | 类型 | 文档说明 | 接入建议 |
| --- | --- | --- | --- |
| `model` | string | 必填；fast 通道使用 `video-ds-2.0-fast` | 由模型映射 `actual_model_name` 透传，支持 `video-ds-2.0`、`video-ds-2.0-fast` |
| `prompt` | string | 必填，描述视频内容 | 沿用当前系统 `prompt` |
| `seconds` | integer | 可选；当前 fast 视频流程主要按 15 秒设计 | 首版默认 15，允许 15；是否允许其他秒数需联调验证 |
| `aspect_ratio` | enum | 可选；常用 `9:16`、`16:9`、`1:1` | 当前系统用 `size`，需转换为 `aspect_ratio` |
| `images` | array | 可选；参考图片 URL 或 base64，数量以上游限制为准 | 当前上传文件转 data URL 后放入 `images` |
| `videos` | array | 可选；参考视频 URL，数量以上游限制为准 | 当前接口暂未接收参考视频，预留内部字段 |
| `audios` | array | 可选；参考音频 URL，通常 1 个音频文件 | 当前接口暂未接收音频，预留内部字段 |

### 文档示例形态

```json
{
  "model": "video-ds-2.0-fast",
  "prompt": "A cinematic 9:16 video of a cat running through warm sunlight",
  "seconds": 15,
  "aspect_ratio": "9:16"
}
```

参考素材示例：

```json
{
  "model": "video-ds-2.0-fast",
  "prompt": "Use the reference image style and create a smooth product video",
  "seconds": 15,
  "aspect_ratio": "9:16",
  "images": ["https://example.com/input.png"],
  "videos": ["https://example.com/input.mp4"],
  "audios": ["https://example.com/input.mp3"]
}
```

## 当前系统现状

### 已有视频入口

- `backend/app/api/proxy/video_proxy.py`
  - `POST /v1/videos`
  - `POST /v1/created/video`
  - `GET /v1/videos/{video_id}`
  - `GET /v1/videos/{video_id}/content`
- `backend/app/services/proxy_service.py`
  - 普通 OpenAI 视频渠道：向上游 multipart `POST /v1/videos`
  - CPA Grok 视频渠道：`provider_variant=cpa-grok-video`，向上游 JSON `POST /v1/videos/generations`
  - 查询、下载、异步任务路由、完成后计费、视频 URL 内容校验都已具备

### 已有 CPA Grok 特殊分支

当前 CPA Grok 的关键差异：

- 渠道仍使用 `protocol_type=openai`。
- 通过 `provider_variant=cpa-grok-video` 识别。
- 创建请求 JSON 字段为 `duration`、`size`、`resolution`、`image`、`reference_images`。
- 查询走 `/v1/videos/{id}`。
- 完成后从上游 `video.url` 获取真实 mp4 URL，再代理下载。

zz1cc 与 CPA Grok 的差异：

- zz1cc 创建路径是 `/v1/videos`，不是 `/v1/videos/generations`。
- zz1cc 请求体是 JSON，但字段为 `seconds`、`aspect_ratio`、`images`、`videos`、`audios`。
- zz1cc 文档说明 content 直接走 `/v1/videos/{task_id}/content`，不需要先解析 `video.url`。

## 技术方案设计

### 渠道类型设计

新增 OpenAI 渠道类型：

- `provider_variant = "zz1cc-video"`
- 管理端展示名：`zz1cc 视频`
- 协议类型仍为 `openai`
- `base_url` 填 `https://zz1cc.cc.cd` 或 `https://zz1cc.cc.cd/v1`
- `api_key` 填 zz1cc token
- 默认关闭健康监控，原因是视频生成长耗时且不适合通用健康检查

不建议复用 `cpa-grok-video`：

- CPA Grok 已绑定 Grok 专用字段和路径。
- zz1cc 是 OpenAI compatible video 任务接口，但请求体是 JSON，不是现有普通 OpenAI 视频的 multipart。
- 单独 variant 可以降低后续新增 `videos/audios`、状态标准化差异时的回归风险。

### 模型设计

新增或确保存在两个统一视频模型：

- `video-ds-2.0`
- `video-ds-2.0-fast`

建议属性：

- `model_type = "video"`
- `model_series = "other"`
- `protocol_type = "openai"`
- `billing_type = "image_credit"`
- `image_credit_multiplier` 按实际成本配置，默认可先与现有视频模型一致或由运营配置
- `enabled = 1`

模型映射：

- 统一模型 `video-ds-2.0` 映射到 zz1cc 渠道，`actual_model_name = "video-ds-2.0"`
- 统一模型 `video-ds-2.0-fast` 映射到 zz1cc 渠道，`actual_model_name = "video-ds-2.0-fast"`

### 参数转换设计

当前用户侧视频接口保留不变，仍接收：

- `model`
- `prompt`
- `seconds`
- `size`
- `resolution_name`
- `preset`
- `input_reference[]`

zz1cc 分支内部转换：

| 当前系统字段 | zz1cc 字段 | 说明 |
| --- | --- | --- |
| `model` / mapping | `model` | 使用 `actual_model_name` |
| `prompt` | `prompt` | 直接透传 |
| `seconds` | `seconds` | 首版按 15 秒能力处理 |
| `size` | `aspect_ratio` | `720x1280`、`1024x1792` -> `9:16`；`1280x720`、`1792x1024`、CPA 横屏尺寸 -> `16:9`；`1024x1024` -> `1:1` |
| `input_reference[]` | `images` | 文件转 data URL 后数组传给上游 |
| 预留 `videos` | `videos` | 首版 API 不暴露，后续可扩展 JSON 或 multipart 字段 |
| 预留 `audios` | `audios` | 首版 API 不暴露，后续可扩展 JSON 或 multipart 字段 |

`resolution_name`、`preset` 当前 zz1cc 文档未声明支持，首版不向上游传递，避免上游 400。

### Plan v2 关键约束

- `/v1/chat/completions` 视频兼容入口首版不支持 `zz1cc-video` 渠道；命中该 variant 时返回 `VIDEO_MODEL_NOT_SUPPORTED`，避免误走普通 OpenAI 视频 chat 分支。
- `seconds` 规则明确为：缺省或空值默认 `15`；仅允许 `15`；非 `15` 返回 `INVALID_VIDEO_SECONDS`，不做静默截断。
- 创建响应任务 ID 统一规则：`id = body.id || body.task_id || body.request_id`。本地 `_store_video_task_route()`、对外 `id`、后续 retrieve/content 都使用归一后的 `id`。
- 创建失败：非 2xx 统一抛 `OPENAI_VIDEO_GENERATION_FAILED`，保留上游 HTTP 状态和最多 1000 字符 body 到 upstream detail。
- 查询失败：如果 400/422 JSON body 含 `error`、`code`、`message` 或 `status`，标准化为 `failed` 任务响应；网络错误或 5xx 抛 `OPENAI_VIDEO_RETRIEVE_FAILED`。
- content 校验：zz1cc 直接校验 `/v1/videos/{id}/content`；JSON/text/空内容按现有视频内容校验逻辑视为失败。
- 参考图：只允许 `image/*` MIME；非图片上传返回 `INVALID_VIDEO_REFERENCE`。首版沿用当前系统最多 7 张接收上限，但文档标记为需联调确认的上游限制。
- SQL 策略：升级脚本只 upsert 两个统一模型并更新 variant 注释/健康检查，不自动绑定未知渠道；映射通过注释提供示例，管理员创建 `zz1cc-video` 渠道后手动绑定。
- 默认媒体积分倍率暂定 `0.500` 积分/秒，运营上线后可在模型管理中调整。

### 响应与任务状态设计

沿用当前系统对外响应结构：

- 创建后返回标准任务 JSON，并补充 `request_id`、`usage`。
- `POST /v1/created/video` 仍自动轮询直到完成，再返回 `final_status`、`content_url`、`retrieve_url`。
- `GET /v1/videos/{video_id}` 代理 zz1cc 状态查询，并做状态标准化。
- `GET /v1/videos/{video_id}/content` 代理 zz1cc `/v1/videos/{id}/content`。

状态标准化建议：

- `completed`、`succeeded`、`success`、`done` -> `completed`
- `pending`、`queued` -> `queued`
- `in_progress`、`processing`、`running` -> `in_progress`
- `failed`、`error`、`cancelled`、`canceled`、`expired` -> `failed`

### 计费设计

沿用当前视频计费策略：

- 创建阶段不扣费。
- 任务完成后，先校验 `/content` 返回有效视频内容，再扣媒体积分。
- 扣费数量为 `image_credit_multiplier * 实际秒数`。
- `request_type = "video_generation"`。
- 查询或下载触发完成计费时，通过现有 `video_task_billing_snapshot` / 内存路由避免重复扣费。

由于 zz1cc 文档显示 fast 流程主要按 15 秒设计：

- `video-ds-2.0-fast` 首版能力建议只展示 `15` 秒。
- `video-ds-2.0` 是否支持非 15 秒需联调确认；首版也先按 `15` 秒处理，避免前端允许用户选到上游不支持的秒数。

## 实施步骤概要

1. 在后端渠道服务中新增 `zz1cc-video` provider variant，并在管理端渠道表单中暴露该选项。
2. 在模型服务中增加 `video-ds-2.0`、`video-ds-2.0-fast` 的视频能力：尺寸以 `720x1280`、`1280x720`、`1024x1024` 为主，秒数首版只开放 `15`。
3. 在 `ProxyService` 中新增 `_is_zz1cc_video_channel()`、URL 解析方法、`size -> aspect_ratio` 转换方法。
4. 新增 `_non_stream_zz1cc_video_request()`：将当前系统 multipart 入参转换为 zz1cc JSON，请求 `/v1/videos`。
5. 在 `handle_video_request()` 中按渠道分支：CPA Grok -> CPA 方法；zz1cc -> zz1cc 方法；其他 OpenAI -> 原 multipart 方法。
6. 扩展查询和内容代理：zz1cc 查询走 `/v1/videos/{id}`，内容走 `/v1/videos/{id}/content`。
7. 保持当前完成后计费逻辑，只在内容校验成功后扣费。
8. 新增 SQL 升级脚本，插入两个统一视频模型，并给渠道映射提供示例或说明。
9. 更新用户端媒体工作台模型能力文案，确保选择 `video-ds-2.0-fast` 时默认参数符合文档。
10. 补充 `docs/zz1cc-video-integration.md`，记录上游参数、当前系统对外参数、计费和限制。
11. 增加后端单元测试，覆盖参数转换、秒数校验、状态标准化、content 代理 URL。
12. 联调真实 zz1cc API，确认 `video-ds-2.0` 是否与 fast 参数一致、返回 ID 字段名、状态枚举、content 响应类型、参考素材数量限制。

## 风险点与待确认项

- 文档参数表只明确写了 `video-ds-2.0-fast`，普通 `video-ds-2.0` 的具体时长和素材限制需要真实联调确认。
- 文档未明确任务创建响应中的任务 ID 字段名，需兼容 `id`、`task_id`、`request_id`。
- 文档未明确状态枚举，需做宽松标准化。
- 文档说明 `images/videos/audios` 支持 URL 或 base64，但当前系统只接收图片上传；视频/音频参考素材需要后续扩展用户侧接口。
- 当前系统现有普通 OpenAI 视频分支使用 multipart，zz1cc 使用 JSON，因此必须独立 variant，不能直接走默认 OpenAI 视频分支。
