# 计费-上下文与Fast倍率 Plan v2

## v2 修订说明

- 根据 Plan Review 修正套餐 `quota_cost` 口径：长上下文倍率会进入套餐官方成本额度，但系统 `price_multiplier` 仍不进入 `quota_cost`。
- 明确本方案中的“上下文”是“计费上下文 token”，口径为输入 + 输出 + 缓存读取，不等同于模型请求窗口。
- 明确同步更新全部初始化 SQL：`backend/sql/init.sql`、`backend/sql/initData.sql`、`sql/initData.sql`。
- 统一由后端返回 `effective_price_multiplier_snapshot`，前端也可按 `price * fast * context` 兜底计算。

## 用户原始需求

当前系统接收用户请求后，需要新增计费处理：

- 请求上下文超过 256k 后，收费按 2 倍。
- 如果开启 fast 模式，则 fast 按 2 倍。
- 如果同时开启 fast 且上下文超过 256k，则总倍率为 4 倍。
- 前台页面需要展示具体扣费详情：
  - `admin/logs`
  - `admin/agent-logs`
  - `agent/logs`
  - `user/balance`
- 展示说明需体现“超 256k 是 2 倍、开启 fast 是 2 倍、同时开启是 4 倍”。

## 当前系统分析

### 请求与计费主链路

- 文本请求最终通过 `backend/app/services/proxy_service.py::_finalize_successful_text_request` 进入本地计费。
- 真实扣费集中在 `ProxyService._deduct_balance_and_log_once`：
  - 读取上游 `input_tokens`、`output_tokens`。
  - 从 `RequestCacheSummaryService.build_request_log_fields(cache_info)` 读取上游缓存读取 token。
  - 当前 `raw_total_tokens = raw_input_tokens + raw_output_tokens + raw_cache_read_input_tokens`。
  - 当前 `total_tokens = input_tokens + output_tokens + cache_read_input_tokens`，其中三者会乘系统 `token_multiplier`。
  - 当前价格倍率为 `price_multiplier * fast_price_multiplier`。
  - 当前缓存读取按输入单价 `0.1` 计费。
- fast 模式已有：
  - `service_tier=priority` 时 `_build_text_billing_context` 标记 `fast_price_multiplier=2`。
  - `request_log` 和 `consumption_record` 已有 `fast_price_multiplier_snapshot`。
  - 前端已有 fast 标签和部分计费过程展示。

### 日志与展示

- `LogService.list_request_logs` 给 `admin/logs`、`admin/agent-logs`、`agent/logs` 返回请求日志。
- `BalanceService.get_consumption_records` 和 `user/profile/usage-logs` 给用户余额/使用页面返回账单数据。
- 前端相关文件：
  - `frontend/src/views/admin/RequestLog.vue`
  - `frontend/src/views/admin/AgentRequestLog.vue`
  - `frontend/src/views/agent/RequestLog.vue`
  - `frontend/src/views/user/BalanceLog.vue`
- 这些页面已经存在计费 tooltip / 展开行 / 详情抽屉，但只展示基础倍率与 fast 倍率，缺少长上下文倍率及组合倍率解释。

## 技术方案设计

### 1. 上下文阈值口径

定义计费上下文 token：

```text
context_tokens = raw_input_tokens + raw_output_tokens + raw_cache_read_input_tokens
```

理由：

- 用户示例中的“入 1,498 + 出 1,163 + 缓存读取 61,824 = 合计 64,485”正是该口径。
- 该口径是计费上下文 token，不是模型请求窗口 token；因此可能出现请求前预估未超阈值、请求完成后因输出或缓存读取超过阈值而按长上下文计费的情况。
- 使用 raw 上游用量，不受系统 `token_multiplier` 影响，避免展示或套餐 token 倍率改变后误触发长上下文收费。
- 缓存创建 token 当前“不额外计费”，不纳入本次需求中的“输入输出缓存读取”口径。

### 2. 新增长上下文倍率

新增常量：

- `LONG_CONTEXT_THRESHOLD_TOKENS = 262144`
- `LONG_CONTEXT_PRICE_MULTIPLIER = 2`
- 默认倍率 `1`

当 `context_tokens > 262144` 时：

- `context_price_multiplier_snapshot = 2`
- 否则为 `1`

最终文本请求价格倍率：

```text
effective_price_multiplier =
  price_multiplier_snapshot
  * fast_price_multiplier_snapshot
  * context_price_multiplier_snapshot
```

示例：

- 普通且未超 256k：`1`
- 仅 fast：`2`
- 仅超 256k：`2`
- fast 且超 256k：`4`

### 3. 数据库存储

在 `request_log` 和 `consumption_record` 新增字段：

- `context_tokens_snapshot INT DEFAULT 0`
- `context_token_threshold_snapshot INT DEFAULT 262144`
- `context_price_multiplier_snapshot DECIMAL(12, 6) DEFAULT 1`

用途：

- 保留历史请求当时判断依据。
- 支持前端展示具体原因，不依赖未来配置变化。
- 支持后续审计和账单回溯。

需要新增升级脚本：

- `backend/sql/upgrade_long_context_billing_20260522.sql`

同步更新：

- `backend/app/models/log.py`
- 同步更新完整初始化结构文件：
  - `backend/sql/init.sql`
  - `backend/sql/initData.sql`
  - `sql/initData.sql`

### 4. 后端计费改造

修改 `ProxyService._deduct_balance_and_log_once`：

- 根据 raw token 计算 `context_tokens_snapshot`。
- 计算 `context_price_multiplier_snapshot`。
- 将其纳入 `effective_price_multiplier_decimal`。
- `input_cost`、`output_cost`、`cache_read_cost` 均使用新总倍率。
- `total_cost` 真实扣费同步变更。
- 套餐 `quota_cost` 同步纳入 fast 与长上下文倍率，但继续不纳入系统 `price_multiplier`，保持现有“官方成本额度”语义。
- 写入 `ConsumptionRecord` 与 `RequestLog` 新字段。
- `UserApiKey.total_cost` 使用变更后的 `total_cost`。

修改 `_build_text_quota_precheck`：

- 使用请求前估算输入/输出 token 计算预估上下文 token。
- 预检查阶段没有真实缓存读取 token，只能按估算输入 + 估算输出判断，作为下限保护。
- 如果请求体估算已超过 256k，则预检查成本也按长上下文 2 倍计算。
- `estimated_total_cost` 使用 `price_multiplier * fast * context`，用于余额预检查。
- `estimated_quota_cost` 使用 `fast * context`，不乘系统 `price_multiplier`，用于官方成本口径套餐预检查。
- 返回 `context_*_snapshot` 便于后续调试和测试。

说明：

- 实际扣费以请求完成后的真实上游 token 为准。
- 预检查只是防止明显余额/套餐不足，不承诺完全等于最终扣费。

### 5. 后端日志 API 改造

在以下序列化输出补充字段：

- `context_tokens_snapshot`

### 6. 前端展示改造

## 影响范围分析

### 影响模块

- 后端真实计费：`proxy_service.py`
- 后端日志查询：`log_service.py`、`balance_service.py`、`subscription_service.py`
- ORM：`models/log.py`
- DB 迁移：`backend/sql/upgrade_long_context_billing_20260522.sql`
- 前端页面：
  - 管理端请求日志
  - 管理端代理请求日志
  - 代理端请求日志
  - 用户余额/账单页面

### 风险点

- 计费倍率重复叠加：需要保证 long context 是独立快照，只在 effective multiplier 中乘一次。
- 预检查与实际扣费不完全一致：请求前无真实缓存读取 token，且请求可能没有输出上限，只能估算。实际扣费以请求后真实 usage 为准。
- 历史数据兼容：历史记录没有新字段时前端必须默认 `context_price_multiplier_snapshot=1`。
- DB 未迁移时后端写入会失败：必须提供升级脚本，并在部署说明中要求先执行迁移。
- 前端 4 个页面存在重复逻辑，短期按现有风格局部同步，后续可抽公共 mixin/组件。
