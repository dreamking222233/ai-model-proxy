# 计费-套餐预检放行 Plan

## 用户原始需求

用户套餐（月卡、周卡、日卡）当日额度仍有剩余时，对话请求不应因为预计扣费超出而直接返回 402。希望改为：先允许请求发送到上游，请求完成后按真实消耗扣套餐额度；如果本次请求把额度扣成负数，则展示负数，下一次请求再拒绝。

## 技术方案设计

当前套餐前置校验在存在 `quota_precheck` 时，会用预估成本或预估 Token 与当日剩余额度比较。该逻辑会导致“额度仍有剩余，但预估本次请求超过剩余额度”时提前拦截。

调整为：

- 套餐成本额度：请求前只判断当日剩余额度是否大于 0；大于 0 即放行。
- 套餐 Token 额度：请求前只判断当日剩余 Token 是否大于 0；大于 0 即放行。
- 请求完成后继续复用现有 `consume_quota_after_request` 真实记账，允许 `used_amount` 超过 `quota_limit`。
- 如果下一次请求时 `remaining_amount <= 0`，继续走现有套餐不足处理；余额不足时返回 `INSUFFICIENT_BALANCE`。
- 余额计费用户不在本次范围内，仍保持预计扣费覆盖检查。

## 实施步骤概要

1. 修改 `SubscriptionService.check_quota_before_request`，取消套餐预估消耗与剩余额度的前置比较。
2. 保留已用额度达到或超过上限时的拒绝逻辑。
3. 更新套餐前置校验单测，覆盖“预估超剩余但仍有剩余额度时放行”。
4. 增加“剩余额度为 0 或负数时拒绝”的测试。
5. 运行相关单测验证。
6. 创建实施记录并进行自审。
