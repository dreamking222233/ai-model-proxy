# 计费-请求前额度余额预检查 Plan

## 用户原始需求

用户反馈：套餐额度耗尽后继续发起请求，后端虽然最终标记为 error 且不扣费，但请求已经发送到上游并消耗了上游资源。需要完善当前计费和用户额度/余额检查体系：用户发起请求后，先判断套餐额度和余额是否正常，没问题再继续发送到上游；如果当日套餐额度不足且余额也不足，直接判定失败并返回原因。

## 技术方案设计

当前文本代理入口已经在解析请求体、模型解析后调用 `ProxyService._assert_text_request_allowed` 做前置检查，并由 `SubscriptionService.check_quota_before_request` 校验套餐周期额度。根据最新规则，请求前不要求预计费用完全覆盖，只要求当日套餐美元额度或余额大于 `0.1` 美元；本次请求完成后按实际用量扣费/计入套餐，允许本次把余额或额度扣到 0 或负数，下一次请求再因阈值不足被拦截。

本次方案：

1. 在 `SubscriptionService` 定义文本请求最小美元阈值 `0.1`。
2. 余额计费用户请求前只判断余额是否大于 `0.1` 美元，大于则继续请求，否则直接返回余额不足。
3. 成本口径套餐请求前判断当日剩余额度是否大于 `0.1` 美元，大于则继续请求，否则触发余额兜底。
4. 套餐额度不足时，继续复用余额兜底逻辑：余额大于 `0.1` 美元才放行，否则直接返回 `INSUFFICIENT_BALANCE`。
5. 请求完成后的真实计费按实际用量扣费/累加额度，允许本次请求后余额或套餐周期额度变为 0 或负数；下一次请求再被前置阈值拦截。
