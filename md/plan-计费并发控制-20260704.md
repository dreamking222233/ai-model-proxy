# 计费并发控制 Plan

## 用户原始需求

用户要求优化当前计费系统：

1. 当用户套餐额度不足，但账户余额仍有且余额大于 2 时，允许正常请求。
2. 当用户无套餐，或余额小于 2 后，请求并发数控制为 3。
3. 保持当前系统的计费优先级：仍然优先走套餐扣费，套餐不足时再走余额兜底或混合扣费。
4. 当套餐当天额度小于 2，并且账户无余额时，也限制请求并发数为 3。
5. 先基于当前系统逻辑分析，再出具详细 plan。

## 当前系统逻辑分析

### 文本请求准入逻辑

当前文本类请求在发送到上游前会构造 `quota_precheck`，再调用：

- `ProxyService._build_text_quota_precheck(...)`
- `ProxyService._assert_text_request_allowed(...)`

主要入口包括：

- OpenAI Chat Completions 链路
- Anthropic Messages 链路
- Responses 链路
- 部分按请求计费的视频链路会复用 `_assert_text_request_allowed`

`quota_precheck` 当前包含：

- `estimated_total_tokens`
- `estimated_total_cost`
- `estimated_quota_cost`
- `estimated_cost_is_exact`
- 价格倍率快照，如全局倍率、用户专属倍率、fast 倍率、长上下文倍率

### 套餐优先逻辑

`ProxyService._assert_text_request_allowed` 当前先刷新用户套餐缓存：

- 如果有有效套餐：
  - 调用 `SubscriptionService.check_quota_before_request(...)` 做套餐额度预检。
  - 如果套餐额度足够，直接放行。
  - 如果套餐额度不足，调用 `_can_fallback_to_balance_for_quota_precheck(...)` 判断余额兜底。
  - 对 `request` 这类精确费用模型，会要求余额足够覆盖预计金额。
  - 对普通 token 计费模型，只要余额为正，即使预计费用高于余额，也会放行。
- 如果没有有效套餐：
  - 调用 `_can_balance_cover_text_precheck(...)` 判断余额。
  - 普通 token 计费模型同样是“余额为正即可放行”。

### 落账逻辑

请求成功后统一进入：

- `ProxyService._deduct_balance_and_log(...)`
- `ProxyService._deduct_balance_and_log_once(...)`

落账顺序如下：

1. 重新读取最新用户、API Key、余额。
2. 重新计算真实 token、缓存 token、上下文倍率、价格倍率。
3. 如果用户是 `balance` 模式，直接扣余额并写入 `consumption_record`、`request_log`。
4. 如果用户是套餐模式：
   - 优先消耗套餐周期额度。
   - 如果套餐额度不足，则按未覆盖部分扣余额。
   - 如果套餐完全不可用且余额可用，则回退到余额扣费。
5. 写入 `request_log`、`consumption_record`，更新 API Key 统计。

这部分逻辑已经符合“优先套餐，再余额兜底”的要求，本次不应重写最终扣费顺序。

### 当前异常根因

当前问题来自请求前准入策略过宽：

- 普通 token 计费模型在余额为正时会放行。
- 余额低、套餐额度低、并发请求多时，多笔请求会同时打到上游。
- 前面的请求落账后把余额扣低或扣负，后面的请求虽然上游已经成功，但本地落账时余额不足。
- 结果是 `request_log` 写入 `本地计费或记账失败：[INSUFFICIENT_BALANCE]`，但没有对应 `consumption_record`。

所以本次优化重点不是改真实计费，而是在“低资产状态”下限制并发，减少并发穿透带来的上游成本和本地计费失败。

## 规则解释与落地口径

### 金额阈值

本方案将用户提出的 `2` 解释为美元口径：

- 账户余额：`user_balance.balance > 2`
- 套餐当天额度：成本口径套餐按 `quota_limit - used_amount > 2`

如果套餐是 token 口径：

- 不直接把 `2` 当 token 阈值。
- 并发风险判断优先看余额。
- 如需精确支持 token 口径套餐，可以另加系统配置把 `$2` 折算为对应模型的预计成本阈值，本次不强行改变旧 token 套餐语义。

### 并发限制触发条件

按用户字面需求，文本/请求型美元计费请求在以下情况下使用每用户 3 并发限制：

1. 用户没有有效套餐。
2. 用户账户余额小于 2。
3. 用户有套餐，但当天成本口径套餐剩余额度小于 2，且账户余额小于等于 0。

以下情况不限制为 3 并发：

1. 用户有有效套餐，且当天成本口径套餐剩余额度大于等于 2。
2. 用户有有效套餐，套餐额度不足或低于 2，但账户余额大于 2。

重要说明：并发限制只控制“同时在跑的上游请求数”，不改变最终扣费来源。最终仍由 `_deduct_balance_and_log_once` 按当前逻辑优先套餐、再余额兜底。

## 技术方案设计

### 1. 新增计费准入结果对象

在 `ProxyService` 内新增轻量结果对象，或新建 dataclass：

```python
@dataclass
class BillingAdmissionDecision:
    user_id: int
    limited: bool
    limit: int = 3
    reason: str | None = None
    balance: Decimal = Decimal("0")
    active_subscription_id: int | None = None
    quota_metric: str | None = None
    quota_remaining_amount: Decimal | None = None
```

调整 `_assert_text_request_allowed(...)`：

- 保留现有异常行为。
- 从原来的只返回 `None`，改为返回 `BillingAdmissionDecision`。
- 现有调用方必须接收该返回值。
- 准入失败仍然抛 `ServiceException`，不返回 decision。

### 2. 新增低资产并发判断辅助方法

新增方法：

- `_build_billing_admission_decision(...)`
- `_get_active_subscription_quota_remaining(...)`
- `_should_limit_billing_concurrency(...)`

判断所需数据：

- `UserBalance.balance`
- `SubscriptionService.resolve_active_subscription(...)`
- `SubscriptionService._get_or_create_cycle(...)`
- `SubscriptionService._get_effective_quota_metric(...)`
- `SubscriptionService._get_effective_quota_limit(...)`

建议常量或系统配置：

- `billing_low_asset_threshold_usd`，默认 `2`
- `billing_low_asset_concurrency_limit`，默认 `3`
- `billing_concurrency_lease_ttl_seconds`，默认 `7200`

### 3. 新增 Redis 并发闸门服务

新增文件：

- `backend/app/services/billing_concurrency_service.py`

核心能力：

```python
class BillingConcurrencyService:
    @classmethod
    def acquire(cls, user_id: int, request_id: str, limit: int, ttl_seconds: int) -> BillingConcurrencyLease:
        ...

    @classmethod
    def release(cls, lease: BillingConcurrencyLease) -> None:
        ...
```

推荐 Redis 数据结构：

- key：`billing_concurrency:user:{user_id}`
- value：sorted set
- member：`request_id`
- score：过期时间戳

获取租约时使用 Lua 脚本保证原子性：

1. 删除已过期 member。
2. 统计当前活跃 member 数。
3. 如果活跃数大于等于 `limit`，返回拒绝。
4. 否则写入当前 `request_id` 和过期时间。
5. 设置 key 级别过期时间。

释放租约：

- `ZREM billing_concurrency:user:{user_id} request_id`

Redis 不可用策略：

- 对需要并发限制的低资产请求，建议 fail closed，返回 `429` 或 `503`，避免继续打上游造成无法计费。
- 对不需要并发限制的正常资产请求，不依赖 Redis。

错误建议：

- HTTP 状态：`429`
- error_code：`BILLING_CONCURRENCY_LIMITED`
- message：`当前余额或套餐额度较低，同时请求较多，请等待前面的请求完成后重试`

### 4. 在上游调用外层接入并发租约

调用点改造原则：

1. 请求体解析、模型解析、安全扫描、额度预检完成后，再获取并发租约。
2. 获取租约成功后才调用上游。
3. 上游调用、流式响应、异常处理结束后，在 `finally` 中释放租约。
4. 如果准入 decision 不需要限制，则不触发 Redis。

需要调整的主要位置：

- OpenAI Chat Completions 入口
- Anthropic Messages 入口
- Responses HTTP 入口
- Responses WebSocket 入口
- 复用 `_assert_text_request_allowed` 的请求型视频入口

为了降低侵入，可以封装上下文管理器：

```python
with BillingConcurrencyService.guard(decision, request_id):
    # existing upstream call
```

异步/流式入口需要 async 友好的封装，或手动：

```python
lease = BillingConcurrencyService.acquire_if_needed(decision, request_id)
try:
    ...
finally:
    BillingConcurrencyService.release(lease)
```

### 5. 保持现有扣费逻辑不变

本次不修改以下核心逻辑：

- `_deduct_balance_and_log_once`
- `SubscriptionService.consume_quota_amount_after_request`
- `SubscriptionService.check_quota_before_request` 的最终额度判定语义
- `consumption_record`、`request_log` 写入字段

原因：

- 当前最终扣费顺序已经是套餐优先、余额兜底。
- 本需求要解决的是低余额/低额度并发穿透问题，不是重构计费模型。

### 6. 可观测性

新增日志：

- 低资产并发限制决策日志：
  - `user_id`
  - `request_id`
  - `limited`
  - `reason`
  - `balance`
  - `quota_remaining_amount`
  - `limit`
- 获取租约失败日志：
  - 当前活跃请求数
  - 限制值
  - Redis key
- 释放租约异常日志：
  - 只记录 warning，不影响已经完成的请求响应

建议后续可在管理端请求日志中增加 `billing_admission_reason`，但本次不要求改表。

## 测试验证计划

### 单元测试

新增或补充以下用例：

1. 无套餐用户，余额 `100`，decision 为 `limited=True`，limit 为 `3`。
2. 有套餐，成本口径当天剩余额度 `3`，余额 `0`，decision 为 `limited=False`。
3. 有套餐，成本口径当天剩余额度 `1.5`，余额 `2.01`，decision 为 `limited=False`。
4. 有套餐，成本口径当天剩余额度 `1.5`，余额 `0`，decision 为 `limited=True`。
5. 有套餐，余额 `1.99`，按字面规则 decision 为 `limited=True`。
6. token 口径套餐不把 `2` 当 token 阈值，至少不误判为“额度大于 2 美元”。
7. Redis acquire 前 3 次成功，第 4 次返回 `BILLING_CONCURRENCY_LIMITED`。
8. release 后可再次 acquire。
9. TTL 过期的租约会被清理，不永久占用并发槽。

### 集成测试

模拟低余额用户连续发起 4 个请求：

- 前 3 个获取并发租约成功。
- 第 4 个在上游调用前返回 429。
- 未产生上游调用。
- 未产生异常消费流水。

模拟套餐优先：

- 用户有套餐，余额不足但套餐额度足够。
- 请求成功后仍走套餐扣费。
- 并发 decision 可按规则限制或不限制，但不改变 `billing_mode=subscription`。

模拟套餐不足余额兜底：

- 套餐剩余额度不足。
- 余额大于 `2`。
- 请求正常放行，不触发 3 并发限制。
- 落账仍按现有 mixed/balance fallback 逻辑。

## 风险与注意事项

1. 低资产并发限制必须在上游调用前执行，否则无法阻止上游成本。
2. 流式请求必须确保 finally 释放租约；客户端断开也要释放。
3. Redis 不可用时，对需要限制的请求建议 fail closed，否则会回到旧风险。
4. 不要在本次改动中直接把 token 计费请求改成“预计费用必须完全覆盖”，否则可能改变当前产品体验。
5. 不要改最终落账顺序，避免引入套餐扣费回归。
6. 当前“无套餐用户始终限制 3 并发”是按用户需求字面实现；如果希望余额大于 2 的纯余额用户不限制，需要在实施前调整规则。

## 验收标准

1. 用户无套餐时，同一用户最多 3 个并发上游请求。
2. 用户余额小于 2 时，同一用户最多 3 个并发上游请求。
3. 套餐当天成本额度小于 2 且无余额时，同一用户最多 3 个并发上游请求。
4. 套餐额度不足但余额大于 2 时，请求可正常发起。
5. 请求完成后仍按当前逻辑优先扣套餐，套餐不足再扣余额。
6. 第 4 个受限并发请求在上游调用前被拒绝。
7. 不再大规模出现“上游成功但本地计费失败”的低余额并发异常。
