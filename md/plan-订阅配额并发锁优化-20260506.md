## 用户原始需求

- 深度分析当前目录下的项目，定位 `subscription_usage_cycle` 长事务与锁等待根因。
- 修复高并发下 `SELECT ... FOR UPDATE` 导致的订阅配额锁超时问题。
- 优先解决：
  - 缩小或移除 `subscription_usage_cycle` 上的悲观锁范围
  - 避免事务在外部请求期间持锁
  - 增加必要的重试与测试覆盖

## 当前问题分析

### 现象

- 数据库中 `subscription_usage_cycle` 存在持续锁等待，清理后会快速复现。
- 高并发代理请求会争抢同一用户同一周期记录。
- 锁等待最终表现为 MySQL `innodb_lock_wait_timeout` 超时。

### 已确认根因

1. `backend/app/services/subscription_service.py`
   - `check_quota_before_request()` 在存在估算值时调用 `_get_or_create_cycle(..., lock=True)`。
   - `consume_quota_after_request()` 固定调用 `_get_or_create_cycle(..., lock=True)`。
   - `_get_or_create_cycle()` 对 `subscription_usage_cycle` 执行 `with_for_update()` 后，再在 Python 中完成额度判断和累加。

2. 这套流程属于典型“读后改”悲观锁模型：
   - `SELECT ... FOR UPDATE`
   - Python 读取 `used_amount`
   - Python 计算 `next_used_amount`
   - ORM 回写 `used_amount/request_count/last_request_id`

3. 在高并发下，同一 `subscription_id + cycle_date` 只能串行通过：
   - 后续事务必须等待前一个事务释放行锁
   - 当请求量持续时，会形成锁队列
   - 即使单个事务不算很长，整体吞吐仍会被锁串行化拖垮

4. 预检阶段当前还会主动加锁，进一步放大争用：
   - `ProxyService._assert_text_request_allowed()` 会在正式转发前调用 `check_quota_before_request()`
   - 对文本请求，`quota_precheck` 基本总是存在
   - 这意味着大量“请求前判断”本身就在竞争同一条周期记录

### 事务生命周期判断

- `proxy_service.py` 在真正调用上游前多数路径会执行 `release_session_connection(db)`，因此“持锁跨外部 API 调用”不是当前唯一根因。
- 但当前方案仍然在请求前、请求后都使用悲观锁，足以在高并发下造成严重串行化。
- 因此核心修复点应放在“改记账模型”，而不是只做事务收缩。

## 修复目标

1. 预检阶段不再对 `subscription_usage_cycle` 使用 `FOR UPDATE`。
2. 请求后记账改成数据库原子条件更新，避免“先锁行再读改写”。
3. 首次创建周期记录时继续保留唯一键竞争恢复能力。
4. 对瞬时死锁/锁超时增加有限重试，提升高并发下恢复性。
5. 不改变现有业务语义：
   - unlimited 超限仍报 `SUBSCRIPTION_UNLIMITED_DAILY_TOKEN_EXCEEDED`
   - daily_quota 超限仍报 `SUBSCRIPTION_DAILY_QUOTA_EXCEEDED`
   - quota 超限且余额足够时，仍允许回退余额计费

## 技术方案设计

### 方案总览

- 保留“请求前预检 + 请求后真实记账”两段式语义。
- 将预检改为无锁快照读取，仅作为提前失败提示。
- 将真实记账改为单条 SQL 原子更新：
  - `UPDATE subscription_usage_cycle`
  - `SET used_amount = used_amount + :consumed_amount, request_count = request_count + 1, last_request_id = :request_id`
  - `WHERE id = :cycle_id AND used_amount + :consumed_amount <= quota_limit`
- 若更新影响行数为 0，则判定为并发下额度已被其他事务占用完毕，返回原有业务异常。

### 具体实现

#### 1. 周期读取/创建拆分职责

- `_get_or_create_cycle()` 保留“查询或创建周期记录”的职责，但不再承担加锁记账。
- 新增原子记账辅助方法，专门负责：
  - 获取或创建周期记录
  - 执行条件更新
  - 更新成功后重新查询最新周期快照并返回记账结果

#### 2. 预检无锁化

- `check_quota_before_request()` 中调用 `_get_or_create_cycle(..., lock=False)`。
- 预检只读取当前 `used_amount`，不阻塞其他请求。
- 允许多个并发请求同时通过预检，最终以请求后原子记账结果为准。

#### 3. 记账原子化

- `consume_quota_after_request()` 不再先 `FOR UPDATE`。
- 改为：
  1. 读取或创建周期记录
  2. 执行条件 `UPDATE`
  3. 在同一条更新语句中同步最新 `quota_metric/quota_limit`
  4. 若成功，查询最新周期返回 `subscription_cycle_id/quota_used_after`
  5. 若更新影响行数为 0，先重读周期再判断是否真实超限

#### 4. 有限重试

- 对以下并发异常做短重试：
  - MySQL lock wait timeout
  - deadlock found
- 重试放在 `ProxyService._deduct_balance_and_log()` 的事务边界，而不是在同一个 Session 内原地重试。
- `subscription_service` 仅负责抛出可识别的并发数据库异常或业务超限异常。
- 重试次数控制在 2-3 次，失败则按业务异常或原异常返回。

#### 5. 兼容现有回退逻辑

- `ProxyService._deduct_balance_and_log()` 保持现有流程：
  - 套餐记账成功则按 subscription 记账
  - quota 超限可回退 balance
  - unlimited 超限直接失败

## 影响范围分析

### 直接修改文件

- `backend/app/services/subscription_service.py`
- `backend/app/test/test_subscription_compatibility.py`
- `backend/app/test/test_request_audit_hardening.py`

### 间接受影响模块

- `backend/app/services/proxy_service.py`
  - 依赖 `check_quota_before_request()` 与 `consume_quota_after_request()` 的行为
- 订阅统计、消费记录、请求日志
  - 依赖 `subscription_cycle_id/quota_used_after` 等返回值

### 风险点

1. 预检无锁后，前置校验可能放过一部分最终会在请求后失败的请求。
   - 这是可接受变化，本质上是将强一致从“请求前”调整为“请求后原子记账”。
   - 现有余额回退逻辑已支持部分场景兜底。

2. 原子更新后需要重新读取周期快照，保证日志字段准确。

3. 重试实现不能吞掉真实业务超限错误，必须区分：
   - 并发异常
   - 额度真实超限
   - 其他数据库异常

4. 并发异常若已污染当前 Session，必须回滚整个本地记账事务后再重试，不能在同一事务内盲目继续。
