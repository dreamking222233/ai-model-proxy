# 请求失败重试机制实施方案

## 用户原始需求

当前项目是中转站系统，接收用户请求后进行计费、请求上游渠道并返回结果。部分请求会因为本系统或上游临时原因失败，失败信息会直接返回给用户，影响稳定性体验。需要新增请求失败重试机制：请求失败后重试 3 次，超过重试次数仍失败时再按现有逻辑返回失败信息并记录失败请求。

## 当前系统分析

- 核心入口集中在 `backend/app/services/proxy_service.py`：
  - 文本：`handle_openai_request`、`handle_anthropic_request`、`handle_responses_request`
  - Responses WebSocket：`handle_responses_websocket`
  - 图片：`handle_image_request`、`handle_image_edit_request`
- 计费与日志在上游成功后执行：
  - 文本：`_finalize_successful_text_request` -> `_deduct_balance_and_log`
  - 图片：`_deduct_image_credits_and_log`
  - 失败：`_log_failed_request`
- 已有能力：
  - `_post_with_retries` 已覆盖部分非流式 JSON POST 请求，但当前常量语义是“最多尝试 3 次”，与“失败后重试 3 次”不完全一致。
  - 多渠道 failover 已存在：单渠道失败后会尝试下一个渠道。
- 主要缺口：
  - Google 图片生成、OpenAI 图片编辑、Vertex 图片生成未完整复用重试能力。
  - OpenAI via Anthropic 非流式请求未复用 `_post_with_retries`。
  - 流式请求的失败多在 `StreamingResponse` 生成器内捕获并直接返回错误事件，入口层无法再透明重试或 failover。

## 技术方案设计

1. 统一重试语义
   - 新增 `_UPSTREAM_RETRY_MAX_RETRIES = 3`。
   - `_UPSTREAM_RETRY_ATTEMPTS = _UPSTREAM_RETRY_MAX_RETRIES + 1`，表示首次请求 + 3 次重试。
   - 保留现有可重试状态码：`408/409/425/429/500/502/503/504`。
   - 保留现有可重试传输异常：超时、连接失败、读失败、远端协议异常、网络异常。

2. 扩展上游调用重试工具
   - 保留并调整 `_post_with_retries`。
   - 新增 `_post_files_with_retries`，用于 multipart 图片编辑请求。
   - 新增 `_stream_lines_with_retries`，用于可安全替换的流式请求首行前透明重试。

3. 非流式请求覆盖
   - OpenAI / Responses / Anthropic 现有非流式 JSON POST 继续使用 `_post_with_retries`。
   - OpenAI via Anthropic 非流式改为 `_post_with_retries`。
   - Google 图片生成改为 `_post_with_retries`。
   - OpenAI 图片编辑改为 `_post_files_with_retries`。
   - Vertex 图片生成在服务内部对单个候选模型增加 3 次重试，候选模型之间仍保留原有 fallback。

4. 流式请求覆盖
   - 对 OpenAI stream、OpenAI via Anthropic stream、Responses stream/WebSocket，在“尚未向客户端产生有效输出”前允许重试。
   - 一旦已经收到会被下游消费的 line/payload，则不再透明重试，避免重复输出、重复工具调用或计费状态混乱。
   - Responses WebSocket 复用 `_iter_responses_upstream_payloads`，该迭代器同样只在首个 payload 前重试，空行和 `event:` 行不关闭重试窗口。
   - Anthropic passthrough stream 当前已有 prompt-cache variant fallback 与压缩 fallback，暂不强行套通用 helper，避免破坏既有专属 fallback。

5. 日志与计费
   - 重试过程只写 warning 日志，不写请求失败记录。
   - 只有最终仍失败时保留现有 `_log_failed_request`。
   - 成功只执行一次成功计费与成功日志。

## 实施步骤概要

1. 调整重试常量语义为“失败后重试 3 次”。
2. 增加 multipart 与流式首行前重试 helper。
3. 替换未覆盖的非流式上游调用。
4. 替换 OpenAI/Responses 相关可安全覆盖的流式上游迭代调用。
5. 为 Vertex 图片服务增加候选模型内重试。
6. 补充单元测试覆盖重试次数、不可重试状态、流式首行前重试、首行后不重试。
7. 运行相关测试与语法检查。
