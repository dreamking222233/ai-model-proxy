# 错误提示与重试配置实施方案

## 用户原始需求

修改当前系统的错误提示：如果是上游渠道返回的错误信息，不要直接返回给用户；用户请求调用上游失败后增加重试机制，默认 3 次，最好可在 `admin/config` 页面动态配置重试次数。超出重试次数仍不可用时，用户只看到类似 `503 调用失败，渠道异常` 的简略信息；管理员在 `admin/logs` 详情中仍可看到具体错误，包括上游返回内容。同时添加必要禁用词过滤，比如 `providers=codex` 以及上游返回的所有中文字。补充需求：如果用户请求上下文超过 250000 token，且本系统配置的请求模型只支持 256k 上下文，调用失败/预检失败时返回 `请求超出最大上下文,请压缩对话`。

## 技术方案设计

1. 复用现有 `system_config.max_retry_count` 配置，新增读取 helper，把重试次数从静态常量改为按请求从数据库读取，默认 3，限制范围避免异常配置拖垮请求。
2. 调整 `proxy_service.py` 的 JSON、multipart、流式首个有效输出前重试 helper，支持传入动态重试次数。
3. 统一上游错误的用户可见响应：最终所有渠道失败时不拼接上游原文，只返回 `503 调用失败，渠道异常，请稍后重试`；失败日志仍保存原始 `error_detail`。
4. 上游 4xx 请求错误不再把上游 body/message 直接透传给用户，改为平台简略错误；原始错误仍进入日志或后端日志。
5. 增加用户可见错误文本兜底清洗：过滤 `providers=codex` 和中文片段，避免上游中文原文出现在对外响应。
6. 扩展本地请求长度预检：结合 `unified_model.max_tokens`，当模型上下文上限不超过 256000 且预估 token 超过 250000 时返回压缩对话提示。
7. 在 `admin/config` 页面新增“上游失败重试次数”输入项，保存到 `max_retry_count`。
8. 补齐初始化 SQL 中 `max_retry_count` 的描述和必要升级脚本，保证新部署、老部署都有该配置。
9. 补充或调整测试，覆盖动态重试读取、最终响应脱敏、日志保存原始错误、256k 模型上下文超限提示。
