## Findings

未发现新的阻断级问题，当前实现已满足本轮需求主线：

- Google 生图模型可按分辨率配置积分
- `image_size / imageSize / size` 已具备兼容解析
- Google 请求已可透传 `imageSize`
- 图片积分已支持小数
- 失败请求仍保持不扣费

## 已确认点

### 1. 分辨率规则与官方能力约束

已在服务层做硬约束，不仅是前端限制：

- `gemini-2.5-flash-image` 仅允许 `1K`
- `gemini-3.1-flash-image-preview` 允许 `512 / 1K / 2K / 4K`
- `gemini-3-pro-image-preview` 允许 `1K / 2K / 4K`

同一模型下：

- `resolution_code` 唯一
- 默认档位必须且只能有一个
- 默认档位必须处于启用状态

### 2. 小数图片积分

代码链路已从整数切到 `Decimal / DECIMAL(12,3)`：

- 模型默认倍率
- 用户图片积分余额
- 图片积分流水
- 请求日志中的实际图片扣费
- 后台图片积分充值 / 扣减

已补测试验证 `4.5` 这类值不会在服务层被截断。

### 3. 参数兼容

当前兼容策略正确：

- `image_size` 优先
- `imageSize` 次之
- `size` 同时兼容新分辨率语义和旧尺寸映射语义

并已补测试覆盖：

- `size=1K`
- `size=1024x1024`
- 非法 `size`

## 残余风险

### 1. 真实上游联调尚未完成

虽然本地环境已确认存在：

- Google 渠道
- 有效 API Key
- 有图片积分余额的测试用户

但完整实测在用户切换需求后中止，当前还没有把：

- `1K / 2K / 4K` 的真实请求
- 上游返回
- 本地扣费结果

完整跑通并固化成记录。

这不是代码阻断问题，但仍属于发布前建议补齐项。

### 2. 细分 CRUD 测试仍偏轻

当前测试已覆盖：

- 能力白名单
- 参数兼容
- 失败不扣费
- 小数扣费值处理

但还没有基于真实 DB Session 的分辨率规则 CRUD 集成测试。

## 结论

当前版本可以提交代码。

若后续作为正式上线版本，建议在提交后继续补两项：

1. 真实 Google 上游联调记录
2. 分辨率规则 CRUD 集成测试
