## 发现的问题

### 1. 缺少“当前系统指向自身/形成回环”的保护，容易把新变体配置成递归调用

相关代码：

- `backend/app/services/proxy_service.py:8841-8858`
- `backend/app/services/proxy_service.py:9383-9385`
- `backend/app/services/proxy_service.py:9509-9511`
- `backend/app/services/health_service.py:93-99`
- `backend/app/services/health_service.py:464-470`

说明：

- 新变体是显式给“当前系统作为上游”使用的，因此把 `base_url` 误配为本机地址、同域入口，或两套系统互相回指的概率，明显高于普通 OpenAI 兼容渠道。
- 当前实现只有路径切换，没有任何自引用检测、回环标记、最大转发层级控制。
- 一旦误配到自身，`/v1/image/created` 或 `/v1/image/edit` 会再次进入同一套选路逻辑，最终表现为递归转发、请求超时、重复消耗资源，健康检查手动开启后也会触发同类问题。

建议：

- 至少增加一层防护：禁止把渠道 `base_url` 配置为当前系统自身对外地址。
- 更稳妥的做法是在转发请求里加内部标记头，例如 `X-Relay-Hop`，超过 1 层直接拒绝，避免跨实例回环。

### 2. 实施记录中的测试命令不可直接复现，交接时会误导后续回归

相关文档：

- `md/impl-图片渠道-当前系统上游适配-20260521.md:91-96`

复核结果：

- 文档里写的是在仓库根目录执行 `python -m unittest app.test.test_openai_image_channel`
- 该命令在当前仓库根目录下会报 `ModuleNotFoundError: No module named 'app'`
- 实际可复现方式是进入 `backend/` 目录后执行，或显式设置 `PYTHONPATH`

建议：

- 把实施记录中的命令改成：

```bash
cd backend
python -m unittest app.test.test_openai_image_channel
python -m unittest app.test.test_channel_health_monitor_switch
python -m unittest app.test.test_vertex_image_channel
```

或写成单行带工作目录的形式，避免后续人按文档执行失败。
