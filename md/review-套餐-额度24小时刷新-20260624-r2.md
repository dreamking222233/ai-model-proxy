# 套餐-额度24小时刷新 Review R2

## 审查结论

通过。

实现已按 `user_subscription.start_time` 作为用户当前套餐额度刷新锚点，套餐额度周期改为 rolling 24 小时窗口。北京时间 0 点不再触发刷新，日卡在 `start_time + 24h` 到期边界不会获得额外完整额度周期。

管理端 `admin/subscription`、代理端 `agent/subscription`、用户端 `user/recharge` 三类开通入口最终复用 `SubscriptionService` 的周期获取和扣费逻辑，因此会继承同一套刷新规则。

## 已检查重点

1. 周期计算
   - `_get_subscription_cycle_window(...)` 使用 `start_time + N * 24h` 计算周期。
   - `now >= end_time` 时按 `end_time - 1 microsecond` 归属到最后一个有效周期，避免到期点新开周期。
   - 固定使用 `Asia/Shanghai` 语义，符合“所有时间按北京时间算”。

2. 旧数据兼容
   - `_get_or_create_cycle(...)` 命中旧 0 点周期时通过 `_sync_cycle_snapshot(...)` 重建窗口和用量快照。
   - 新建周期前会从 `ConsumptionRecord` 聚合已有消费，不会给部署前已有消费用户额外空额度。
   - 新增 SQLite ORM session 级测试覆盖旧自然日周期、跨 0 点消费、满 24 小时后新周期重建。

3. 用户端展示
   - `user/balance` 套餐卡片新增“下次刷新”。
   - 前端直接格式化后端北京时间字符串，避免浏览器本地时区转换。
   - R1 建议中的无限套餐文案已修正，限额套餐显示“每24小时刷新”，无限套餐保留“无限套餐”语义。

4. 接口语义
   - daily quota 套餐 summary 返回 `refresh_anchor_at`、`next_refresh_at`、`refresh_period_hours=24`。
   - 余额模式默认 summary 不返回刷新周期值。
   - 套餐相关时间字符串明确带 `+08:00` 北京时间偏移，管理端、代理端、用户端展示口径一致。

## 剩余建议

无阻断项。

后续如果管理端或代理端增加大批量套餐列表并展示当前周期，建议再做批量聚合优化，避免每个活跃套餐单独聚合 `ConsumptionRecord`。
