# 套餐美元日限额改造 Review

日期：20260510

## 结论

本次实现与方案基本一致，未发现阻断交付的逻辑缺陷。

已核对通过的重点：

- 无限套餐日 / 周 / 月按存量 `120 USD/天`、新开 `100 USD/天`
- 畅享套餐按存量 `60 USD/天`、新开 `50 USD/天`
- 套餐额度成本与余额计费成本已拆分，套餐额度不复用 `price_multiplier` / `token_multiplier`
- 管理端、代理端和用户侧订阅摘要展示已切到解析后的额度口径
- 旧 `subscription_usage_cycle` 如果仍是 `total_tokens` 口径，会在读取时按当日消费记录自动重建为当前额度口径，避免历史 `used_amount` 被误判成美元额度

## 已确认的实现点

- `backend/app/services/subscription_service.py`
  - 新增订阅额度策略解析，基于订阅快照识别无限套餐日 / 周 / 月与畅享套餐。
  - 存量用户判定基于 `created_at < 2026-05-10 00:00:00`。
  - `check_quota_before_request()` 和 `consume_quota_after_request()` 已改为读取有效额度口径。
  - 新开无限套餐日 / 周 / 月和畅享套餐会写入新的美元额度快照。
  - 当历史周期记录的 `quota_metric` 与当前策略不一致时，会根据周期窗口内的 `ConsumptionRecord` 重建 `used_amount/request_count/last_request_id`，而不是直接复用旧值。

- `backend/app/services/proxy_service.py`
  - 新增 `estimated_quota_cost` 作为请求前套餐额度预检成本。
  - 新增 `quota_cost` 作为请求后套餐额度实际累计成本。
  - 余额扣费仍继续使用原有倍率后 `total_cost`。

- 前端展示
  - 管理端套餐模板页 / 套餐记录页已改为展示解析后的额度。
  - 代理端库存页 / 发放记录页已改为展示解析后的额度，并把记录列改为“当日已用”。
  - 用户资料页在存在 `current_cycle` 时会展示当日剩余额度，不再只限 `daily_quota`。

## 残余风险

- 请求前预检仍依赖 `max_tokens` / `max_output_tokens` 等显式输出上限；未声明上限时，预检能力有限，最终仍以请求后实际记账兜底。
- 畅享套餐目前默认保留原有 `daily_quota` 的余额兜底行为；如果业务后续要求也改成硬阻断，需要单独再收敛。
- 这次没有修改数据库模板数据；运行时展示与发放逻辑已经按新规则处理，但如果后续希望数据库中的模板快照也彻底落成美元口径，可再补一轮数据清理。
- 已存在的历史周期会在下一次校验或记账时被自动纠正；如果希望上线后立即修正所有当天活跃用户的周期展示，需要再补一条批量修复脚本或 SQL。
