**审查结论**

发现以下问题，建议修复后再认为实现通过：

1. [frontend/src/views/user/MediaWorkbench.vue](/Volumes/project/modelInvocationSystem/frontend/src/views/user/MediaWorkbench.vue:106) 引用了 `videoSecondsOptions`，但组件中没有定义该字段或 computed。结果是「时长」下拉为空，视频参数无法正常选择，不符合“视频参数：时长”的需求。建议在 `data()` 中补充如 `[5, 10, 15]` 或按模型能力生成。

2. [frontend/src/views/user/MediaWorkbench.vue](/Volumes/project/modelInvocationSystem/frontend/src/views/user/MediaWorkbench.vue:523) 视频内容拉取失败时会回退到 `contentUrl`，但后端 [backend/app/api/proxy/video_proxy.py](/Volumes/project/modelInvocationSystem/backend/app/api/proxy/video_proxy.py:93) 的 `/v1/videos/{video_id}/content` 需要 `Authorization`。回退后的 `<video src>`、打开、下载都不会带鉴权头，用户会遇到 401 或空白视频。建议取消无鉴权 URL 回退，或让下载/预览始终走带 Authorization 的 `fetch` blob。

3. [backend/app/services/media_workbench_service.py](/Volumes/project/modelInvocationSystem/backend/app/services/media_workbench_service.py:122) 的 `not_(local_error_expr)` 存在 SQL 三值逻辑风险：当失败日志 `channel_id is null` 且 `error_message is null` 时，表达式结果为 `NULL`，会被 `WHERE` 过滤掉。这与方案里“无法准确区分的失败仍作为最近调用成功率统计”不一致，会让健康度偏高。建议对 `error_message` 使用 `func.coalesce(RequestLog.error_message, "")` 后再 `like`。

4. [frontend/src/views/user/MediaWorkbench.vue](/Volumes/project/modelInvocationSystem/frontend/src/views/user/MediaWorkbench.vue:308) `loadInitialData()` 用 `Promise.all`，但 `loadModels()`、`loadApiKey()` 没有局部 catch。API Key 获取失败或模型接口失败会导致初始化整体 rejected，页面只表现为按钮不可用且无明确提示。建议至少给 `loadApiKey()` 加错误提示，模型加载失败时设置 `errorMessage`。

5. [frontend/src/views/user/MediaWorkbench.vue](/Volumes/project/modelInvocationSystem/frontend/src/views/user/MediaWorkbench.vue:248) `image_resolution_rules` 分支没有过滤 `512`，而 fallback 分支过滤了 `512`。如果后端返回启用的 `512` 规则，页面会展示不在本次需求内的分辨率，并且路由提示会误显示“2K/4K 使用 Native Size”。建议统一只允许 `1K/2K/4K`。

整体上，路由、菜单、健康接口、1K compatible 优先排序、API Key helper 抽取方向符合方案；但上面第 1、2 项会直接影响用户视频工作流，第 3 项会影响健康度可信度，建议优先修复。本次我做的是静态代码审查，没有重新运行测试或构建。
