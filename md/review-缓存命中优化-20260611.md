# 缓存命中优化自审记录

## 审查结论

实现整体符合需求，可以进入提交与服务器灰度观察。

## 已检查点

- 渠道亲和 key 包含协议、用户请求模型、代理 ID、用户 ID、会话来源 hash，不会跨用户或跨模型复用。
- 亲和绑定只保存 `affinity_key -> channel_id`，不缓存上游响应内容，不会产生响应内容串用风险。
- OpenAI Chat、Anthropic Messages、Responses 文本入口均已接入亲和排序；图片、视频未接入，符合本次“文本缓存命中”目标。
- 成功绑定在 `_finalize_successful_text_request` 中执行，覆盖文本流式/非流式成功后的统一记账路径。
- Anthropic history 断点已从最新 user 改为倒数第二个 user，避免最新输入变化破坏缓存前缀。
- cache_control 注入后限制最多 4 个断点，并规范 1h/5m TTL 顺序。
- SQL 升级脚本和初始化 SQL 已补渠道亲和配置；root `sql/initData.sql` 已同步补齐之前缺失的 Prompt Cache 策略配置。

## 发现并已处理的问题

1. root `sql/initData.sql` 缺少 `anthropic_prompt_cache_control_policy`、`anthropic_prompt_cache_normalize_user_ids`、`anthropic_prompt_cache_normalize_agent_ids`。

处理：已同步补齐，避免新环境只有 root 初始化 SQL 时前端配置项缺失。

## 风险与观察项

- 当前亲和缓存是进程内 TTL 存储，多进程或多节点部署时只能在单进程内生效。先用于灰度验证，如果生产多 worker 下收益不稳定，下一步改为 Redis。
- 如果某个渠道本身缓存池表现很差，亲和只能减少跨渠道切换损耗，不能保证该渠道达到首尔渠道相同命中率。
- `codex exec` 自动审查进程在本地收尾阶段卡住，已手动结束并完成上述人工审查。
