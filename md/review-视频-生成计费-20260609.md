审查结论：实现还不建议通过，核心需求“错误返回、确认成功后才计费、失败不计费”仍有未覆盖路径。

**主要问题**

1. 流式 chat-completions 分支先把成功结束发给客户端，之后才校验和计费  
   [proxy_service.py](/Volumes/project/modelInvocationSystem/backend/app/services/proxy_service.py:11272) 在收到上游 `[DONE]` 时立即 `yield "data: [DONE]"`，实际 URL 校验在 [proxy_service.py](/Volumes/project/modelInvocationSystem/backend/app/services/proxy_service.py:11339) 之后才执行。  
   结果是：如果最终没有视频 URL、URL 返回 403/JSON 错误或计费失败，客户端已经收到正常结束，错误无法正确返回。建议拦截上游 `[DONE]`，先完成校验和 `_log_video_success`，失败时发送 error SSE，再发送 `[DONE]`。

2. `/v1/created/video` 等待完成后仍未校验视频内容端点  
   [proxy_service.py](/Volumes/project/modelInvocationSystem/backend/app/services/proxy_service.py:11623) 在任务状态 completed 后直接扣费，随后才设置本地 `content_url`。但没有请求上游 `/content` 验证是否 2xx、非空、非 JSON/text 错误。  
   这仍会出现“任务状态完成但内容不可下载/返回错误内容也计费”。建议在扣费前对上游 content URL 做 Range GET 校验，通过后才 `_log_video_success`。

3. chat-completions 非流式响应中，只要提取到 URL 就跳过失败语义判断  
   [proxy_service.py](/Volumes/project/modelInvocationSystem/backend/app/services/proxy_service.py:10541) 提取 content，随后 [proxy_service.py](/Volumes/project/modelInvocationSystem/backend/app/services/proxy_service.py:10543) 有 URL 就直接返回，失败文本检查在后面。  
   如果上游返回 “video_generation_failed ... https://...” 之类内容，仍可能进入 URL 校验并计费。建议先检查 content/status/error/finish_reason 的失败语义，再接受 URL。

4. `/v1/videos` 仍是创建成功即计费，未满足全量“确认成功后才计费”  
   路由 [video_proxy.py](/Volumes/project/modelInvocationSystem/backend/app/api/proxy/video_proxy.py:46) 默认调用 `handle_video_request`，而 [proxy_service.py](/Volumes/project/modelInvocationSystem/backend/app/services/proxy_service.py:10789) 默认 `bill_on_create=True`。  
   impl 文档已把它列为待优化项，但如果原始需求覆盖“用户调用视频生成”的所有入口，这就是功能缺口。建议引入 pending video task 持久化表，retrieve/content 确认成功后幂等扣费。

5. 测试脚本包含硬编码地址和 API Key  
   [created_video.py](/Volumes/project/modelInvocationSystem/backend/app/test/created_video.py:12) 和 [created_video.py](/Volumes/project/modelInvocationSystem/backend/app/test/created_video.py:13) 有真实形态的 base URL/API key。建议改为环境变量必填或示例占位值，并补充可自动断言的单元测试/集成测试，覆盖“不扣费”和“成功扣费”分支。
