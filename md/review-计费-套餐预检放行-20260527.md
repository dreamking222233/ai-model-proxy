# 计费-套餐预检放行 Review

## 审查结论

未发现阻塞性问题。实现与 `plan-计费-套餐预检放行-20260527.md` 的核心需求一致：

- `check_quota_before_request` 已改为只判断 `remaining_amount <= 0`，不再用 `estimated_quota_cost` / `estimated_total_tokens` 提前拦截套餐请求。
- 余额用户路径仍保留预估费用覆盖检查，未被这次改动影响。
- `_serialize_cycle` 已允许 `remaining_amount` 返回负数，符合“本次扣成负数后展示负数”的要求。
- 单测覆盖了预估超剩余但仍放行、额度为 0 拒绝、负数 remaining 序列化。

## 优化建议

1. 可补充 `remaining_amount < 0` 的前置拒绝单测。当前代码 `remaining_amount <= 0` 已覆盖负数逻辑，但测试只有 `used_amount == quota_limit` 的 0 场景。
2. 可补充代理层集成式单测：套餐已负数、余额不足时，`ProxyService._assert_text_request_allowed` 最终返回 `INSUFFICIENT_BALANCE`，锁住完整外部行为。
