# 反代项目缓存机制研究总结（20260611）

## 研究范围

本次在 `缓存研究/` 目录拉取并阅读了三个项目：

- `CLIProxyAPI`：`https://github.com/router-for-me/CLIProxyAPI`
- `new-api`：`https://github.com/QuantumNous/new-api`
- `sub2api`：`https://github.com/Wei-Shaw/sub2api`

重点查看请求转发、渠道/账号选择、prompt cache、`cache_control`、`prompt_cache_key`、会话粘性、用量统计与失败重试相关实现。

## 结论先行

这三个项目基本都没有做“通用 LLM 响应缓存”。它们优化缓存命中的核心思路是：

1. 保留或生成稳定的会话标识，例如 `prompt_cache_key`、`session_id`、`metadata.user_id`。
2. 把同一会话或同一缓存 key 固定到同一上游渠道/账号，避免跨上游池导致缓存失效。
3. 为 Anthropic/Claude 请求自动或半自动补 `cache_control` 断点，尤其是 tools、system、历史消息。
4. 收集 cache read/cache creation token，用于观测缓存效果，但很少直接基于缓存命中率做自动调度。

对当前系统最有帮助的不是“自己缓存模型输出”，而是增加：

- 会话/缓存 key 级别的渠道亲和；
- 按渠道统计 cache read ratio，并作为调度评分；
- 更稳的 Claude `cache_control` breakpoint 策略；
- Codex/OpenAI 兼容请求的 `prompt_cache_key` 稳定化。

## CLIProxyAPI

### 发现

CLIProxyAPI 在缓存优化上比较激进，重点不是本地响应缓存，而是请求形态稳定化。

关键代码：

- `缓存研究/CLIProxyAPI/internal/runtime/executor/claude_executor.go`
- `缓存研究/CLIProxyAPI/internal/runtime/executor/codex_executor.go`
- `缓存研究/CLIProxyAPI/internal/runtime/executor/helps/cache_helpers.go`
- `缓存研究/CLIProxyAPI/internal/runtime/executor/helps/session_id_cache.go`
- `缓存研究/CLIProxyAPI/internal/runtime/executor/helps/user_id_cache.go`
- `缓存研究/CLIProxyAPI/internal/cache/codex_reasoning_replay_cache.go`

主要机制：

- Claude 请求没有 `cache_control` 时自动注入：
  - 最后一个 tool：缓存全部工具定义；
  - 最后一个 system block：缓存系统提示；
  - 倒数第二个 user turn：缓存多轮历史前缀。
- 限制 Anthropic 最多 4 个 cache breakpoint，超出后按优先级移除。
- 规范化 `ttl=1h/5m` 顺序，避免 5m 后面又出现 1h 导致上游拒绝。
- 对 Codex/Claude Code 类请求生成稳定 `prompt_cache_key`，并同步到 session header。
- 通过内存 TTL 缓存 fake `user_id`、`session_id`，避免每轮变化破坏上游缓存。
- Codex reasoning replay cache 保存可复用的 reasoning/function_call/custom_tool_call 片段，让 stateless 下一轮请求能保持上下文连续。

### 对当前系统的价值

可直接借鉴：

- `cache_control` 注入位置：tools[-1]、system[-1]、倒数第二个 user turn。
- 最多 4 个 breakpoint 的清理策略。
- 1h/5m TTL 顺序规范化。
- 按 API key 或会话生成稳定 `metadata.user_id` / `prompt_cache_key`。

需要谨慎：

- reasoning replay 属于 Codex/Responses 深度适配，和普通 Claude/OpenAI 代理不是一类问题，不建议现在直接迁移。
- 自动注入过多 cache_control 可能引发上游 400，需要保留 fallback：出错后降级为较少断点或无缓存版本重试。

## new-api

### 发现

new-api 的最大参考价值是“渠道亲和”。

关键代码：

- `缓存研究/new-api/service/channel_affinity.go`
- `缓存研究/new-api/middleware/distributor.go`
- `缓存研究/new-api/setting/operation_setting/channel_affinity_setting.go`
- `缓存研究/new-api/service/channel_select.go`
- `缓存研究/new-api/model/channel_cache.go`

主要机制：

- 从请求中提取亲和 key：
  - Codex：`prompt_cache_key`
  - Claude CLI：`metadata.user_id`
  - 也支持 header、上下文变量、gjson path。
- 第一次请求按正常渠道选择，成功后把 `affinity_key -> channel_id` 写入 Redis/内存缓存。
- 后续相同 key 优先使用同一 channel。
- channel 被禁用或不再支持该模型时清理亲和缓存。
- 可配置 TTL、规则名、是否包含 group/model、是否失败后跳过重试。
- 记录每个 affinity key 的 cache usage：hit/total、cached tokens、prompt tokens、cache hit tokens。
- 渠道基础选择还是按 priority/weight/random，没有看到按缓存命中率自动调度。

### 对当前系统的价值

非常值得迁移：

- `user_id/requested_model/session_key -> channel_id` 的亲和绑定。
- 成功后再写绑定，失败不污染。
- channel 禁用后清理绑定。
- 绑定 key 包含 group/model，避免不同模型或代理分组串扰。
- 单独统计 affinity key 的缓存效果，用于后续分析。

当前系统现象“首尔渠道缓存好、zz1 渠道缓存差”，本质上很适合这种机制：同一个用户/会话不要在多个上游池之间漂移。

## sub2api

### 发现

sub2api 更像多账号网关，重点是账号池调度、session sticky、OpenAI/Codex 兼容。

关键代码：

- `缓存研究/sub2api/backend/internal/service/openai_account_scheduler.go`
- `缓存研究/sub2api/backend/internal/service/openai_sticky_compat.go`
- `缓存研究/sub2api/backend/internal/service/openai_gateway_service.go`
- `缓存研究/sub2api/backend/internal/service/openai_compat_prompt_cache_key.go`
- `缓存研究/sub2api/backend/internal/service/gateway_messages_cache.go`

主要机制：

- 账号选择分三层：
  - `previous_response_id` 命中账号；
  - `session_hash` 命中账号；
  - 负载均衡选择账号。
- `session_hash` 来源优先级：
  - header `session_id`
  - header `conversation_id`
  - body `prompt_cache_key`
  - 内容 fallback：model + system + tools + first user message
- session hash 使用短 hash，并兼容旧 hash，缓存绑定关系有 TTL。
- 账号不可用、模型不匹配、并发满、错误率/TTFT 过高时可逃逸 sticky。
- 对 OpenAI/Anthropic 兼容路径自动派生 `prompt_cache_key`：
  - 基于模型、reasoning、tools、functions、system、first user message。
  - Anthropic 如果已有 cache_control，则可以基于 cache_control anchor 派生 key。
- `gateway_messages_cache.go` 会先移除 messages 内旧 `cache_control`，再统一打稳定断点：
  - 最后一条 message；
  - messages 数量 >= 4 时倒数第二个 user message。

### 对当前系统的价值

值得借鉴：

- 渠道亲和 key 可以不只依赖用户显式提供的 session，也可以用内容 fallback 生成。
- 绑定关系要支持“逃逸”：上游失败率高、channel disabled、并发满时可以切走。
- 对没有 `prompt_cache_key` 的客户端，可以生成一个稳定 key，避免每轮都变。
- 对用户乱打的 message cache_control，可以选择 normalize：先清理，再由系统统一打稳定断点。

需要谨慎：

- 内容 fallback 不能用完整消息 hash，否则多轮对话每轮都会变，粘性价值下降。更合适是 model + system + tools + first user + 用户/API key/代理维度。
- 账号级调度比当前系统复杂，如果当前只有 channel 级别，不建议一次性引入账号池复杂模型。

## 对当前系统的判断

当前系统已经有 `backend/app/services/anthropic_prompt_cache_service.py`，具备：

- Anthropic prompt cache 开关；
- `preserve/augment/normalize` 策略；
- cache_control 计数、移除、注入；
- 5m/1h fallback；
- upstream cache read/cache creation 用量解析和日志字段。

目前更明显的短板是：

- 没有“同一会话固定到同一渠道”的机制；
- 没有把渠道近期缓存表现纳入路由；
- `_inject_history_breakpoint` 当前更偏向“最新 user”，CLIProxyAPI/sub2api 更倾向“倒数第二个 user 或最后 message + 倒数第二 user”，这对多轮缓存更稳定；
- 对 OpenAI/Codex 兼容请求的 `prompt_cache_key` 稳定化还可以加强。

## 不建议做的方案：通用本地响应缓存

不建议在当前中转层做通用 LLM 响应缓存，原因：

- 流式响应、tool call、thinking、图片、web search、副作用工具都很难安全复用。
- temperature、top_p、reasoning、工具、系统提示、权限上下文稍有不同就不能复用。
- 用户隐私和跨用户串缓存风险很高。
- 即使命中，也只能复用“完全相同请求”的输出，无法解决现在的主问题：长上下文每轮都要被上游重新读入。

可接受的本地缓存只适合小范围：

- `/models`、健康检查、渠道配置、价格配置等元数据；
- 非流式、确定性、明确 opt-in 的完全相同请求；
- embedding/rerank 这类确定性较强的接口，且必须按用户/API key 隔离。

## 推荐落地方案

### P0：渠道亲和

新增一个轻量 `channel_affinity` 能力：

- key 维度：
  - `user_id`
  - `agent_id`
  - `requested_model`
  - `group` 或代理归属
  - `session_key`
- session_key 来源优先级：
  - request body `prompt_cache_key`
  - Anthropic `metadata.user_id`
  - header `session_id` / `conversation_id`
  - Codex `X-Codex-Turn-Metadata.prompt_cache_key`
  - fallback：model + system + tools + first user message + API key/user id hash
- value：`channel_id`
- TTL：建议 1h，可后台配置。
- 成功响应后写入，失败不写。
- channel disabled、模型不匹配、连续失败时清理。

这一步对“首尔缓存好、zz1 缓存差”最直接：让同一个会话不要随机漂移。

### P1：缓存感知渠道评分

基于现有 `RequestLog` 的字段聚合：

- `upstream_input_tokens`
- `upstream_cache_read_input_tokens`
- `upstream_cache_creation_input_tokens`
- `upstream_prompt_cache_status`
- `channel_id`
- `requested_model`
- `user_id/agent_id`

计算每个 `(channel_id, requested_model)` 最近 15 分钟/1 小时：

- success_count
- fail_count
- cache_read_tokens / upstream_input_tokens
- READ/NONE/BYPASS 数量

路由时：

- 长上下文请求优先缓存表现好的渠道；
- 短请求、无明显会话 key 的请求可以走普通权重；
- 缓存差但可用的渠道作为 overflow/fallback。

### P2：优化 Anthropic cache_control 断点

参考 CLIProxyAPI/sub2api 改进当前策略：

- tools：最后一个 tool 打 breakpoint；
- system：最后一个 system block 打 breakpoint；
- messages：不要总打最新 user，优先倒数第二个 user；可选再打最后一条 message；
- 超过 4 个 breakpoint 时按低价值顺序移除；
- 规范化 TTL，避免 5m 后面出现 1h；
- normalize 模式下先移除用户消息上的旧 cache_control，再统一打系统断点。

### P3：Codex/OpenAI prompt_cache_key 稳定化

对 Codex/OpenAI Responses 类请求：

- 如果已有 `prompt_cache_key`，保持并用于渠道亲和；
- 如果没有，但能从 header/session/metadata 推导，则生成稳定 key；
- 不要把每轮完整 messages hash 当唯一 key，否则多轮会话每次都变；
- 可把生成 key 写入请求体或上游 header，前提是目标上游支持该字段。

## 优先级建议

1. 先做 P0 渠道亲和，成本最低，最贴合当前问题。
2. 再做 P2 cache_control 断点调整，提高单渠道内的缓存命中稳定性。
3. 接着做 P1 缓存感知评分，让系统知道哪个渠道缓存表现差。
4. 最后做 P3 Codex/OpenAI key 稳定化，按实际请求类型逐步放开。

## 风险点

- 亲和过强可能导致某个渠道热点，需要并发满/失败率高时允许逃逸。
- 内容 fallback key 如果设计不好，会跨用户串扰或每轮变化，必须加入用户/API key/代理维度。
- 自动改 `cache_control` 有可能和客户端自带策略冲突，必须保留 `preserve/augment/normalize` 配置。
- 不能把上游错误或响应体作为缓存 key 明文存储，日志里只存 hash/短指纹。

## 最终判断

这些项目对当前系统优化缓存有明显帮助，但帮助点不是“中转层自己缓存回答”，而是：

- 让同一会话更稳定地进入同一上游缓存池；
- 让 Anthropic cache_control 更稳定；
- 让系统通过日志知道哪些渠道缓存效果差；
- 根据缓存表现做路由策略。

结合当前首尔渠道和 zz1 渠道的实测差异，下一版最建议先实现“会话/缓存 key 到 channel 的亲和绑定 + 缓存命中统计评分”。这比继续单纯调整 cache_control 更可能改善代理用户缓存命中率。
