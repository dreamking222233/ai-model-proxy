# 扩展代理端实现记录

## 一、背景与本次需求来源

当前项目本质上是一个 AI 中转站，核心职责包括：

1. 将用户请求转发到上游渠道
2. 提供统一的用户体系
3. 维护用户余额、图片积分、套餐、请求记录、消费记录
4. 提供管理端进行渠道、模型、用户、兑换码、套餐、日志等运营管理

本次扩展目标是在现有系统基础上增加“代理端”，形成：

- 平台管理端
- 代理端
- 终端用户

的三层结构。

你的核心诉求包括：

1. 平台可以创建多个代理
2. 代理不能继续发展下级代理
3. 代理有自己的前台名称、公告、联系方式、快速开始地址
4. 代理可以管理自己名下用户
5. 代理可以为下游用户充值余额、扣减余额、充值图片积分、扣减图片积分
6. 代理给下游用户发放套餐时，只能使用平台已有套餐模板，不能自建模板
7. 代理只能生成固定面额兑换码，固定面额由平台控制
8. 代理只能查看自己范围内的请求记录、消费记录、使用排行
9. 代理端不能看到真实上游模型 `actual_model`，真实模型只允许平台管理端查看
10. 代理前台要能绑定独立域名

你后续又明确补充了两条重要口径：

1. 代理账号由平台管理端发展多个代理，但代理不能继续发展代理
2. 代理给子用户充值余额后，如果代理手工扣减子用户余额，这部分额度需要自动回流到代理余额池

另一个体验要求是：

- 错误返回不要英文，尽量改成中文

## 二、方案设计概览

本次实现不是简单“加一个 `agent` 角色”，而是按“租户化”思路重构。

### 2.1 设计原则

1. 代理必须是独立实体，而不是单个 role 值
2. 所有代理侧查询必须基于 `agent_id` 做隔离
3. 前台白牌能力必须基于域名 `Host` 识别，而不是为每个代理单独构建一套前端
4. 平台管理端保留真实上游模型与全局监控能力
5. 代理端只看统一模型名，不暴露真实上游模型名

### 2.2 当前实现后的业务结构

当前系统已经扩展为以下逻辑结构：

1. 平台管理端
- 创建代理
- 维护代理站点配置
- 给代理充值余额、图片积分、套餐库存
- 配置代理可使用的固定面额兑换码规则
- 查看代理及子用户请求记录

2. 代理端
- 登录后进入 `/agent/*`
- 查看自己的站点配置
- 管理自己名下终端用户
- 给终端用户充值余额 / 扣减余额
- 给终端用户充值图片积分 / 扣减图片积分
- 给终端用户发放套餐
- 生成和删除代理兑换码
- 查看本代理下游用户的日志与排行

3. 终端用户
- 在代理站点注册 / 登录
- 获取自己的 API Key
- 调用统一模型
- 请求日志与消费记录自动归属到对应代理

### 2.3 2026-04-25 架构调整：前台分域名，API 改为统一共享域名

代理端第一版落地时，前端请求层默认采用了：

- `happy.xiaoleai.team -> happy-api.xiaoleai.team`

这种“前台域名 + API 域名都按代理拆分”的模式。

你随后明确要求改为：

- 代理前台继续按域名区分
- 后端接口统一收敛到 `api.xiaoleai.team`

因此当前代理端正式目标结构已经调整为：

1. 平台前台
- `www.xiaoleai.team`

2. 代理前台
- `happy.xiaoleai.team`
- `joy.xiaoleai.team`

3. 统一 API
- `api.xiaoleai.team`

这意味着代理识别逻辑不能再只看 API 请求的 `Host`，而需要优先识别浏览器前台来源站点。

## 五、当前代理端的实际实现逻辑

### 5.1 代理如何识别

系统当前不是基于路径识别代理，而是优先基于“请求来源站点”识别代理。

例如：

- `happy.xiaoleai.team` -> 某代理前台
- `api.xiaoleai.team` + `X-Site-Host: happy.xiaoleai.team` -> 同一代理请求

后端根据：

- `agent.frontend_domain`
- `X-Site-Host / Origin / Referer / Host`

匹配当前代理。

### 5.2 登录与鉴权逻辑

#### 1. 注册

- 用户在某代理站点注册时
- 自动写入 `sys_user.agent_id`

#### 2. 登录

- 平台管理员只能登录平台站点
- 平台直营用户只能登录平台站点
- 代理用户 / 代理终端用户只能登录自己所属代理站点

#### 3. API Key 调用

- API Key 调用时同样会检查域名归属
- 不允许代理 A 的用户拿着自己的 Key 去代理 B 域名调用

### 5.3 代理余额逻辑

当前代理余额的规则已经实现为：

1. 平台给代理充值余额
- 进入 `agent_balance.balance`

2. 代理给终端用户充值余额
- `agent_balance.balance` 减少
- `user_balance.balance` 增加

3. 代理扣减终端用户余额
- `user_balance.balance` 减少
- `agent_balance.balance` 回流增加

这是你明确要求的逻辑，当前已经按此落地。

### 5.4 图片积分逻辑

当前图片积分也已经做成代理池划转结构：

1. 平台给代理充值图片积分
2. 代理给终端用户充值图片积分
3. 代理扣减终端用户图片积分

但你最初重点明确的是“余额扣减回流代理余额池”。

当前图片积分扣减也已经做了回收能力，不过如果你后续要“图片积分必须严格与余额完全同逻辑回流”的业务口径，我建议再做一次专项确认和联调。

### 5.5 套餐库存逻辑

1. 平台创建套餐模板
2. 平台给代理充值套餐库存
3. 代理只能从自己的套餐库存中发放给终端用户
4. 代理不能自建套餐模板

### 5.6 兑换码逻辑

当前代理兑换码采用：

- **生成时预占代理余额**
- **删除未使用码时释放回代理余额**

生成兑换码时：

1. 必须选固定面额规则
2. 检查代理余额是否足够
3. 扣减代理余额池
4. 写入代理余额流水
5. 创建兑换码

删除未使用兑换码时：

1. 返还金额到代理余额池
2. 写入释放流水
3. 删除兑换码

用户兑换时：

1. 校验兑换码归属代理
2. 给用户余额入账
3. 不再重复扣代理余额，因为生成时已经预占

### 5.7 日志与排行逻辑

1. 真实请求写入 `RequestLog` / `ConsumptionRecord` / `ImageCreditRecord` 时会带 `agent_id`
2. 代理端日志接口按 `agent_id` 过滤
3. 代理端日志会强制隐藏 `actual_model`
4. 代理端排行只统计：
   - 当前代理
   - `role=user`

不会统计：

- 平台管理员
- 代理账号本身
- 其他代理

### 5.8 前台白牌逻辑

当前前台已经部分接入白牌配置：

1. 登录页品牌名和副标题 -> `/api/public/site-config`
2. 用户侧边栏站点名称 -> `/api/user/profile/site-config`
3. 用户仪表盘公告 / 微信 / QQ -> `/api/user/profile/site-config`
4. 请求层按域名自动推导 API 地址
