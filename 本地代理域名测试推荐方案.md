# 本地代理域名测试推荐方案

## 一、核心结论

现在可以本地测代理端，而且在当前新架构下：

- 不需要给每个代理再配一个本地 API 子域名
- 只需要区分前台域名
- API 可以统一走一个本地共享地址

也就是说，当前本地推荐结构已经变成：

- 平台前台：`platform.localhost:8080`
- 代理前台：`happy.localhost:8080`
- 统一 API：`api.localhost:8085`

---

## 二、为什么不能只用单一 localhost

`http://localhost:8080/admin/agents` 里的 `/admin/agents` 只是路由，不是代理标识。

如果所有页面都还是：

- `localhost:8080`

那么浏览器当前站点信息就只有一个，后端无法知道你访问的是：

- 平台前台
- 还是某个代理前台

所以：

- 单一 `localhost` 不适合验证多代理白牌逻辑

---

## 三、本地正确做法

### 推荐方式：不同前台域名 + 统一 API 域名

建议本地这样测：

- 平台前台：`platform.localhost:8080`
- 代理前台：`happy.localhost:8080`
- 统一 API：`api.localhost:8085`

或使用 hosts：

```txt
127.0.0.1 platform.local
127.0.0.1 happy.local
127.0.0.1 api.local
```

然后访问：

- `http://platform.local:8080`
- `http://happy.local:8080`
- `http://api.local:8085`

---

## 四、当前为什么这样能工作

当前代理识别不再主要依赖 API `Host`，而是优先依赖：

1. `X-Site-Host`
2. `Origin`
3. `Referer`
4. `Host`

前端请求层已经自动附带：

- `X-Site-Host: 当前 window.location.host`

所以即使浏览器请求最终打到统一 API：

- `api.localhost:8085`

后端仍然可以知道这个请求其实来自：

- `happy.localhost:8080`

---

## 五、数据库如何配置

假设你本地测试代理 `happy`，推荐配置为：

### 使用 `*.localhost`

- `frontend_domain = happy.localhost`
- `api_domain = NULL`
- `quickstart_api_base_url = http://api.localhost:8085`

### 使用 hosts

- `frontend_domain = happy.local`
- `api_domain = NULL`
- `quickstart_api_base_url = http://api.local:8085`

核心原则：

- `frontend_domain` 要和你浏览器实际打开的前台域名一致
- `quickstart_api_base_url` 指向统一 API 域名
- `api_domain` 可以不填

---

## 六、前端 dev proxy 当前状态

当前 `frontend/vue.config.js` 已经是：

- `allowedHosts: 'all'`
- `/api` 代理到 `127.0.0.1:8085`
- `/v1` 代理到 `127.0.0.1:8085`
- `changeOrigin: false`

同时这次共享 API 改造后，本地即使 dev proxy 改写了 Host，影响也已经明显降低，因为后端优先看的是：

- `X-Site-Host`

而不是 API `Host`。

---

## 七、最终建议

当前最推荐的本地测试结构是：

1. 平台前台：`platform.localhost:8080`
2. 代理前台：`happy.localhost:8080`
3. 统一 API：`api.localhost:8085`

不要再按旧方案继续配：

1. `happy-api.localhost`
2. `happy-api.local`

除非你只是为了兼容历史脚本临时保留。
